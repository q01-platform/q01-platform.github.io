Composition of HTTP Request and Error Handling

Platform Api URL​

To access the Platform APIs, it is necessary to compose a URL using the following parts:

PartDescription
Protocolhttps://
Base Urlcluster domain
Web Api pathfixed to "/api"
Versionapi Version (actually v4)
Service namethe name of the service to invoke. Generally, it's the name of a microservice (/settings, /scorecards, /users, etc...) or /core in case you want to point to the CoreService.
Resource / Routethe last part of the URL contains the resource to query (exceptions will be shown below for constructing a valid URL based on the type of data to retrieve).

The URL to use will therefore be composed of the parts: Protocol + Base URL + Web API path + Version + Service name + Resource/Route.

HTTP methods​

HTTP requests can use different methods. The Platform APIs only use the following methods:

MethodScope
GETUsed to retrieve
POSTUsed to insert new records into an entity. It supports bulk inserts (in transaction) and returns 201 upon successful insertion.
PUTUsed for modifying a record. The PUT method is used for both full and partial modification of a record; therefore, the PATCH method is not used. It returns 200 upon successful update.
DELETEUsed for deleting a record from an entity. It returns 205 upon successful deletion.

HTTP headers​

The Platform APIs only support JSON as a format.

Therefore, every request must include the headers Accept and Content-Type with the value application/json, even if sending the body is not required. The errors returned by the APIs are in JSON format.

Even if the code were to work without modifying the headers, it is recommended as a best practice to add the following values to the headers, especially if we are sending JSON (the content-type must be application/json).

Accept: application/json
Content-Type: application/json
microservice: mssettings

Custom headers​

Adding parameters in the headers is also used for other purposes.

For example, it's possible to use the canary version (if available) of a microservice by including in the headers:

canary-mode: on

Similarly, it's possible to use the developer key to point to a Skaffold instance of a microservice (when we want to share development with the team) by including:

developer: peter

Status Code​

Whether an HTTP call succeeds or fails, the response will include its status code. The status codes include the following codes:

CodeDescriptionType
200 OKWhen the operation is successfulSuccess
201 CreatedWhen the POST creation is successfulSuccess
204 No ContentWhen the operation is successful but nothing is returnedSuccess
403 ForbiddenThe following errors are returned:
  • Access Denied
  • Invalid Access Rights
Client Error
401 UnauthorizedThe following errors are returned:
  • Bad Auth
  • Expired Auth Token
  • Invalid Auth Token
  • Invalid User
  • Missing Token
  • Unauthorized Access
Client Error
400 BadRequestWhen an invalid argument is sentClient Error
404 Not FoundResource does not existsClient Error
405 Method Not AllowedWhen the method used is not valid:Client Error
500 BadRequest
501 Not ImplementedWhen the resource exists but is not implemented:Server Error
503 Service UnavailableEhwn the request service is not availableServer Error

Errors response​

The error messages returned are in JSON format. They always contain the keys "code" (or "status") and "message".

{
  "code": 401,
  "message": "Invalid token provided!",
  "data": [],
  "errors": {
    "token": [
      0: "validation failed!"
    ], ...
  }
}

In the case where the call goes through multiple microservices (for example, the client invokes the backend of a microservice which in turn calls the core service and then the core query/write APIs), the responses are nested, for example:

{
  "code": 400,
  "message": {
    "code": 400,
    "message": {
        "status": 400,
        "message": "Invalid",
        "data": [],
        "errors": {
        } 
    },
    "data": [],
    "errors": {}
  },
  "data": [],
  "errors": {}
  "username": [
    0: "Invalid length, min chars 6",
    1: "The username should contain uppercase letter, numbers and a symbol"
  ],
  "password": [
    "0": "password must be at least 8 chars!"
  ]
}
