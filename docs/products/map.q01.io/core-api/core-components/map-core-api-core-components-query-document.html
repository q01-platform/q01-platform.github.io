Query Documents

The core APIs offer a powerful tool for querying the database called documents.

When there is a need to extract a block of coherent data from the database starting from a master table (such as an Item with all associated barcodes and price lists), it is possible to generate a new document in the dimension TB_ANAG_DOCDIM00 (useful if you want to make a document with pre-set characteristics available to others) or by invoking the generation of the document "on the fly" (all necessary information for the creation of the document must be passed in the body of the API call, which presupposes knowledge of the database structure and the relationships between the dimensions). The generated document can then be queried via API to provide all the information it references.

The advantages are several:

Only one call is necessary.

The obtained data are coherent.

The relationships with the related tables are metadata-driven, so simply modifying the metadata will yield the correct data in response.

Definition of Documents​

A document requires the definition of the following parameters

Code Description
Source Dim Alfanumerico unique identifier of the document
Description Document description
Source Dim The master dimension on which the query is performed and on which any filters are applied
Key Source Dim It is the field used to filter data from the master dimension. The value is passed as an argument in the API route (see the following section).
Aggregate Dim

JSON object with the key being the code of the aggregated dimension (i.e., pointed to by the VARS/AC/PROP fields) to be added to the document. If empty, all related dimensions will be shown according to metadata logic; otherwise, only those defined in the JSON will be shown. It is possible to specify the list of fields to display or discard for each related dimension.

{
  "BCODE": [
    "XBCODE03",
    "XBCODE04"
  ], 
  "LST":[
    "!LST_ID" 
  ]
}

The "!" before a field means "exclude." In the previous example, all fields of LST will be printed except LST_ID, while only the fields XBCODE03 and XBCODE04 will be printed.

Aggregate Dim Conditions

If it is necessary to apply conditions on the join tables, these will be defined in a JSON object with the following format.

{
  "LST":[
    "XLST03 = '%codLst%'",
    "XLST25 = '%areaListino%'"
  ]
}

The values to be applied to the fields can be static or dynamic. If you want to use dynamic values, simply write the values between "%%" and pass the token code as a parameter in the API call.

The system will reject calls that do not include all tokens sent via query params.

Microservice Microservice name
Module Applicative module
Public Flag to define the public/private [default] status of the document

Query Pre-Defined Documents​

Using the documents described in the previous paragraph as a reference, let's see how to call the documents saved in TB_ANAG_DOCDIM00 via API and what return objects are provided.

The optional parameters that the service accepts are:

$top to limit the numbers of records to be returned

$offset and $num_rows are used to define the number of records to return. If used, the result is an array with the first element being a metadata object containing information about the number of pages, the current page, and the items per page returned.

debug if true, returns the sql query executed as first element of result array.

depth defines the recursion depth level possible based on the metadata. The default value is 3 (we can have up to three nested recursion levels)

dim_occurrences is used to prevent infinite loops based on the relationships described in the metadata. The default value is 3 (we can have a dimension at most three times at any depth level; from the fourth level onwards, it will no longer be expanded in detail).

compact defines the depth level for returning data.

  • false = returns the fields of the master dimension and the fields of the related dimensions
  • true = returns only the columns of the master table N.B. compact does not apply to the dimensions defined in "Related Dimensions". These dimensions will always be shown; at most, undesired fields will be hidden, or the obtained results will be filtered.

use_public_code to obtain the clear descriptions of constant codes (for example, "ArticleCode" instead of "XART03" if these fields are correctly populated). In case the public description is not available, the system returns the code field.

center_dett display|new|detail|edit|allviews are the possible values (see Query Data - Request for specific properties section)

markers if we send the parameter markers=1, the fields of the dimension like ID, CDATA, LDATA, OWNER, and LOWNER are replaced respectively with: 0, CDATA, LDATA, OWNER, LOWNER This allows the actualization of these values in case the selected document is used to populate a database (seeds).

Additionally, there are parameters related to the conditions to be applied to the related dimensions (if present in the form of tokens). For example, if we take the conditions defined in SHORT_ARTICLE_INFO (see previous paragraph), it will be necessary to send the following parameters as query params:

codLst the list code associated with the store areaListino the list area associated with the store

N.B. The tokens, if present in the document definition, are MANDATORY! The system responds with a status 500 and a message explaining the error.

{
  "status": 500,
  "message": "Bad request!, Missing codLst param"
}

Query Documents onFly​

It is possible to obtain documents even without specifying a record in DOCDIM. This is obviously not very useful when there is a need to reuse a document. However, it is very useful when there is a need to make an ad-hoc or highly customized query (for example, constructing the document through a user interface).

The parameters accepted by the service, which uses the POST method, are the same as those previously discussed and are also passed as JSON to the endpoint. The fields that were previously read from the database in the TB_ANAG_DOCDIM00 dimension are also sent in the JSON. Additionally, the conditions to be applied to the related dimensions are not passed as query param arguments but directly in the body.

Let's look at an example of an on-the-fly document that returns the same data described in the SHORT_ARTICLE_INFO document.

Request​
POST https://q01.io/api/v4/document
{
  "dim": "ART",
  "val_key": "055570",
  // It is possible to replace "val_key" with "list_val_keys" to obtain an array of documents.
  //"list_val_keys": [
  // "055570",
  //  "055570",
  //],
  // Or alternatively, use a filter to obtain an array of documents.
  //"filter_val_key": "contains(XART20,'MELIN') OR equals(XART03,'534508')",
  "related_dim": {
    "BCODE": [
      "XBCODE03"
    ], 
    "LST":[
      "XLST10",
      "XLST13",
      "XLST14",
      "XLST15"
    ]
  },
  "related_dim_condition": {
    "LST":[
      "XLST03 = '1'",
      "XLST25 = 'ITA'"
    ]
  },
  "compact": true,
  "use_public_code": false,
  "center_dett": "allviews",
  "$offset": "all",
}
