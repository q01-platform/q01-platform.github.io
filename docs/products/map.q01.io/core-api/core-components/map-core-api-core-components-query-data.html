Query Data

To obtain data from a DIM, generally (except for particular cases), we use the GET method. It's possible to apply parameters (options or filters) to filter and organize the returned data.

Basic example​

Core APIs offer various basic methods to obtain data from a DIM, useful for retrieving the details of a record, a list of records, or a document organized according to the structure of the metadata of the queried DIM.

Request​
curl "https://q01.io/api/v4/core/BCODE?&source=settings-2&center_dett=visualizza" \
--header 'Accept: application/json' \
--header 'Content-Type: application/json' \
--header 'Authorization: Bearer eyJ0eXAiOiJKV1QiLCJhbGciOiJSUzI1NiJ9.....'

Limit of returned records​

By default, APIs return a maximum of 25 records per call if no arguments are passed. To obtain a certain number of records, or to manage pagination of results, you need to use the parameters $offset and $num_rows.

The parameter $num_rows defines how many records we want to obtain in the response. The parameter $offset indicates the initial record from which to start reading the result set of the query. If $offset is set to 'all', then it's not necessary to pass the $num_rows parameter, and the system will return the entire result set.

warning

Using $offset=all is highly discouraged except for DIMs that have a limited number of records or when pagination cannot be used.

The use of $num_rows and $offset, in the case of pagination of records, should be used together with the $order parameter. To obtain the total number of records that the query could return (based also on the set filters), simply add the $count=true parameter.

Let's see some examples:

Limit the first 3 records read in the DB (without using the $order parameter, records are returned in the order assigned by the database).

Request for specific properties​

The properties of a dimension that are returned are based on the COD_ON_OFF attribute associated with the metadata and the center_dett sent during the request. Below is the table describing the relationship between center_dett and the content of COD_ON_OFF.

center_dettCOD_ON_OFF
visualizzaThe "L" must be present, for example D, N, M, L or L, R
dettaglioThe "D" must be present (D,N,M or D,L,R)
nuovoThe letter "R" must be present
editThe letter "N" must be present
allviewsThe content of COD_ON_OFF is ignored and all the properties of the registry are returned

If you want to force the columns to be obtained (not in terms of weight but of center_dett), you can use the forceListCost parameter. The value is a list of constant codes separated by commas that bypass the center_dett rules (which obviously no longer makes sense to pass as a parameter). Therefore, unlike the $select parameter which provides a subset of columns compared to those displayed (based on the center_dett), the forceListCost returns only and exclusively the requested columns with additional metadata of the table (ID, OWNER, CDATA, LOWNER, LDATA, TIMESTAMP, TREC, AMB, and AMB_RIF).

Filters​

It is possible to filter the data using the $filter parameter.

Standard operators​

OperatorsDescriptionExample
Comparison
eqEquals$filter=XLST10 eq 9.90
neNot Equals$filter=XLST10 ne 9.90
gtGreater then$filter=XLST10 gt 9.90
geGreater then or equals$filter=XLST10 ge 9.90
ltLess then$filter=XLST10 lt 9.90
leLess then or equals$filter=XLST10 le 9.90
Logical Operators
andlogical AND$filter=XLST10 gt 7.90 and XLST10 lt 10.90
orlogical OR$filter=contains(XART20,'(kg)') or contains(XART20,'pizza')
Grouping Operators
()Grouping precedence(contains(XART20,'pasta') or contains(XART20,'pizza')) and XLST10 ge 6.00

Standard functions​

The following query functions are supported.

FunctionExampleSql
equals$filter=equals(XART20,'(kg)')$filter=XART20 = '(kg)'
notequals$filter=notequals(XART20,'pizza')$filter=XART20 != '(kg)'
contains$filter=contains(XART20,'(kg)')$filter=XART20 like '%(kg)%'
endswith$filter=endswith(XART20,'pomodoro')and XART20 like '%pomodoro' trova "pizza al pomodoro", "pasta al pomodoro" ecc...
startswith$filter=startswith(XART20,'pasta')and XART20 like 'pasta%' trova "pasta e patate", "pasta e pesto" ecc...
in$filter=in(XART06,'12,13,54')and XART20 in(12,13,54)
in_s$filter=in_s(XART07,'134,abcd,AX23YE')and XART20 in('123','ascd','AX23YE')

Special filters​

Filter on TREC​

The TREC field, present in each record, cannot be used like a normal field with standard filtering functions. This is because at the core API level, the condition on TREC is always automatically applied, disregarding logically deleted records:

ifnull(TREC,'N') != 'C'

So, even if we send the parameter $filter=(equals(TREC,'C') or equals(TREC,'N')) in an API call, it will be ANDed with the previous condition, effectively nullifying the OR:

ifnull(TREC,'N') != 'C' and (TREC = 'C' OR TREC = 'N')

To control the filter on the TREC column, it's necessary to use the query parameter cond_trec, bearing in mind that the value (or values separated by commas) will be used in the database query with NOT IN mode, and the default condition will not be applied (ifnull(TREC,'N') != 'C').

Below are some examples with their corresponding SQL conversion:

// Same behavior as the default condition, in fact, nothing changes.
GET https://q01.io/api/v4/core/ART?source=settings-2&center_dett=visualizza&cond_trec=C

and TREC not in('C')
---

GET https://q01.io/api/v4/core/ART?source=settings-2&center_dett=visualizza&cond_trec=S,A

and TREC not in('S','C')
---
// In this way, we bypass the condition on TREC, effectively retrieving all the data present in the database.
GET https://q01.io/api/v4/core/ART?source=settings-2&center_dett=visualizza&cond_trec=null

and TREC not in('null')
---

Orders​

It is possible to define the sorting of the received data using the $order parameter.

Of course, there is the possibility to define the type of sorting through the suffixes ASC or DESC.

GET https://q01.io/api/v4/core/ART?source=settings-2&center_dett=visualizza&$select=XART03,XART20,XART25&$order=XART20 asc,XART03 desc

Groups​

It is possible to define the grouping of the received data using the $group parameter.

GET https://q01.io/api/v4/core/BCODE?source=settings-2&center_dett=visualizza&$select=XBCODE10,XBCODE04,XART25&$group=XBCODE10

Query on AC/VARS fields​

Particular attention is deserved by the field types AC and VARS.

In both cases, they are foreign keys linked to the main dimension; the difference lies in their graphical representation.

An example of AC or VARS is:

  • User Registry (TB_ANAG_UT00)

    Field XUT26 is the store code and refers to column XPV03 of table TB_ANAG_PV00

  • Item Registry (TB_ANAG_ART00)

    Field XART53 is the department code and refers to column XDEPFOOD03 of table TB_ANAG_DEPFOOD00

Autocomplete field (AC)​

The field type AC allows activating a search on the destination table and bringing back the value contained in the code (specified in the metadata) into the column of the source table (defined precisely with field type AC). It is used when the data in the destination table are many, or generally when it is useful to have a search to filter the data in the destination table.

For example, it is advisable to use an AC to select the country to associate with a user, it is advisable to use an AC to select a loyalty card code to associate with a customer, etc.

Select option filed (VARS)​

The field type VARS allows displaying an already expanded list of data coming from another dimension and reporting the code related to the values shown in the source table. It is used when the data in the destination table are of limited number and there is no need to filter the data to be displayed.

For example, it is advisable to use a VARS to display the genre of a movie, or the selection of the VAT code on a product, etc.

To query the core APIs and obtain the correct information regarding the field types AC or VARS, you must use the following endpoint.

In addition to the mandatory source, the endpoint also requires the field parameter, which represents the code of the field in the source table of type AC or VARS. In the case of VARS fields, it is not necessary to send any additional information to the server; it will respond with the list of data that can be attached to the source field (in our example to XART11).

Request​
GET https://q01.io/api/v4/core/vars?source=settings-2&field=XART11
Response​
[

In the case where the same field is of type AC and not VARS, you need to consider the following:

The request

Request​
GET https://q01.io/api/v4/core/vars?source=settings-2&field=XART11

would produce an empty response because AC requires a filter to be set.

Request​
GET https://q01.io/api/v4/core/vars?source=settings-2&field=XART11&$filter=startwith(XCL104,'Magl')
Response​
[
  {
    "XCL103": "002",
    "XCL104": "Magl.vis.cot.fil.fan"
  },
  {
    "XCL103": "001",
    "XCL104": "Magl.merinf.p.stoffa"
  }
]

However, it is possible to omit the filter by passing the parameter getACList=1, which effectively allows us to handle a field of type AC similarly to a VARS.

This option is useful in cases where it is necessary, on the graphical interface side, to manipulate the data that make up the AC field. For example, rendering the obtained data in a customized way (rows of different colors based on the first letter of the alphabet with which the description begins, etc.).
