Pre-Insert Functions

What Are Pre-Insert Functions?​

Pre-insert functions are metadata-configured operations that automatically populate field values during record creation (POST operations). Instead of requiring clients to provide every field value, CoreWrite executes functions defined in TB_COST metadata to generate values automatically.

Key Insight:

Pre-insert functions eliminate boilerplate - UUIDs, timestamps, slugs, sequences, hashes, and user IDs are auto-populated based on TB_COST configuration. Clients send minimal data, CoreWrite enriches it before INSERT.

Why Pre-Insert Functions Matter​

Manual Field Population (Without Functions)​

Tedious client code:

// Client must generate UUID
const uuid = crypto.randomUUID();

// Client must get current user
const userId = getCurrentUser();

// Client must slugify title
const slug = slugify(productName);

// Client must hash password
const passwordHash = md5(password);

// POST request with all values
await fetch('/api/v4/core/PRD', {
  method: 'POST',
  body: JSON.stringify({
    XPRD10: uuid,
    XPRD01: 'Premium Widget',
    XPRD11: slug,
    OWNER: userId,
    source: 'productCreate'
  })
});

Problems:

  • ❌ Client logic duplication across platforms (web, mobile, desktop)
  • ❌ Inconsistent UUID formats, hash algorithms
  • ❌ Security risk (client-side user ID manipulation)
  • ❌ Verbose request payloads

Pre-Insert Functions (Q01 Approach)​

Clean client code:

// Client sends minimal data
await fetch('/api/v4/core/PRD', {
  method: 'POST',
  body: JSON.stringify({
    XPRD01: 'Premium Widget',
    source: 'productCreate'
  })
});

CoreWrite automatically:

  • ✅ Generates UUID for XPRD10
  • ✅ Creates slug from XPRD01
  • ✅ Sets OWNER to authenticated user
  • ✅ Populates timestamps, sequences, hashes

Result: Consistent, secure, minimal client code.

TB_COST Configuration​

FUNCTION Field Format​

Syntax:

{action}|{functionName}|{arg1}|{arg2}|...

Components:

  • {action}: BI = Before Insert, PI = Post Insert, BE = Before Edit, etc.
  • {functionName}: Function to execute (uuid, md5, sequence, etc.)
  • {args}: Function-specific arguments (pipe-separated)

Example:

INSERT INTO TB_COST (
  DIM_COD, COD_COST, DESC_COST, TIPO_CAMPO, COD_ON_OFF, FUNCTION, ...
) VALUES (
  'PRD', 'XPRD10', 'Product UUID', 'varchar(36)', 'L,D', 'BI|uuid|microtime', ...
);

Interpretation:

  • BI: Execute before INSERT
  • uuid: Generate UUID
  • microtime: Use microtime as UUID component

Code Reference: DataStore.php:1433-1454

private function getFunction($fields, $action) {
    foreach ($fields as $codCost => $costArr) {
        if ($costArr['function']) {
            $arrFunc = explode("|", $costArr['function']);
            $prePostAction = array_shift($arrFunc);  // BI

            if ($action == "nuovo" && $prePostAction == 'BI') {
                $funcName = array_shift($arrFunc);  // uuid
                $funcObj[$funcName . "__" . $codCost] = [
                    "key" => $codCost,
                    "args" => $arrFunc  // [microtime]
                ];
            }
        }
    }
}

Available Functions​

1. uuid - Generate UUID​

Purpose: Create globally unique identifier using UUID v5.

Configuration:

FUNCTION = 'BI|uuid|microtime'
FUNCTION = 'BI|uuid|pad,XPRD02,10,0,LEFT|microtime'

Behavior:

// Example 1: Simple microtime UUID
$uuid = date("YmdHis") . str_replace(".", "", microtime(true));
$result = UUID::v5('00000000-0000-0000-0000-000000000000', $uuid);
// Result: "A7F3C2B1-5D4E-5678-9ABC-DEF012345678"

// Example 2: UUID with padded field
$uuid = str_pad($param['XPRD02'], 10, '0', STR_PAD_LEFT) . "-" .
        date("YmdHis") . str_replace(".", "", microtime(true));
$result = UUID::v5('00000000-0000-0000-0000-000000000000', $uuid);

Arguments:

  • microtime: Append current date + microsecond timestamp
  • pad,FIELD,LENGTH,CHAR,DIRECTION: Pad field value before UUID generation

Use Cases:

  • Globally unique product IDs
  • Session identifiers
  • Transaction IDs

Code Reference: DataStore.php:1474-1502

2. md5 - Hash Field Value​

Purpose: Generate MD5 hash of field value (e.g., passwords).

Configuration:

FUNCTION = 'BI|md5'

Behavior:

$param['XUSER_PASSWORD'] = md5($param['XUSER_PASSWORD']);
// "secret" -> "5ebe2294ecd0e0f08eab7690d2a6ee69"

Use Cases:

  • Password hashing (legacy systems - prefer bcrypt for new systems)
  • Checksum generation
  • Simple data fingerprints

Code Reference: DataStore.php:1519-1521

Security Warning: MD5 is not cryptographically secure for passwords. Use for checksums only.

3. sequence - Auto-Increment Field​

Purpose: Generate sequential numbers within a partition (e.g., line item sequence per order).

Configuration:

FUNCTION = 'BI|sequence|#XORDITEM01#'

Behavior:

// Get max sequence for partition
SELECT MAX(XORDITEM04) + 1 as NEXT_SEQ
FROM TB_ANAG_ORDITEM00
WHERE XORDITEM01 = '555';  // Partition key

// If no records: NEXT_SEQ = 1
// If max = 5: NEXT_SEQ = 6
$param['XORDITEM04'] = $nextSeq;

Arguments:

  • #FIELD#: Partition field wrapped in # (e.g., #XORDITEM01#)

Use Cases:

  • Line item numbering (1, 2, 3, ... per order)
  • Step sequence in workflows
  • Position in lists

Code Reference: DataStore.php:1542-1574

Example:

TB_COST Configuration:

-- Order item sequence field
INSERT INTO TB_COST (DIM_COD, COD_COST, FUNCTION, ...)
VALUES ('ORDITEM', 'XORDITEM04', 'BI|sequence|#XORDITEM01#', ...);

Create Order Item Request:

POST /api/v4/core/ORDITEM
{
  "XORDITEM01": "555",
  "XORDITEM01": "Widget A",
  "source": "orderItemCreate"
}

CoreWrite Automatically:

-- Check existing items for order 555
SELECT MAX(XORDITEM04) FROM TB_ANAG_ORDITEM00 WHERE XORDITEM01 = '555';
-- Returns 2 (items 1 and 2 exist)

-- Insert with sequence = 3
INSERT INTO TB_ANAG_ORDITEM00 (XORDITEM01, XORDITEM01, XORDITEM04, ...)
VALUES ('555', 'Widget A', 3, ...);

4. slugify - Create URL-Friendly Slug​

Purpose: Convert text to URL-safe slug (lowercase, hyphens, no special chars).

Configuration:

FUNCTION = 'BI|slugify|XPRD01'

Behavior:

$text = $param['XPRD01'];  // "Premium Widget™ 2025"
$slug = $this->slugify($text);
// Result: "premium-widget-2025"

$param['XPRD11'] = $slug;

Slugify Algorithm:

function slugify($text, $divider = '-') {
    // Replace non-alphanumeric with divider
    $text = preg_replace('~[^\pL\d]+~u', $divider, $text);

    // Transliterate to ASCII
    $text = iconv('utf-8', 'us-ascii//TRANSLIT', $text);

    // Remove unwanted characters
    $text = preg_replace('~[^-\w]+~', '', $text);

    // Trim and lowercase
    $text = trim($text, $divider);
    $text = strtolower($text);

    // Remove duplicate dividers
    $text = preg_replace('~-+~', $divider, $text);

    return $text;
}

Arguments:

  • FIELD: Source field to slugify

Use Cases:

  • SEO-friendly URLs (/products/premium-widget-2025)
  • Unique identifiers from names
  • File names from titles

Code Reference: DataStore.php:1576-1608

5. current_user - Set Current User ID​

Purpose: Auto-populate field with authenticated user ID from session.

Configuration:

FUNCTION = 'BI|current_user'

Behavior:

$param['XPRD23'] = $this->session->get("login_id", null, "grant_session");
// Result: "user_5"

Use Cases:

  • Created by tracking (alternative to OWNER)
  • Modified by tracking
  • Ownership assignment

Code Reference: DataStore.php:1581-1583

Security: Cannot be overridden by client - always uses session user ID.

6. concat - Concatenate Fields​

Purpose: Combine multiple field values into single string.

Configuration:

FUNCTION = 'BI|concat|XPRD01|XPRD02'
FUNCTION = 'BI|concat|pad,XPRD02,10,0,LEFT|XPRD01'

Behavior:

// Example 1: Simple concatenation
$concat = $param['XPRD01'] . $param['XPRD02'];
// "Widget" + "Premium" -> "WidgetPremium"

// Example 2: With padding
$concat = str_pad($param['XPRD02'], 10, '0', STR_PAD_LEFT) . $param['XPRD01'];
// 123 + "Widget" -> "0000000123Widget"

Arguments:

  • FIELD1|FIELD2|...: Fields to concatenate
  • pad,FIELD,LENGTH,CHAR,DIRECTION: Pad specific field

Use Cases:

  • Composite keys
  • Display names (first + last name)
  • Code generation

Code Reference: DataStore.php:1523-1540

7. replace - Copy Field Value​

Purpose: Copy value from one field to another.

Configuration:

FUNCTION = 'BI|replace|SOURCE_FIELD'

Behavior:

$param['XPRD29'] = $param['XPRD01'];

Use Cases:

  • Field duplication
  • Default value from another field
  • Backup original values

Code Reference: DataStore.php:1468-1472

8. ifnull - Default Value If Empty​

Purpose: Set field value only if not already provided.

Configuration:

FUNCTION = 'BI|ifnull|DefaultValue'
FUNCTION = 'BI|ifnull|#XPRD01#'

Behavior:

// If field not set or empty
if (!isset($param['XPRD13']) || $param['XPRD13'] == "") {
    // Use static value
    $param['XPRD13'] = 'ACTIVE';

    // OR use value from another field (wrapped in #)
    $param['XPRD05'] = $param['XPRD01'];
}

Arguments:

  • Static value: 'DefaultValue'
  • Dynamic value: '#FIELD#' (wrapped in #)

Use Cases:

  • Default status values
  • Fallback to other fields
  • Required field auto-population

Code Reference: DataStore.php:1504-1517

Execution Flow​

CREATE Request Lifecycle​

1. Client Request:

POST /api/v4/core/PRD
{
  "XPRD01": "Premium Widget",
  "XPRD02": 49.99,
  "source": "productCreate"
}

2. CoreWrite Loads Metadata:

// Load TB_COST metadata for PRD dimension
$fields = [
    'XPRD01' => ['TIPO_CAMPO' => 'varchar(255)', 'FUNCTION' => null],
    'XPRD02' => ['TIPO_CAMPO' => 'decimal(10,2)', 'FUNCTION' => null],
    'XPRD10' => ['TIPO_CAMPO' => 'varchar(36)', 'FUNCTION' => 'BI|uuid|microtime'],
    'XPRD11' => ['TIPO_CAMPO' => 'varchar(255)', 'FUNCTION' => 'BI|slugify|XPRD01'],
    'XPRD23' => ['TIPO_CAMPO' => 'varchar(50)', 'FUNCTION' => 'BI|current_user']
];

3. Extract Pre-Insert Functions:

$functions = getFunction($fields, 'nuovo');
// Result:
[
    'uuid__XPRD10' => ['key' => 'XPRD10', 'args' => ['microtime']],
    'slugify__XPRD11' => ['key' => 'XPRD11', 'args' => ['XPRD01']],
    'current_user__XPRD23' => ['key' => 'XPRD23', 'args' => []]
]

4. Execute Functions:

foreach ($functions as $funcName => $object) {
    preInsert($param, $funcName, $object);
}

// After execution, $param enriched:
$param = [
    'XPRD01' => 'Premium Widget',
    'XPRD02' => 49.99,
    'XPRD10' => 'A7F3C2B1-5D4E-5678-9ABC-DEF012345678',  // Generated
    'XPRD11' => 'premium-widget',                         // Generated
    'XPRD23' => 'user_5',                            // Generated
    'source' => 'productCreate'
];

5. INSERT with Enriched Data:

INSERT INTO TB_ANAG_PRD00 (
  XPRD01, XPRD02, XPRD10, XPRD11, XPRD23,
  OWNER, CDATA, LOWNER, LDATA, TREC
) VALUES (
  'Premium Widget', 49.99, 'A7F3C2B1-...', 'premium-widget', 'user_5',
  'user_5', '20250119120000', 'user_5', '20250119120000', 'N'
);

Best Practices​

✅ DO: Use Pre-Insert Functions for Derived Values​

Prefer metadata configuration:

FUNCTION = 'BI|slugify|XPRD01'

Over client-side logic:

// Avoid this
const slug = slugify(productName);
fetch('/api/v4/core/PRD', { body: { XPRD01: productName, XPRD11: slug } });

✅ DO: Use uuid for Global Identifiers​

For distributed systems:

FUNCTION = 'BI|uuid|microtime'

Benefit: No collision risk across tenants/microservices.

✅ DO: Use sequence for Partition-Scoped Sequences​

For line items, steps, positions:

FUNCTION = 'BI|sequence|#PARENT_ID_FIELD#'

Benefit: Consistent numbering per partition (order, workflow, etc.).

✅ DO: Use current_user for Audit Fields​

Track creator without client input:

FUNCTION = 'BI|current_user'

Benefit: Tamper-proof user tracking.

❌ DON'T: Use md5 for Password Hashing (New Systems)​

Wrong (insecure):

FUNCTION = 'BI|md5'

Right (use bcrypt or Argon2):

$hashedPassword = password_hash($password, PASSWORD_BCRYPT);

Why: MD5 is cryptographically broken. Use for checksums only.

❌ DON'T: Manually Override Pre-Insert Fields​

Wrong:

POST /api/v4/core/PRD
{
  "XPRD01": "Widget",
  "XPRD10": "CUSTOM-UUID",  // Manual override
  "source": "productCreate"
}

Why: Pre-insert function will overwrite manual value.

Workaround (if needed):

POST /api/v4/core/PRD
{
  "XPRD01": "Widget",
  "skip_preinsert": {
    "XPRD10": true  // Skip UUID generation for this field
  },
  "XPRD10": "CUSTOM-UUID",
  "source": "productCreate"
}

Code Reference: DataStore.php:1498-1500

❌ DON'T: Combine Incompatible Functions​

Wrong:

-- md5 and slugify on same field
FUNCTION = 'BI|md5|BI|slugify|XPRD01'

Why: Only one function per field. Use separate fields for different transformations.

Common Pitfalls​

Pitfall 1: Expecting Client-Provided UUID to Persist​

Problem: Client sends UUID, but pre-insert function overwrites it.

Solution: Either:

  1. Remove pre-insert function from TB_COST
  2. Use skip_preinsert parameter

Pitfall 2: Using sequence Without Partition​

Problem:

FUNCTION = 'BI|sequence'  -- Missing partition field

Result: Sequences collide across all records.

Solution: Always specify partition:

FUNCTION = 'BI|sequence|#PARENT_ID#'

Pitfall 3: Slugifying Mutable Fields​

Problem: Slug changes when source field updates.

Solution: Only slugify on INSERT (BI), not UPDATE (BE):

FUNCTION = 'BI|slugify|XPRD01'  -- Only on create

Alternative: Use immutable field as slug source.

Summary​

Pre-Insert Functions provide:

  1. ✅ Automatic field population - UUIDs, slugs, sequences, hashes
  2. ✅ Metadata-driven - Configure in TB_COST, no code changes
  3. ✅ Consistent logic - Same transformations across all clients
  4. ✅ Security - Server-side user ID population prevents tampering
  5. ✅ Minimal client code - Clients send essential data only

Key Insight:

Pre-insert functions are the data enrichment layer - transforming minimal client input into complete, consistent database records. Define once in metadata, benefit everywhere.
