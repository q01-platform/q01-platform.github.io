MAP Framework: The Metadata Engine

What is MAP?​

MAP (Metadata Application Platform) is the interconnected system of metadata tables that define ALL behavior in Q01 Core APIs.

Instead of hardcoding business logic in code, Q01 stores:

  • Entity structures (dimensions, fields, types)
  • Validation rules (required fields, data types, constraints)
  • Permissions (who can see/edit which fields)
  • Relationships (foreign keys, lookups, hierarchies)
  • Business logic (counters, pre-insert functions, cascades)
  • Index definitions (unique constraints, composite indexes)

Core APIs interpret this metadata at runtime to dynamically construct queries, validate data, and enforce rules.

The Core Metadata Tables​

MAP consists of 7 primary metadata tables:

1. TB_DIM - Dimension Registry​

Purpose: Registers all entity types (dimensions) in the system.

-- Find all descendants of a menu
SELECT child.*
FROM TB_MENU parent
JOIN TB_MENU child ON child.NLEFT BETWEEN parent.NLEFT AND parent.NRIGHT

SELECT parent.*
FROM TB_MENU parent
JOIN TB_MENU child ON child.NLEFT BETWEEN parent.NLEFT AND parent.NRIGHT
WHERE child.COD_MENU = 'artwork-2';

{
  "select": "a.*, c.DESC_CATEGORY",
  "from": "TB_ANAG_ART00 a",
  "join": [
    {
      "type": "LEFT JOIN",
      "table": "TB_ANAG_CATEGORY00 c",
      "on": "a.XART_CATEGORY = c.CATEGORY_ID"
    }
  ],
  "where": "a.TREC != 'C'"
}

Usage in Core APIs:

  • TB_DIM references TB_OBJECT via NAME_OBJ
  • TYPE_OBJ determines query construction strategy
  • VIEW_SCHEMA provides view definition for complex queries

Usage in Core APIs:

  • CoreWrite loads TB_INDEX before creating dimension tables
  • Generates CREATE INDEX statements based on metadata
  • Validates unique constraints during INSERT/UPDATE
  • Optimizes queries using defined indexes

CoreWrite Metadata Queries:

-- 1. Load dimension
SELECT * FROM TB_DIM WHERE COD_DIM = 'ART';
-- Result: TABLE_NAME_PUB = TB_ANAG_ART00, COD_COD_ANAG = ART_ID

-- 2. Load fields (center_dett=nuovo)
SELECT * FROM TB_COST
WHERE COD_DIM = 'ART'
  AND (contains(COD_ON_OFF,'N') OR COD_ON_OFF = 'I')
ORDER BY COD_SEQUENZA;
-- Returns: XART01, XART02 (I), XART03 (counter), XART04, XART05, XART21

-- 3. Check permissions
SELECT * FROM TB_MENU WHERE COD_MENU = 'artwork-2';
-- Validates: user grant, ENDPOINTS includes ART

-- 4. Load counter
SELECT * FROM TB_COUNTER
WHERE COD_DIM = 'ART' AND FIELD_KEY = 'XART03';
-- Format: ART-%year%-%month%-#NEXT#

-- 5. Check unique index
SELECT * FROM TB_INDEX
WHERE COD_DIM = 'ART' AND UNIQUE_IDX = 1;
-- XART03 is unique - validate no duplicates

Metadata Relationships​

TB_DIM (COD_DIM='ART')
  │
  ├─→ TB_COST (COD_DIM='ART') - Field definitions
  │     └─→ TB_TYPE_FIELDS (TYPE_FIELD='text', 'counter', 'VARS') - Field types
  │
  ├─→ TB_INDEX (COD_DIM='ART') - Index definitions
  │
  ├─→ TB_OBJECT (COD_DIM='ART') - Object registration
  │
  ├─→ TB_MENU (DIM_COD='ART') - Security areas
  │
  └─→ TB_COUNTER (COD_DIM='ART') - Counter configuration

MAP Framework consists of 7 interconnected metadata tables:

TablePurposeRows per Dimension
TB_DIMDimension registry1
TB_OBJECTObject registration1
TB_COSTField definitions10-50
TB_INDEXIndex definitions1-10

TB_MENUSecurity areas1-10
TB_COUNTERCounter configuration0-5

Key Principles:

  1. ✅ Metadata is code - behavior defined in tables, not files
  2. ✅ Runtime interpretation - APIs read metadata on every request
  3. ✅ Zero deployment - changes take effect immediately
  4. ✅ Consistency guaranteed - one source of truth for all rules
  5. ✅ Multi-tenant isolation - MARKET field in all tables
