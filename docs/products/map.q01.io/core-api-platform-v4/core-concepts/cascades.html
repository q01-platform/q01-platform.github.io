Cascade Operations

What Are Cascade Operations?​

Cascade operations are metadata-driven automatic propagations of CRUD operations across related dimensions (entity types). When you create, update, or delete a parent record, CoreWrite automatically performs corresponding operations on child records.

Key Insight:

Cascades are not hardcoded foreign key constraints - they're flexible, metadata-configured relationships that maintain referential integrity while providing full control over propagation behavior.

Why Cascades Matter​

Traditional database foreign keys with ON DELETE CASCADE are rigid:

  • ❌ All-or-nothing cascading
  • ❌ No custom logic during propagation
  • ❌ No outbox event publishing for cascaded changes
  • ❌ No soft delete support

Q01 Cascades provide:

  • ✅ Metadata-driven - Configure in TB_COST, not SQL DDL
  • ✅ Soft delete aware - Cascade TREC='C' instead of physical delete
  • ✅ Event publishing - Every cascaded operation goes to outbox
  • ✅ Transactional - All cascades in single ACID transaction
  • ✅ Flexible mappings - Transform data during cascade
  • ✅ Conditional propagation - Add extra conditions

The Three Cascade Types​

Cascade TypeTriggerPurposeCode Reference
INSERT_CASCADEParent record created (POST)Auto-create related child recordsDataStore.php:1372
UPDATE_CASCADEParent key field updated (PUT/PATCH)Update foreign keys in child recordsDataStore.php:1878
DELETE_CASCADEParent record deleted (DELETE)Soft delete child records (TREC='C')DataStore.php:2361

INSERT_CASCADE​

What It Does​

When you create a parent record, automatically create related child records.

Use Cases:

  • Creating order with default line items
  • Creating user with default settings
  • Creating product with default pricing tiers

Metadata Structure​

Configured in dimension metadata (loaded by CoreWrite):

'INSERT_CASCADE' => [
    'CHILD_DIM' => [
        'replace' => [
            'PARENT_FIELD' => 'CHILD_FIELD'
        ],
        'body' => [
            [
                'CHILD_FIELD_1' => 'value1',
                'CHILD_FIELD_2' => 'value2'
            ]
        ]
    ]
]

Fields:

  • CHILD_DIM: Child dimension code (e.g., 'ORDITEM')
  • replace: Field mapping from parent to child
  • body: Array of records to create (can be multiple)

Example: Order with Default Item​

Scenario: When creating an order, automatically create a default "Shipping Fee" line item.

Metadata Configuration:

// In ORD dimension metadata
'INSERT_CASCADE' => [
    'ORDITEM' => [
        'replace' => [
            'ORD_ID' => 'XORDITEM01'  // Map parent ID to child FK
        ],
        'body' => [
            [
                'XORDITEM01' => 'Shipping Fee',
                'XORDITEM02' => 9.99,
                'XORDITEM03' => 1  // quantity
            ]
        ]
    ]
]

Create Order Request:

POST /api/v4/core/ORD
{
  "XORD01": "2025-01-19",
  "XORD02": "John Doe",
  "source": "orderCreate"
}

What Happens:

  1. Parent INSERT:
INSERT INTO TB_ANAG_ORD00 (XORD01, XORD02, OWNER, CDATA, LOWNER, LDATA, TREC)
VALUES ('2025-01-19', 'John Doe', 'user_5', '20250119120000', 'user_5', '20250119120000', 'N');

-- Returns ORD_ID = 555
  1. Cascade INSERT (automatic):
INSERT INTO TB_ANAG_ORDITEM00 (
  XORDITEM01,  -- Replaced with ORD_ID=555
  XORDITEM01,
  XORDITEM02,
  XORDITEM03,
  OWNER, CDATA, LOWNER, LDATA, TREC
) VALUES (
  555,  -- From parent ORD_ID
  'Shipping Fee',
  9.99,
  1,
  'user_5', '20250119120000', 'user_5', '20250119120000', 'N'
);

Response:

[{
  "code": 201,
  "insertedId": "555",
  "body": {
    "XORD01": "2025-01-19",
    "XORD02": "John Doe"
  },
  "INSERT_CASCADE": {
    "ORDITEM": {
      "0": {
        "code": 201,
        "insertedId": "1001"
      }
    }
  }
}]

Code Reference: DataStore.php:1372-1823

if (sizeof($this->curr_metadata) && isset($this->curr_metadata['INSERT_CASCADE'])) {
    foreach ($this->curr_metadata['INSERT_CASCADE'] as $tbl => $cascadeInfo) {
        $replaceArray = $cascadeInfo['replace'];
        $currBody = $cascadeInfo['body'];
        // ... insert child records
    }
}

Multi-Insert Cascade​

Multiple Child Records:

'INSERT_CASCADE' => [
    'ORDITEM' => [
        'replace' => [
            'ORD_ID' => 'XORDITEM01'
        ],
        'body' => [
            ['XORDITEM01' => 'Shipping Fee', 'XORDITEM02' => 9.99],
            ['XORDITEM01' => 'Handling Fee', 'XORDITEM02' => 5.00],
            ['XORDITEM01' => 'Tax', 'XORDITEM02' => 0.00]
        ]
    ]
]

Result: Creates 3 child records automatically.

UPDATE_CASCADE​

What It Does​

When you update a parent's key field, automatically update foreign keys in all related child records.

Use Cases:

  • Changing order number (update all line items)
  • Renaming category (update all products)
  • Updating customer ID (update all addresses)

Metadata Structure​

'UPDATE_CASCADE' => [
    'srcKey' => 'PARENT_KEY_FIELD',
    'applyTo' => [
        'CHILD_DIM' => [
            'destKey' => 'CHILD_FK_FIELD',
            'extraCond' => ''  // Optional additional WHERE clause
        ]
    ]
]

Fields:

  • srcKey: Field in parent dimension being updated
  • applyTo: Array of child dimensions
  • destKey: Foreign key field in child dimension
  • extraCond: Optional SQL WHERE condition

Example: Update Order Code​

Scenario: When order code changes, update all line items to reference new code.

Metadata Configuration:

// In ORD dimension metadata
'UPDATE_CASCADE' => [
    'srcKey' => 'XORD02',
    'applyTo' => [
        'ORDITEM' => [
            'destKey' => 'XORDITEM02'
        ]
    ]
]

Update Order Request:

PUT /api/v4/core/ORD/555
{
  "XORD02": "ORD-2025-NEW",
  "source": "orderEdit"
}

What Happens:

  1. Parent UPDATE:
UPDATE TB_ANAG_ORD00 SET
  XORD02 = 'ORD-2025-NEW',
  LOWNER = 'user_5',
  LDATA = '20250119130000',
  TREC = 'M'
WHERE ORD_ID = 555;
  1. Cascade UPDATE (automatic):
UPDATE TB_ANAG_ORDITEM00 SET
  XORDITEM02 = 'ORD-2025-NEW',
  LOWNER = 'user_5',
  LDATA = '20250119130000',
  TREC = 'M'
WHERE XORDITEM02 = 'ORD-2025-OLD';  -- Old value

Code Reference: DataStore.php:1878-2140

if ($this->metadata['metadata'] && isset($this->metadata['UPDATE_CASCADE'])) {
    $srcKeyVal = $param[$this->metadata['UPDATE_CASCADE']['srcKey']];

    foreach ($this->metadata['UPDATE_CASCADE']['applyTo'] as $tbl => $updateInfo) {
        $destKey = $updateInfo['destKey'];
        // ... update child records
    }
}

Extra Conditions​

Add Conditional Logic:

'UPDATE_CASCADE' => [
    'srcKey' => 'XORD02',
    'applyTo' => [
        'ORDITEM' => [
            'destKey' => 'XORDITEM02',
            'extraCond' => "AND XORDITEM05 = 'ACTIVE'"  // Only update active items
        ]
    ]
]

Generated SQL:

UPDATE TB_ANAG_ORDITEM00 SET
  XORDITEM02 = 'ORD-2025-NEW',
  LOWNER = 'user_5',
  LDATA = '20250119130000',
  TREC = 'M'
WHERE XORDITEM02 = 'ORD-2025-OLD'
  AND XORDITEM05 = 'ACTIVE';  -- Extra condition

DELETE_CASCADE​

What It Does​

When you delete a parent record, automatically soft delete (TREC='C') all related child records.

Key Behavior:

  • Soft delete by default - Sets TREC='C' on children
  • Physical delete if parent forced - If forceDelete=true, physically deletes children too
  • Outbox events - Publishes delete event for each child

Use Cases:

  • Deleting order deletes all line items
  • Deleting category soft-deletes all products
  • Deleting customer soft-deletes all addresses

Metadata Structure​

'DELETE_CASCADE' => [
    'srcKey' => 'PARENT_KEY_FIELD',
    'applyTo' => [
        'CHILD_DIM' => 'CHILD_FK_FIELD'
    ]
]

Fields:

  • srcKey: Field in parent dimension (usually primary key)
  • applyTo: Map of child dimensions to foreign key fields

Example: Delete Order and Items​

Metadata Configuration:

// In ORD dimension metadata
'DELETE_CASCADE' => [
    'srcKey' => 'ORD_ID',
    'applyTo' => [
        'ORDITEM' => 'XORDITEM01'
    ]
]

Delete Order Request:

DELETE /api/v4/core/ORD/555

What Happens:

  1. Lookup Parent Key:
SELECT ORD_ID as KEYVAL FROM TB_ANAG_ORD00 WHERE ORD_ID = 555;
-- Returns KEYVAL = 555
  1. Cascade Soft Delete (automatic):
UPDATE TB_ANAG_ORDITEM00 SET
  TREC = 'C',
  LDATA = '20250119140000'
WHERE XORDITEM01 = 555;
  1. Parent Soft Delete:
UPDATE TB_ANAG_ORD00 SET
  TREC = 'C',
  LOWNER = 'user_5',
  LDATA = '20250119140000'
WHERE ORD_ID = 555;

Result: Order and all line items now have TREC='C', recoverable.

Code Reference: DataStore.php:2361-2380

if ($metadata['metadata'] && isset($metadata['DELETE_CASCADE'])) {
    $sql = "select {$metadata['DELETE_CASCADE']['srcKey']} as KEYVAL
            from TB_ANAG_{$dim}00 where {$dim}_ID = ?";
    $cascadeVal = $data['KEYVAL'];

    foreach ($metadata['DELETE_CASCADE']['applyTo'] as $tbl => $joinField) {
        $q = "update TB_ANAG_{$tbl}00 set TREC = 'C', LDATA = '" . date("YmdHis") . "'
              where {$joinField} = '{$cascadeVal}'";
        // ... execute cascade delete
    }
}

Physical Delete Cascade​

Force Delete Request:

DELETE /api/v4/core/ORD/555?forceDelete=true

Cascade Physical Delete:

-- Delete children first
DELETE FROM TB_ANAG_ORDITEM00 WHERE XORDITEM01 = 555;

-- Then delete parent
DELETE FROM TB_ANAG_ORD00 WHERE ORD_ID = 555;

Warning: Irreversible - no TREC='C', no recovery.

Cascade and Outbox Pattern​

Critical: All cascade operations publish events to TB_ANAG_OUTBOX00.

Example: Delete Order with Outbox​

Delete Operation:

DELETE /api/v4/core/ORD/555

Outbox Entries Created:

  1. Parent Delete Event:
{
  "XOUTBOX01": "ORD",
  "XOUTBOX02": "logicalDelete",
  "XOUTBOX03": "{\"ORD_ID\": \"555\"}",
  "XOUTBOX04": "2025-01-19 14:00:00",
  "XOUTBOX05": 0
}
  1. Cascaded Child Delete Event:
{
  "XOUTBOX01": "ORDITEM",
  "XOUTBOX02": "logicalDelete",
  "XOUTBOX03": "{\"XORDITEM01\": \"555\"}",
  "XOUTBOX04": "2025-01-19 14:00:00",
  "XOUTBOX05": 0
}

Result: RabbitMQ fanout exchange publishes both events, consumers notified.

Code Reference: DataStore.php:2377-2380

$sqlOutboxDelete[$i] = [
    "dim" => $tbl,
    'delete_keys' => [
        $joinField => $cascadeVal
    ]
];

Multi-Level Cascades​

Cascades can chain across multiple levels:

ORDER (parent)
  └─ ORDITEM (child, has DELETE_CASCADE)
       └─ ORDITEM_DISCOUNT (grandchild, has DELETE_CASCADE)

Delete Order 555:

  1. DELETE_CASCADE: ORD → ORDITEM
  2. DELETE_CASCADE: ORDITEM → ORDITEM_DISCOUNT (triggered by step 1)

Result: All three levels soft deleted (TREC='C').

Configuring Cascades in TB_COST​

While cascade metadata is typically loaded programmatically, TB_COST stores relationship information:

INSERT_CASCADE Example​

Field Definition:

INSERT INTO TB_COST (
  DIM_COD, COD_COST, DESC_COST, TIPO_CAMPO,
  COD_ON_OFF, COD_SEQUENZA, INSERT_CASCADE, ...
) VALUES (
  'ORD', 'ORD_ID', 'Order ID', 'int',
  'L,D', 10, 'ORDITEM:XORDITEM01', ...
);

INSERT_CASCADE Format: CHILD_DIM:CHILD_FK_FIELD

UPDATE_CASCADE Example​

INSERT INTO TB_COST (
  DIM_COD, COD_COST, UPDATE_CASCADE, ...
) VALUES (
  'ORD', 'XORD02', 'ORDITEM:XORDITEM02', ...
);

DELETE_CASCADE Example​

INSERT INTO TB_COST (
  DIM_COD, COD_COST, DELETE_CASCADE, ...
) VALUES (
  'ORD', 'ORD_ID', 'ORDITEM:XORDITEM01,ORDNOTE:XORDNOTE01', ...
);

Multiple Children: Comma-separated CHILD1:FK1,CHILD2:FK2

Transaction Guarantees​

All cascades execute in a single ACID transaction:

try {
    $this->db_data->beginTransaction();

    // 1. Execute parent operation
    // 2. Execute cascade operations
    // 3. Write to outbox

    $this->db_data->commit();
} catch (Exception $e) {
    $this->db_data->rollback();
    // All changes rolled back - no partial cascades
}

Guarantee: Either all operations succeed or none do.

Best Practices​

✅ DO: Use Soft Delete Cascades​

Prefer TREC='C' propagation:

'DELETE_CASCADE' => [
    'srcKey' => 'ORD_ID',
    'applyTo' => [
        'ORDITEM' => 'XORDITEM01'
    ]
]

Benefit: Recoverable, maintains audit trail.

✅ DO: Configure Bidirectional Updates​

Example: Category name change updates products:

// In CATEGORY dimension
'UPDATE_CASCADE' => [
    'srcKey' => 'XCAT_NAME',
    'applyTo' => [
        'PRODUCT' => ['destKey' => 'XPRD05_NAME']
    ]
]

✅ DO: Use Extra Conditions for Selective Cascades​

Only update active records:

'UPDATE_CASCADE' => [
    'srcKey' => 'XORD04',
    'applyTo' => [
        'ORDITEM' => [
            'destKey' => 'XORDITEM03',
            'extraCond' => "AND XORDITEM06 = 0"
        ]
    ]
]

✅ DO: Test Cascade Chains​

Multi-level cascades require testing:

  1. Create parent with cascade
  2. Update parent key
  3. Delete parent
  4. Verify all levels affected correctly

❌ DON'T: Cascade on Non-Key Fields​

Wrong:

'UPDATE_CASCADE' => [
    'srcKey' => 'XORD03',  // Not a key field!
    'applyTo' => [...]
]

Why: Performance - updates trigger cascade on every total recalculation.

Right:

// Only cascade on key fields (IDs, codes)
'UPDATE_CASCADE' => [
    'srcKey' => 'XORD02',
    'applyTo' => [...]
]

❌ DON'T: Create Circular Cascades​

Wrong:

// ORD dimension
'UPDATE_CASCADE' => ['applyTo' => ['ORDITEM' => [...]]]

// ORDITEM dimension
'UPDATE_CASCADE' => ['applyTo' => ['ORD' => [...]]]  // Circular!

Why: Infinite loop, transaction timeout.

❌ DON'T: Mix Physical and Soft Deletes​

Wrong:

// Parent: soft delete
DELETE /api/v4/core/ORD/555

// Child: force delete (inconsistent)
DELETE /api/v4/core/ORDITEM/1001?forceDelete=true

Why: Parent TREC='C' but child physically gone - referential integrity broken.

Right: Both soft delete or both physical delete.

Common Pitfalls​

Pitfall 1: Forgetting to Configure Both Directions​

Problem: Configured DELETE_CASCADE but forgot INSERT_CASCADE.

Scenario:

  • Delete order → items cascaded ✅
  • Create order → items not created ❌

Solution: Configure INSERT_CASCADE for default items:

'INSERT_CASCADE' => [
    'ORDITEM' => [...]
]

Pitfall 2: Cascade Performance on Large Datasets​

Problem: Deleting parent with 10,000 children times out.

Solution: Add pagination or async processing:

// Option 1: Batch delete children first
DELETE FROM TB_ANAG_ORDITEM00
WHERE XORDITEM01 = 555
LIMIT 1000;

// Option 2: Queue for background processing

Pitfall 3: Not Validating Cascade Permissions​

Problem: User has permission to delete parent but not children - cascade fails.

Solution: Validate permissions before cascade:

// Check user has grant for both ORD and ORDITEM
if (!hasGrant('ORD', 4) || !hasGrant('ORDITEM', 4)) {
    throw new PermissionException();
}

Summary​

Cascade Operations provide:

  1. ✅ Automatic relationship maintenance - No manual child record management
  2. ✅ Metadata-driven configuration - Define once in TB_COST
  3. ✅ ACID guarantees - All or nothing, no partial cascades
  4. ✅ Soft delete support - TREC='C' propagation preserves data
  5. ✅ Event publishing - Outbox pattern for every cascaded operation
  6. ✅ Flexible conditions - Extra WHERE clauses, field mappings

Key Insight:

Cascades are the referential integrity engine of Q01 Core APIs. Configure relationships in metadata, CoreWrite handles the rest - transactionally, consistently, with full event publishing.
