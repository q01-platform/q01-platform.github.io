Context Chain: The Request Fingerprint

What is the Context Chain?​

Every request to Q01 Core APIs carries a context chain - a set of parameters that determine:

  • Which fields are visible (COD_ON_OFF filtering)
  • Which data rows are accessible (ambiente filtering)
  • What operations are permitted (permission validation)
  • How data is formatted (language, labels)

Context is not optional - it travels with EVERY request and actively shapes the response.

The 6 Context Parameters​

┌──────────────────────────────────────┐
│        CONTEXT CHAIN                 │
├──────────────────────────────────────┤
│ 1. source        (COD_MENU)          │  ← Security area
│ 2. modulo        (module)            │  ← Application context
│ 3. microservice  (service)           │  ← Service identifier
│ 4. center_dett   (UI state)          │  ← Field visibility filter
│ 5. peso          (user weight)       │  ← Permission level
│ 6. ambiente      (environment)       │  ← Tenant isolation
└──────────────────────────────────────┘

Let's explore each parameter in detail.

1. source (Security Area)​

What it is: The menu/security area code from TB_MENU (COD_MENU).

Purpose: Defines the permission context for the operation.

Examples:

  • source=productManagement - Product management security area
  • source=salesDashboard - Sales dashboard area
  • source=adminPanel - Administrator panel

How it's used:

Permission Validation​

-- CoreWrite validates user has access to this source
SELECT LIVELLO_GRANT
FROM TB_MENU
WHERE COD_MENU = :source;  -- e.g., 'productManagement'

-- LIVELLO_GRANT:
-- 2 = Modify permission (PUT/PATCH allowed)
-- 4 = Create permission (POST allowed)

Grant Checking​

// In RequestValidatorSubscriber
$grantSession = $this->session->get($source, 0, 'grant_session');
if (!$grantUtils->checkGrant($grantSession, $requiredLevel)) {
    throw new Exception("Insufficient permissions for source: $source");
}

Required: YES (for all write operations, most read operations)

Example:

POST /api/v4/core/PRD
{
  "source": "productManagement",
  "XPRD01": "Widget Pro"
}

2. modulo (Module Context)​

What it is: The application module or context the request originates from.

Purpose: Enables module-specific behavior and metadata filtering.

Examples:

  • modulo=Default - Standard application
  • modulo=POS - Point of Sale application
  • modulo=B2B - B2B portal
  • modulo=Mobile - Mobile app

How it's used:

Dimension Filtering​

-- Load dimensions for specific module
SELECT *
FROM TB_DIM
WHERE MODULO = :modulo OR MODULO = 'Default';

Menu Filtering​

-- Load menus for specific module
SELECT *
FROM TB_MENU
WHERE MODULO = :modulo;

Multi-Module Support:

  • Different modules can have different menu structures
  • Same dimension can behave differently per module
  • Enables white-labeling and tenant-specific UIs

Default: Default (if not specified)

3. microservice (Service Identifier)​

What it is: Identifier of the microservice making the request.

Purpose: Tags operations for logging, outbox events, and service-specific configuration.

Examples:

  • microservice=mscoreservice - Core API service
  • microservice=orderservice - Order management service
  • microservice=pos-frontend - POS frontend

How it's used:

Outbox Tagging​

// Outbox records tagged with microservice
$outbox['XOUTBOX11'] = $microservice;  // "mscoreservice"

Session Context​

// Microservice-specific session namespace
$this->session->set('ambiente', $value, $microservice);

Logging​

[2025-12-19 15:30:42] [mscoreservice] POST /api/v4/core/PRD - user123

Required: NO (default from environment/config)

4. center_dett (UI State / Field Visibility)​

What it is: The current UI state or operation context - THE MOST CRITICAL CONTEXT PARAMETER.

Purpose: Filters which fields are visible based on COD_ON_OFF bitmask matching.

Standard Values:

center_dettMeaningCOD_ON_OFF FlagUse Case
visualizzaList viewLGrid/table display
dettaglioDetail viewDSingle record display
nuovoNew/Create formNCreate operation
edit / modificaEdit/Update formMUpdate operation
ricercaSearch formRSearch/filter form

How it works:

Field Filtering by center_dett​

-- When center_dett = 'visualizza' (list view)
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('L', COD_ON_OFF)  -- Only fields with 'L' flag
ORDER BY COD_SEQUENZA;

-- When center_dett = 'dettaglio' (detail view)
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND (FIND_IN_SET('D', COD_ON_OFF) OR FIND_IN_SET('L', COD_ON_OFF))
ORDER BY COD_SEQUENZA;

-- When center_dett = 'nuovo' (create form)
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('N', COD_ON_OFF)
ORDER BY COD_SEQUENZA;

Examples:

List View (visualizza)​

GET /api/v4/core/PRD?source=productList&center_dett=visualizza

-- Returns only fields with COD_ON_OFF containing 'L':
{
  "data": [
    {
      "PRD_ID": 1,
      "XPRD01": "Widget Pro",    // COD_ON_OFF='LDNMR' (has L)
      "XPRD02": 49.99            // COD_ON_OFF='LDNMR' (has L)
      // XPRD04 not included (COD_ON_OFF='D', no L)
    }
  ]
}

Detail View (dettaglio)​

GET /api/v4/core/PRD/1?source=productDetail&center_dett=dettaglio

-- Returns fields with 'D' OR 'L':
{
  "data": {
    "PRD_ID": 1,
    "XPRD01": "Widget Pro",           // COD_ON_OFF='LDNMR'
    "XPRD02": 49.99,                  // COD_ON_OFF='LDNMR'
    "XPRD04": "Full description"      // COD_ON_OFF='D' (detail only)
  }
}

Create Form (nuovo)​

GET /api/v4/core/PRD/fields?source=productForm&center_dett=nuovo

-- Returns only fields writable on create (COD_ON_OFF contains 'N'):
{
  "data": [
    {"COD_COST": "XPRD01", "DESC_COST": "Product Name", "CHECK_CAMPO": "not null"},
    {"COD_COST": "XPRD02", "DESC_COST": "Price", "CHECK_CAMPO": "not null"}
    // Fields with COD_ON_OFF='D' (detail-only) NOT included
  ]
}

Required: YES (for most operations)

Common Mistake:

❌ Wrong:
GET /api/v4/core/PRD?source=productList
// Missing center_dett - fields may not filter correctly

✅ Correct:
GET /api/v4/core/PRD?source=productList&center_dett=visualizza

5. peso (User Permission Weight)​

What it is: Numeric permission level from JWT token claims.

Purpose: Field-level access control - users only see fields their peso allows.

Range: 0-100 or * (higher = more permissions)

Examples:

  • peso=10 - Basic user
  • peso=50 - Manager
  • peso=90 - Administrator
  • peso=100 or peso=* - Super admin

How it's used:

Field Filtering by peso​

-- User with peso=30 queries products
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND (COD_UTENTE = '*' OR CAST(COD_UTENTE AS UNSIGNED) <= 30)
ORDER BY COD_SEQUENZA;

-- Fields with COD_UTENTE='*' → visible to everyone
-- Fields with COD_UTENTE='10' → visible to peso >= 10
-- Fields with COD_UTENTE='50' → visible only to peso >= 50 (HIDDEN for peso=30)

Example TB_COST Configuration:

COD_COSTDESC_COSTCOD_UTENTEVisible To
XPRD01Product Name*Everyone
XPRD02Sale Price*Everyone
XPRD_COSTCost Price50peso >= 50 only
XPRD_MARGINProfit Margin90peso >= 90 only

User with peso=30 sees:

{
  "PRD_ID": 1,
  "XPRD01": "Widget Pro",
  "XPRD02": 49.99
  // XPRD_COST and XPRD_MARGIN hidden
}

User with peso=90 sees:

{
  "PRD_ID": 1,
  "XPRD01": "Widget Pro",
  "XPRD02": 49.99,
  "XPRD_COST": 30.00,      // Now visible
  "XPRD_MARGIN": 19.99     // Now visible
}

Source: JWT token claim peso

Required: YES (from token, automatic)

6. ambiente (Environment Filter)​

What it is: Comma-separated list of environment IDs the user can access.

Purpose: Multi-tenant data isolation - ensures users only see data from their environments.

Examples:

  • ambiente=1 - Production environment only
  • ambiente=1,2 - Production and staging
  • ambiente=1,2,3 - All environments

How it's used:

Automatic WHERE Clause Injection​

-- CoreQuery automatically adds ambiente filter
SELECT *
FROM TB_ANAG_PRD00
WHERE FIND_IN_SET(:ambiente, XPRD01_AMBIENTE)

-- If ambiente='1,2':
WHERE FIND_IN_SET('1', XPRD01_AMBIENTE) OR FIND_IN_SET('2', XPRD01_AMBIENTE)

Data Storage:

Every record stores ambiente in X{DIM}01 field:

INSERT INTO TB_ANAG_PRD00 (
  XPRD01, XPRD02,
  XPRD01_AMBIENTE  -- Auto-populated from session
) VALUES (
  'Widget Pro', 49.99,
  '1,2'  -- Accessible in environments 1 and 2
);

Environment Isolation:

User A (ambiente='1'):

-- Can see records with XPRD01_AMBIENTE containing '1'
SELECT * FROM TB_ANAG_PRD00 WHERE FIND_IN_SET('1', XPRD01_AMBIENTE);
-- Results: Records with ambiente='1' or '1,2' or '1,3', etc.

User B (ambiente='2'):

-- Can see records with XPRD01_AMBIENTE containing '2'
SELECT * FROM TB_ANAG_PRD00 WHERE FIND_IN_SET('2', XPRD01_AMBIENTE);
-- Results: Records with ambiente='2' or '1,2' or '2,3', etc.

Field-Level Ambiente Filtering:

TB_COST can also filter fields by environment:

SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND (COD_AMBIENTE IS NULL OR FIND_IN_SET(:ambiente, COD_AMBIENTE))

Source: JWT token claim ambiente

Required: YES (from token, automatic)

Context Chain in Action​

Read Operation Example​

GET /api/v4/core/PRD?source=productList&center_dett=visualizza&$filter=XPRD02>50

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

JWT Token Claims:

{
  "userId": "user123",
  "peso": 50,
  "ambiente": "1,2",
  "grants": {...}
}

CoreQuery Processing:

  1. Extract context from request:

    • source = productList
    • center_dett = visualizza

  2. Extract context from JWT:

    • peso = 50
    • ambiente = 1,2

  3. Load visible fields:

SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('L', COD_ON_OFF)  -- center_dett='visualizza' → L flag
  AND (COD_UTENTE = '*' OR CAST(COD_UTENTE AS UNSIGNED) <= 50)  -- peso filtering
  AND (COD_AMBIENTE IS NULL OR FIND_IN_SET('1', COD_AMBIENTE) OR FIND_IN_SET('2', COD_AMBIENTE))
ORDER BY COD_SEQUENZA;
  1. Build query:
SELECT PRD_ID, XPRD01, XPRD02  -- Only visible fields
FROM TB_ANAG_PRD00
WHERE XPRD02 > 50  -- $filter
  AND (FIND_IN_SET('1', XPRD01_AMBIENTE) OR FIND_IN_SET('2', XPRD01_AMBIENTE))  -- ambiente
  1. Return filtered response:
{
  "data": [
    {
      "PRD_ID": 2,
      "XPRD01": "Gadget Plus",
      "XPRD02": 79.99
      // Only fields with COD_ON_OFF='L' and COD_UTENTE <= 50
    }
  ]
}

Context shaped EVERYTHING:

  • Which fields returned (center_dett + peso)
  • Which data rows returned (ambiente)
  • Permission validation (source)

Write Operation Example​

POST /api/v4/core/PRD
{
  "source": "productManagement",
  "XPRD01": "New Widget",
  "XPRD02": 59.99
}

Authorization: Bearer eyJ...

CoreWrite Processing:

  1. Validate source permission:
SELECT LIVELLO_GRANT FROM TB_MENU WHERE COD_MENU = 'productManagement';
-- Result: 4 (create permission required)
  1. Check user grant:
$grantSession = $session->get('productManagement', 0, 'grant_session');
if ($grantSession < 4) {
    throw new Exception("Grant level 4 required for create");
}
  1. Load writable fields (center_dett='nuovo'):
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('N', COD_ON_OFF)  -- Writable on create
  AND (COD_UTENTE = '*' OR CAST(COD_UTENTE AS UNSIGNED) <= :peso)
  1. Validate required fields:
-- Check XPRD01 and XPRD02 are present and valid
-- CHECK_CAMPO='not null' fields must be provided
  1. Insert with context:
INSERT INTO TB_ANAG_PRD00 (
  XPRD01, XPRD02,
  OWNER, CDATA, TREC,
  XPRD01_AMBIENTE  -- Auto-populated from JWT ambiente
) VALUES (
  'New Widget', 59.99,
  'user123', '20251219153042', 'N',
  '1,2'  -- From JWT: ambiente='1,2'
);

Context ensured:

  • User had permission (source + grant validation)
  • Only writable fields accepted (center_dett='nuovo')
  • Data tagged with user's ambiente (tenant isolation)

Context Chain Best Practices​

✅ DO:​

  • Always provide source for permission validation
  • Always provide center_dett for correct field visibility
  • Trust JWT claims (peso, ambiente) - don't override
  • Use standard center_dett values (visualizza, dettaglio, nuovo, edit, ricerca)
  • Document context requirements in API consumers

❌ DON'T:​

  • Skip source parameter (permission validation will fail)
  • Skip center_dett parameter (field visibility unpredictable)
  • Try to override JWT claims (server-side validation will reject)
  • Invent custom center_dett values without metadata support
  • Assume context is optional

Summary​

Context Chain is the fingerprint of every request:

ParameterSourcePurpose
sourceRequestSecurity area, permission validation
moduloRequest/ConfigModule context
microserviceConfigService identifier
center_dettRequestField visibility filter (CRITICAL)
pesoJWTField-level access control
ambienteJWTTenant data isolation (CRITICAL)

Key Insight:

Context doesn't just describe the request - it actively shapes what you see, what you can do, and what data is accessible.
