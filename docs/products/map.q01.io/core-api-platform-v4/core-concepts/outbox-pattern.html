Outbox Pattern

What is the Outbox Pattern?​

The Outbox Pattern is a write-ahead log mechanism that guarantees at-least-once delivery of data change events to distributed systems, even if message brokers fail during transactions.

Key Insight:

Every write operation (create, update, delete) in CoreWrite first writes to TB_ANAG_OUTBOX00 in the same ACID transaction as the data change. A separate relay process publishes outbox entries to RabbitMQ fanout exchanges.

This ensures no event is lost and eventual consistency across microservices.

Why the Outbox Pattern Matters​

The Dual-Write Problem​

Naive approach (fails):

// Step 1: Write to database
INSERT INTO TB_ANAG_PRD00 (...) VALUES (...);

// Step 2: Publish to RabbitMQ
$channel->basic_publish($msg, 'product.exchange');

Problem: If RabbitMQ is down or network fails after Step 1, event is lost - no retry, no recovery.

Consequences:

  • ❌ Search service never indexes new product
  • ❌ Analytics never receive event
  • ❌ Cache never invalidates
  • ❌ Data inconsistency across microservices

Outbox Pattern Solution​

Reliable approach (Q01 Core APIs):

$this->db_data->beginTransaction();

// Step 1: Write to database
INSERT INTO TB_ANAG_PRD00 (...) VALUES (...);

// Step 2: Write to outbox (SAME transaction)
INSERT INTO TB_ANAG_OUTBOX00 (
  XOUTBOX04,  -- Dimension (PRD)
  XOUTBOX05,  -- Operation (create)
  XOUTBOX06   -- Data (JSON)
) VALUES ('PRD', 'create', '{"PRD_ID": "123", ...}');

$this->db_data->commit();

// Step 3: Separate relay process publishes from outbox
// If RabbitMQ is down, outbox entry remains until published

Guarantees:

  • ✅ Atomic write - Data + outbox entry in single transaction
  • ✅ At-least-once delivery - Relay retries until acknowledged
  • ✅ No event loss - Outbox persists until successfully published
  • ✅ Eventual consistency - All subscribers eventually receive event

TB_ANAG_OUTBOX00 Structure​

Table Schema​

CREATE TABLE TB_ANAG_OUTBOX00 (
  OUTBOX_ID INT PRIMARY KEY AUTO_INCREMENT,

  -- Standard audit fields
  OWNER VARCHAR(50),
  CDATA VARCHAR(14),
  LOWNER VARCHAR(50),
  LDATA VARCHAR(14),
  TIMESTAMP BIGINT,
  TREC CHAR(1) DEFAULT 'N',

  -- Outbox payload fields
  XOUTBOX01 VARCHAR(255),   -- Ambiente (tenant isolation)
  XOUTBOX02 VARCHAR(255),   -- Ambiente riferimento
  XOUTBOX03 VARCHAR(64),    -- Event ID (SHA1 hash)
  XOUTBOX04 VARCHAR(50),    -- Aggregate (dimension code: PRD, ORD, etc.)
  XOUTBOX05 VARCHAR(50),    -- Operation (create, update, delete, logicalDelete)
  XOUTBOX06 TEXT,           -- Data (JSON payload)
  XOUTBOX07 TEXT,           -- Reserved
  XOUTBOX08 INT DEFAULT 0,  -- Retry count
  XOUTBOX09 VARCHAR(255),   -- Tenant ID
  XOUTBOX10 INT DEFAULT 0,  -- Published flag (0=pending, 1=published)
  XOUTBOX11 VARCHAR(100),   -- Microservice
  XOUTBOX12 VARCHAR(255),   -- Tenant ID (duplicate for indexing)
  XOUTBOX13 VARCHAR(255)    -- Resource
);

Key Fields Explained​

FieldPurposeExample
XOUTBOX03Unique event ID (SHA1)"a7f3c2b1..."
XOUTBOX04Dimension (aggregate root)"PRD", "ORD", "CLI"
XOUTBOX05Operation type"create", "update", "delete", "logicalDelete"
XOUTBOX06Event data (JSON){"PRD_ID": "123", "XPRD01": "Widget"}
XOUTBOX08Retry counterIncremented on publish failure
XOUTBOX10Published flag0 = pending, 1 = published
XOUTBOX09Tenant IDFor multi-tenant isolation
XOUTBOX11MicroserviceSource microservice identifier

Event ID Generation​

Unique Event ID (XOUTBOX03):

$nowSha = sha1(str_replace(".", "", microtime(true)));
$tenantSha = sha1($this->session["tenantId"]);
$msSha = sha1($this->session->get("current_microservice"));
$ID = sha1($tenantSha . "." . $msSha . "." . $nowSha);

Components:

  • Tenant ID hash
  • Microservice hash
  • Microsecond timestamp hash

Result: Globally unique event ID preventing duplicates.

Outbox Write Operations​

CREATE Operation​

Request:

POST /api/v4/core/PRD
{
  "XPRD01": "Premium Widget",
  "XPRD02": 49.99,
  "source": "productCreate"
}

Transaction:

BEGIN;

-- 1. Insert product
INSERT INTO TB_ANAG_PRD00 (XPRD01, XPRD02, OWNER, CDATA, LOWNER, LDATA, TREC)
VALUES ('Premium Widget', 49.99, 'user_5', '20250119120000', 'user_5', '20250119120000', 'N');

-- Returns PRD_ID = 123

-- 2. Insert outbox entry (SAME transaction)
INSERT INTO TB_ANAG_OUTBOX00 (
  OWNER, CDATA, LOWNER, LDATA, TIMESTAMP, TREC,
  XOUTBOX01, XOUTBOX02, XOUTBOX03, XOUTBOX04, XOUTBOX05,
  XOUTBOX06, XOUTBOX07, XOUTBOX08, XOUTBOX09, XOUTBOX10,
  XOUTBOX11, XOUTBOX12, XOUTBOX13
) VALUES (
  'user_5', '20250119120000', 'user_5', '20250119120000', '1737283200000000', 'N',
  'tenant_123', 'tenant_123_ref', 'a7f3c2b1...', 'PRD', 'create',
  '{"PRD_ID":"123","XPRD01":"Premium Widget","XPRD02":49.99}', '', 0, 'tenant_123', 0,
  'corewrite', 'tenant_123', 'production'
);

COMMIT;

Result:

  • Product created with PRD_ID=123
  • Outbox entry created with XOUTBOX10=0 (pending publish)

RabbitMQ Fanout Exchange​

Exchange Naming Convention​

Format:

{tenant_hash}.{tenant_ref_hash}.sync.exchange

Example:

$tenantHash = hash('crc32b', $outboxObj[0]["XOUTBOX15"]);  // Tenant ID
$tenantRefHash = hash('crc32b', $outboxObj[0]["XOUTBOX09"]);  // Tenant reference
$exchange = "{$tenantHash}.{$tenantRefHash}.sync.exchange";
// Result: "a3f2c1b4.d5e6f7g8.sync.exchange"

Purpose: Unique exchange per tenant for isolation.

Code Reference: DataStore.php:2978

Publishing to Exchange​

Relay Process:

// 1. Fetch pending outbox entries
SELECT * FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX10 = 0  -- Not yet published
ORDER BY OUTBOX_ID ASC
LIMIT 100;

// 2. For each entry, publish to RabbitMQ
foreach ($outboxEntries as $entry) {
    $exchange = hash('crc32b', $entry['XOUTBOX15']) . "." .
                hash('crc32b', $entry['XOUTBOX09']) . ".sync.exchange";

    $connection = new AMQPStreamConnection(
        $_ENV["hostMQ"],
        $_ENV["portMQ"],
        $_ENV["userMQ"],
        $_ENV["passwdMQ"],
        $environment
    );

    $channel = $connection->channel();
    $channel->exchange_declare($exchange, 'fanout', true, false, false);

    $msg = new AMQPMessage(json_encode($entry));
    $channel->basic_publish($msg, $exchange);

    $channel->close();
    $connection->close();

    // 3. Mark as published
    UPDATE TB_ANAG_OUTBOX00 SET XOUTBOX10 = 1 WHERE OUTBOX_ID = ?;
}

Code Reference: DataStore.php:2980-2993

Fanout Exchange Behavior​

Fanout = Broadcast to all bound queues:

                   ┌──────────────────┐
                   │  Fanout Exchange │
                   │ tenant_X.sync    │
                   └────────┬─────────┘
                            │
         ┌──────────────────┼──────────────────┐
         │                  │                  │
         v                  v                  v
    ┌─────────┐      ┌──────────┐      ┌─────────┐
    │ Queue 1 │      │ Queue 2  │      │ Queue 3 │
    │ Search  │      │Analytics │      │  Cache  │
    └─────────┘      └──────────┘      └─────────┘

Every consumer receives every event.

Why Fanout?

  • Simple pub/sub pattern
  • No routing logic required
  • Every microservice gets all dimension changes
  • Consumers filter events they care about

At-Least-Once Delivery​

Guarantee Mechanism​

Outbox Pattern ensures:

  1. Event persisted in database (ACID transaction)
  2. Relay reads pending events (XOUTBOX10=0)
  3. Publishes to RabbitMQ (retries on failure)
  4. Marks published (XOUTBOX10=1) only after success

If RabbitMQ is down:

  • Events remain in outbox (XOUTBOX10=0)
  • Relay retries every N seconds
  • No event loss

If database commits but relay crashes:

  • Events still in outbox (XOUTBOX10=0)
  • Relay restarts and publishes

Result: At-least-once delivery guaranteed.

Retry Logic​

Retry Counter (XOUTBOX08):

// On publish failure
UPDATE TB_ANAG_OUTBOX00 SET
  XOUTBOX08 = XOUTBOX08 + 1
WHERE OUTBOX_ID = ?;

Exponential backoff (typical pattern):

  • Retry 1: Wait 1 second
  • Retry 2: Wait 2 seconds
  • Retry 3: Wait 4 seconds
  • Retry N: Wait min(2^N, max_delay) seconds

Dead Letter Queue (DLQ): After N retries (e.g., 10), move to DLQ for manual intervention.

Idempotency and Deduplication​

Consumer Responsibility​

Consumers MUST handle duplicate events:

Example: Search Indexer

// Event received
{
  "XOUTBOX03": "a7f3c2b1...",
  "XOUTBOX04": "PRD",
  "XOUTBOX05": "create",
  "XOUTBOX06": {"PRD_ID": "123", "XPRD01": "Widget"}
}

// Check if already processed
const processed = await redis.get(`event:a7f3c2b1...`);
if (processed) {
  console.log("Duplicate event, skipping");
  return;  // Idempotent - safe to skip
}

// Index product
await elasticsearch.index({
  index: 'products',
  id: '123',
  body: { name: 'Widget' }
});

// Mark as processed
await redis.set(`event:a7f3c2b1...`, '1', 'EX', 86400);  // 24h TTL

Key Insight:

XOUTBOX03 (Event ID) enables deduplication. Consumers should track processed event IDs to achieve exactly-once semantics on their end.

Multi-Tenant Isolation​

Outbox per Tenant:

-- Tenant A's exchange
SELECT * FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX09 = 'tenant_a';
-- Publishes to: {hash_a}.{hash_a_ref}.sync.exchange

-- Tenant B's exchange
SELECT * FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX09 = 'tenant_b';
-- Publishes to: {hash_b}.{hash_b_ref}.sync.exchange

Result: Each tenant has isolated RabbitMQ exchange - no cross-tenant event leakage.

Outbox Cleanup​

Published Events:

After successful publish (XOUTBOX10=1), events can be archived or purged:

Option 1: Archive to cold storage

-- Move to archive table
INSERT INTO TB_ANAG_OUTBOX_ARCHIVE00
SELECT * FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX10 = 1
  AND LDATA < DATE_SUB(NOW(), INTERVAL 30 DAY);

-- Delete from active outbox
DELETE FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX10 = 1
  AND LDATA < DATE_SUB(NOW(), INTERVAL 30 DAY);

Option 2: Purge after retention period

-- Delete events older than 7 days
DELETE FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX10 = 1
  AND LDATA < DATE_SUB(NOW(), INTERVAL 7 DAY);

Best Practice: Keep recent published events for debugging, archive older ones.

Enabling/Disabling Outbox​

Environment Variable:

ENABLE_OUTBOX=true   # Enable outbox pattern
ENABLE_OUTBOX=false  # Disable (for testing or slave writes)

Code Reference: DataStore.php:3042-3043

$_ENV['ENABLE_OUTBOX'] = filter_var($_ENV['ENABLE_OUTBOX'], FILTER_VALIDATE_BOOLEAN);
if (isset($_ENV['ENABLE_OUTBOX']) && $_ENV['ENABLE_OUTBOX'] == true) {
    // Write to outbox
}

Slave Writes (Outbox Disabled):

POST /api/v4/core/PRD
{
  "XPRD01": "Widget",
  "outbox": "false"  // Skip outbox write
}

Use Case: Replaying events from master to slave - slave shouldn't republish.

Code Reference: DataStore.php:1263-1265

Best Practices​

✅ DO: Let CoreWrite Manage Outbox​

No manual outbox writes needed:

POST /api/v4/core/PRD
{
  "XPRD01": "Widget",
  "XPRD02": 49.99
}

CoreWrite automatically:

  • Inserts into TB_ANAG_PRD00
  • Inserts into TB_ANAG_OUTBOX00
  • Commits transaction
  • Relay publishes to RabbitMQ

✅ DO: Implement Idempotent Consumers​

Track processed event IDs:

const eventId = event.XOUTBOX03;
if (await isProcessed(eventId)) {
  return;  // Already handled
}

await processEvent(event);
await markProcessed(eventId);

✅ DO: Use Event Sourcing for Replay​

Query outbox for event history:

SELECT XOUTBOX04, XOUTBOX05, XOUTBOX06, LDATA
FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX04 = 'PRD'
  AND XOUTBOX05 IN ('create', 'update')
  AND LDATA > '20250101000000'
ORDER BY OUTBOX_ID ASC;

Rebuild read models from outbox events.

✅ DO: Monitor Outbox Lag​

Alert on high pending count:

SELECT COUNT(*) as pending_events
FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX10 = 0;

If pending_events > threshold: RabbitMQ may be down or relay not running.

❌ DON'T: Bypass Outbox Pattern​

Wrong:

// Skip outbox and publish directly
INSERT INTO TB_ANAG_PRD00 (...) VALUES (...);
$channel->basic_publish($msg, 'product.exchange');  // No ACID guarantee!

Why: Loses at-least-once delivery guarantee.

❌ DON'T: Assume Exactly-Once Delivery​

Misconception: Outbox pattern = exactly-once delivery.

Reality: Outbox pattern = at-least-once delivery.

Solution: Consumers must implement deduplication using XOUTBOX03.

❌ DON'T: Purge Outbox Too Aggressively​

Wrong:

-- Delete immediately after publish
DELETE FROM TB_ANAG_OUTBOX00 WHERE XOUTBOX10 = 1;

Why: Loses audit trail and replay capability.

Right:

-- Keep for 7-30 days
DELETE FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX10 = 1 AND LDATA < DATE_SUB(NOW(), INTERVAL 7 DAY);

Common Pitfalls​

Pitfall 1: Not Handling Duplicate Events​

Problem: Consumer processes same event twice, creates duplicate records.

Example:

// Wrong - not idempotent
async function onProductCreated(event) {
  await db.insert('products', event.XOUTBOX06);  // Duplicate insert on retry!
}

// Right - idempotent
async function onProductCreated(event) {
  const productId = event.XOUTBOX06.PRD_ID;
  await db.upsert('products', { id: productId }, event.XOUTBOX06);
}

Pitfall 2: Ignoring Retry Count​

Problem: Infinite retries on permanent failures.

Solution: Check XOUTBOX08 and move to DLQ after threshold:

-- Move events with >10 retries to dead letter queue
INSERT INTO TB_ANAG_OUTBOX_DLQ00
SELECT * FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX08 > 10 AND XOUTBOX10 = 0;

DELETE FROM TB_ANAG_OUTBOX00
WHERE XOUTBOX08 > 10 AND XOUTBOX10 = 0;

Pitfall 3: Relying on Event Order​

Problem: Expecting events in exact order.

Reality: Fanout exchanges don't guarantee order across consumers.

Solution: Use versioning or timestamps:

// Check event timestamp vs current state
if (event.LDATA <= currentProduct.LDATA) {
  console.log("Stale event, ignoring");
  return;
}

Summary​

Outbox Pattern provides:

  1. ✅ ACID guarantees - Data + event write in single transaction
  2. ✅ At-least-once delivery - No event loss, even if RabbitMQ fails
  3. ✅ Eventual consistency - All microservices eventually synchronized
  4. ✅ Audit trail - Full event history in TB_ANAG_OUTBOX00
  5. ✅ Replay capability - Rebuild systems from outbox events
  6. ✅ Multi-tenant isolation - Separate exchanges per tenant

Key Insight:

The Outbox Pattern is the consistency guarantee for distributed Q01 systems. Every write operation publishes an event - no exceptions, no bypasses. Consumers achieve exactly-once semantics through idempotency and deduplication.
