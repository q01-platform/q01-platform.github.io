Nested Set Model: TB_MENU Hierarchy

What is the Nested Set Model?​

Nested Set is a mathematical model for representing hierarchical tree structures in a relational database using two numbers (NLEFT and NRIGHT) instead of parent-child pointers.

In Q01, TB_MENU uses this model to represent security areas as a hierarchy where permissions and grants cascade from parent to child nodes.

Why Nested Set?​

Traditional parent-child relationships require recursive queries to traverse a tree:

-- Traditional approach (slow, recursive)
WITH RECURSIVE menu_tree AS (
  SELECT * FROM TB_MENU WHERE PARENT_ID = 'root'
  UNION ALL
  SELECT m.* FROM TB_MENU m
  JOIN menu_tree mt ON m.PARENT_ID = mt.COD_MENU
)
SELECT * FROM menu_tree;

Nested Set approach (fast, single query):

-- Get all descendants of a node
SELECT * FROM TB_MENU
WHERE NLEFT > :parentLeft AND NRIGHT < :parentRight
ORDER BY NLEFT;

One query. No recursion. Blazing fast.

TB_MENU Structure​

Key Fields:

FieldTypePurpose
COD_MENUvarchar(50)Security area code (e.g., "productManagement")
DESC_MENUvarchar(255)Human-readable description
NLEFTintLeft boundary of nested set
NRIGHTintRight boundary of nested set
NLEVELintDepth level in hierarchy (0=root)
MODULOvarchar(50)Module context (Default, POS, B2B)
COD_MENU_PADREvarchar(50)Parent menu code (for reference)
GRANTintRequired grant level (0-4)

How Nested Set Works​

The Left-Right Numbering​

Each node gets two numbers:

  • NLEFT: Assigned when entering the node during tree traversal
  • NRIGHT: Assigned when leaving the node
          [Products] (1, 12)
         /                  \
   [View] (2, 5)        [Manage] (6, 11)
      |                    /        \
   [List] (3, 4)   [Create] (7, 8)  [Edit] (9, 10)

Example TB_MENU Data:

COD_MENUDESC_MENUNLEFTNRIGHTNLEVELGRANT
productsProducts11200
productViewView Products2511
productListProduct List3421
productManageManage Products61112
productCreateCreate Product7824
productEditEdit Product91022

Visual Representation​

products (1, 12) GRANT=0
├─ productView (2, 5) GRANT=1
│  └─ productList (3, 4) GRANT=1
└─ productManage (6, 11) GRANT=2
   ├─ productCreate (7, 8) GRANT=4
   └─ productEdit (9, 10) GRANT=2

Key Properties:

  • Parent contains children: parent.NLEFT < child.NLEFT < child.NRIGHT < parent.NRIGHT
  • Siblings don't overlap: sibling1.NRIGHT < sibling2.NLEFT
  • Leaf nodes: NRIGHT = NLEFT + 1

Common Nested Set Queries​

1. Get All Descendants (Subtree)​

-- Get all nodes under "productManage"
SELECT child.*
FROM TB_MENU parent
JOIN TB_MENU child ON child.NLEFT > parent.NLEFT
                   AND child.NRIGHT < parent.NRIGHT
WHERE parent.COD_MENU = 'productManage'
ORDER BY child.NLEFT;

-- Result: productCreate, productEdit

2. Get All Ancestors (Path to Root)​

-- Get path from "productCreate" to root
SELECT parent.*
FROM TB_MENU child
JOIN TB_MENU parent ON child.NLEFT > parent.NLEFT
                    AND child.NRIGHT < parent.NRIGHT
WHERE child.COD_MENU = 'productCreate'
ORDER BY parent.NLEVEL;

-- Result: products, productManage

3. Get Direct Children Only​

-- Get immediate children of "products"
SELECT child.*
FROM TB_MENU parent
JOIN TB_MENU child ON child.NLEFT > parent.NLEFT
                   AND child.NRIGHT < parent.NRIGHT
                   AND child.NLEVEL = parent.NLEVEL + 1
WHERE parent.COD_MENU = 'products';

-- Result: productView, productManage

4. Get Leaf Nodes (No Children)​

-- Get all leaf nodes in "products" tree
SELECT child.*
FROM TB_MENU parent
JOIN TB_MENU child ON child.NLEFT > parent.NLEFT
                   AND child.NRIGHT < parent.NRIGHT
WHERE parent.COD_MENU = 'products'
  AND child.NRIGHT = child.NLEFT + 1;

-- Result: productList, productCreate, productEdit

5. Check if Node A Contains Node B​

-- Is "productCreate" inside "products"?
SELECT COUNT(*) > 0 as is_descendant
FROM TB_MENU parent, TB_MENU child
WHERE parent.COD_MENU = 'products'
  AND child.COD_MENU = 'productCreate'
  AND child.NLEFT > parent.NLEFT
  AND child.NRIGHT < parent.NRIGHT;

-- Result: TRUE

Security and Permissions​

Grant Levels​

TB_MENU nodes have GRANT values representing required permission level:

Grant LevelPermissionTypical Use
0PublicRead-only access
1ViewList/detail views
2ModifyUpdate existing records
3AdvancedSpecial operations
4CreateCreate new records

Permission Validation​

When a request arrives with source=productCreate:

  1. Look up source in TB_MENU:
SELECT COD_MENU, GRANT, NLEFT, NRIGHT
FROM TB_MENU
WHERE COD_MENU = 'productCreate';
-- Result: GRANT=4 required
  1. Check user's grant from JWT:
{
  "grants": {
    "products": 4,
    "productManage": 4
  }
}
  1. Validate permission:
    • User has "products": 4 grant
    • "productCreate" is descendant of "products" (NLEFT=7 is within 1-12)
    • User grant (4) >= Required grant (4)
    • ✅ Permission granted

Hierarchical Grant Inheritance​

If user has grant on parent, they have access to all descendants:

User has grant on "products" (NLEFT=1, NRIGHT=12)
↓
Can access:
- productView (2-5) ✅
- productList (3-4) ✅
- productManage (6-11) ✅
- productCreate (7-8) ✅
- productEdit (9-10) ✅

Query to check access:

-- Check if user can access "productCreate"
SELECT COUNT(*) > 0 as has_access
FROM TB_MENU node
WHERE node.COD_MENU = 'productCreate'
  AND EXISTS (
    -- User has grant on this node or any ancestor
    SELECT 1 FROM TB_MENU ancestor
    WHERE node.NLEFT >= ancestor.NLEFT
      AND node.NRIGHT <= ancestor.NRIGHT
      AND ancestor.COD_MENU IN ('products', 'productManage')  -- User's grants
      AND node.GRANT <= 4  -- User's grant level
  );

Source Parameter Mapping​

The source parameter in API requests maps directly to COD_MENU:

GET /api/v4/core/PRD?source=productList&center_dett=visualizza

CoreQuery processing:

  1. Extracts source=productList
  2. Queries TB_MENU for "productList" node
  3. Validates user has grant on "productList" or ancestor
  4. Uses GRANT level for additional permission checks

Modulo Context​

TB_MENU supports multiple hierarchies via MODULO field:

-- Default module hierarchy
SELECT * FROM TB_MENU WHERE MODULO = 'Default' ORDER BY NLEFT;

-- POS module hierarchy (different security areas)
SELECT * FROM TB_MENU WHERE MODULO = 'POS' ORDER BY NLEFT;

-- B2B module hierarchy
SELECT * FROM TB_MENU WHERE MODULO = 'B2B' ORDER BY NLEFT;

Each module has independent security tree.

Common Patterns​

Pattern 1: Read-Only Public Area​

publicProducts (GRANT=0)
└─ publicProductList (GRANT=0)

Anyone can view, no authentication required for GRANT=0.

Pattern 2: Standard CRUD Hierarchy​

orderManagement (GRANT=1)
├─ orderList (GRANT=1)
├─ orderView (GRANT=1)
├─ orderCreate (GRANT=4)
├─ orderEdit (GRANT=2)
└─ orderDelete (GRANT=3)

Separate grants for different operations.

Pattern 3: Progressive Disclosure​

reports (GRANT=1)
├─ basicReports (GRANT=1)
└─ advancedReports (GRANT=3)
    ├─ financialReports (GRANT=4)
    └─ auditReports (GRANT=4)

More sensitive reports require higher grants.

Common Mistakes​

❌ DON'T: Modify NLEFT/NRIGHT Manually​

-- WRONG - breaks tree structure
UPDATE TB_MENU SET NLEFT = 15 WHERE COD_MENU = 'newNode';

Why: Nested set requires recalculating all affected nodes when inserting/deleting.

❌ DON'T: Create Overlapping Ranges​

-- WRONG - node1 and node2 overlap
INSERT INTO TB_MENU (COD_MENU, NLEFT, NRIGHT) VALUES ('node1', 5, 10);
INSERT INTO TB_MENU (COD_MENU, NLEFT, NRIGHT) VALUES ('node2', 8, 12);

Why: Violates nested set invariant (siblings must not overlap).

❌ DON'T: Assume source is Optional​

GET /api/v4/core/PRD?center_dett=visualizza

Why: source is required for permission validation via TB_MENU.

Best Practices​

✅ DO: Use Consistent Naming​

{module}{Entity}{Action}
productList, productCreate, productEdit
orderList, orderCreate, orderEdit

✅ DO: Assign Grants Conservatively​

  • GRANT=0: Public data
  • GRANT=1: View operations
  • GRANT=2: Update operations
  • GRANT=3: Delete operations
  • GRANT=4: Create operations

Higher grants for more sensitive operations.

✅ DO: Group Related Areas​

products (GRANT=0)
├─ productCatalog (GRANT=0)
│  ├─ productList (GRANT=0)
│  └─ productSearch (GRANT=0)
└─ productManagement (GRANT=2)
   ├─ productCreate (GRANT=4)
   └─ productEdit (GRANT=2)

Logical grouping makes permissions intuitive.

Summary​

Nested Set Model in TB_MENU provides:

  1. ✅ Fast hierarchical queries - single query, no recursion
  2. ✅ Efficient permission validation - check if node in subtree
  3. ✅ Grant inheritance - parent grants cascade to children
  4. ✅ Flexible security areas - multiple hierarchies via MODULO

Key Insight:

TB_MENU is not just a menu structure - it's the security backbone of Q01 Core APIs. Every source parameter validates against this hierarchy.
