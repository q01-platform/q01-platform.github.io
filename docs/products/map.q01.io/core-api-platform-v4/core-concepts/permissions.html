Multi-Level Permissions: Defense in Depth

Overview​

Q01 Core APIs implement a multi-level permission model where access is validated at 5 independent layers.

Unlike traditional systems with single-point permission checks, Q01 enforces security at multiple levels simultaneously. A request must pass ALL layers to succeed.

┌─────────────────────────────────────────┐
│   MULTI-LEVEL PERMISSION MODEL          │
├─────────────────────────────────────────┤
│                                          │
│  Layer 1: JWT Token Validation          │  ← Valid signature, not expired
│           ↓ PASS                         │
│  Layer 2: Grant Level (source)          │  ← User has access to security area
│           ↓ PASS                         │
│  Layer 3: Field-Level Access (peso)     │  ← User weight allows field visibility
│           ↓ PASS                         │
│  Layer 4: Context Visibility (COD_ON_OFF)│  ← Field visible in current context
│           ↓ PASS                         │
│  Layer 5: Data Isolation (ambiente)     │  ← User can access this environment
│           ↓ PASS                         │
│                                          │
│  ✅ Request Succeeds                     │
│                                          │
└─────────────────────────────────────────┘

No single permission failure exposes data - this is defense in depth.

Layer 1: JWT Token Validation​

What: Cryptographic verification of authentication token.

When: Every request, before any other processing.

Enforced By: TokenValidatorSubscriber (both CoreQuery and CoreWrite)

How it works:

Token Structure​

Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJwcm9maWxlIjoiZGV2ZWxvcGVyIiwicGVzbyI6NTAwLCJ0ZW5hbnRJZCI6IjAwMDAwUTAxLTAwMDAtMDAwMS0wMDAwLTAwMENVU1RPTUVSIiwidXNlcklkIjoidXNlcjEyMyIsImFtYmllbnRlIjoiMSwyLDMiLCJncmFudHMiOnt9fQ.signature

Decoded payload:

{
  "profile": "developer",
  "peso": 50,
  "tenantId": "00000Q01-0000-0001-0000-000CUSTOMER",
  "userId": "user123",
  "ambiente": "1,2,3",
  "lingua": "it",
  "grants": {
    "productManagement": 4,
    "salesDashboard": 2
  }
}

Validation Steps​

  1. Signature Verification:
use Firebase\JWT\JWT;

$decoded = JWT::decode($token, $secretKey, ['HS256']);
// Throws exception if signature invalid
  1. Expiration Check:
if ($decoded->exp < time()) {
    throw new Exception("Token expired");
}
  1. Required Claims:
$requiredClaims = ['userId', 'tenantId', 'peso', 'ambiente'];
foreach ($requiredClaims as $claim) {
    if (!isset($decoded->$claim)) {
        throw new Exception("Missing required claim: $claim");
    }
}
  1. Session Population:
// Store claims in session for later use
$session->set('userId', $decoded->userId);
$session->set('tenantId', $decoded->tenantId);
$session->set('peso', $decoded->peso);
$session->set('ambiente', $decoded->ambiente);
$session->set('profile', $decoded->profile);

Bypass: Special profiles bypass validation:

  • profile=superadmin - skips ALL validation (SKIP_ROUTE_VALIDATION)
  • profile=service account - skips validation for service-to-service calls

Result: ✅ Pass → Continue to Layer 2 | ❌ Fail → 401 Unauthorized

Layer 2: Grant Level Validation (source)​

What: Validates user has permission to access the requested security area.

When: Write operations (POST/PUT/PATCH/DELETE) and most read operations.

Enforced By: RequestValidatorSubscriber in CoreWrite, grant checks in CoreQuery

How it works:

Grant System​

TB_MENU defines security areas with required grant levels:

SELECT COD_MENU, DESC_MENU, LIVELLO_GRANT
FROM TB_MENU
WHERE COD_MENU = 'productManagement';

-- Result:
-- COD_MENU: productManagement
-- LIVELLO_GRANT: 4  (create permission required)

Grant Levels:

  • Level 2: Modify permission (PUT/PATCH allowed)
  • Level 4: Create permission (POST allowed, implies level 2)

Grant Checking​

// Extract source from request
$source = $requestData['source'];  // e.g., "productManagement"

// Get user's grant for this source from JWT
$grantSession = $jwtClaims['grants'][$source] ?? 0;

// Get required grant level from TB_MENU
$requiredGrant = TB_MENU::where('COD_MENU', $source)->value('LIVELLO_GRANT');

// Validate
if ($grantSession < $requiredGrant) {
    throw new Exception("Insufficient permissions. Required grant level: $requiredGrant, User has: $grantSession");
}

Examples:

Create Operation (Requires Grant 4)​

POST /api/v4/core/PRD
{
  "source": "productManagement"
  // ...
}

JWT grants:
{
  "productManagement": 4  ✅ PASS (has level 4)
}

Update Operation (Requires Grant 2)​

PUT /api/v4/core/PRD/1
{
  "source": "productManagement"
  // ...
}

JWT grants:
{
  "productManagement": 2  ✅ PASS (has level 2)
}

Insufficient Grant​

POST /api/v4/core/PRD
{
  "source": "productManagement"
  // ...
}

JWT grants:
{
  "productManagement": 2  ❌ FAIL (needs level 4 for create)
}

Response:
{
  "status": 403,
  "message": "Insufficient permissions. Required grant level: 4, User has: 2"
}

Bypass:

  • skipSourceCheck=true in request (only for superadmin/service accounts)
  • sysuser=true in session (system user bypass)

Result: ✅ Pass → Continue to Layer 3 | ❌ Fail → 403 Forbidden

Layer 3: Field-Level Access (peso)​

What: Validates user's peso (permission weight) allows access to specific fields.

When: Loading field metadata, building queries, filtering responses.

Enforced By: CoreQuery and CoreWrite when loading TB_COST

How it works:

Field Access Control via COD_UTENTE​

TB_COST defines minimum peso required to see each field:

SELECT COD_COST, DESC_COST, COD_UTENTE
FROM TB_COST
WHERE DIM_COD = 'PRD';

-- Results:
-- COD_COST      | DESC_COST     | COD_UTENTE
-- XPRD01        | Product Name  | *          (everyone)
-- XPRD02        | Sale Price    | *          (everyone)
-- XPRD_COST     | Cost Price    | 50         (peso >= 50 only)
-- XPRD_MARGIN   | Profit Margin | 90         (peso >= 90 only)

COD_UTENTE Values:

  • * - Everyone (no peso restriction)
  • 10, 50, 90, etc. - Minimum peso required (range 0-100)

Filtering Query​

-- User with peso=30 queries fields
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND (COD_UTENTE = '*' OR CAST(COD_UTENTE AS UNSIGNED) <= 30)
ORDER BY COD_SEQUENZA;

-- Returns:
-- XPRD01 (COD_UTENTE='*') ✅
-- XPRD02 (COD_UTENTE='*') ✅
-- XPRD_COST (COD_UTENTE='50') ❌ FILTERED OUT
-- XPRD_MARGIN (COD_UTENTE='90') ❌ FILTERED OUT

Response Filtering​

User with peso=30:

GET /api/v4/core/PRD/1

Response:
{
  "data": {
    "PRD_ID": 1,
    "XPRD01": "Widget Pro",
    "XPRD02": 49.99
    // XPRD_COST hidden (requires peso >= 50)
    // XPRD_MARGIN hidden (requires peso >= 90)
  }
}

User with peso=90:

GET /api/v4/core/PRD/1

Response:
{
  "data": {
    "PRD_ID": 1,
    "XPRD01": "Widget Pro",
    "XPRD02": 49.99,
    "XPRD_COST": 30.00,     ✅ Now visible
    "XPRD_MARGIN": 19.99    ✅ Now visible
  }
}

Write Validation:

// User tries to set field they don't have access to
POST /api/v4/core/PRD
{
  "source": "productManagement",
  "XPRD01": "Widget",
  "XPRD_COST": 25.00  ❌ User has peso=30, field requires 50
}

// Validation fails:
throw new Exception("Field XPRD_COST requires peso >= 50, user has peso 30");

Result: ✅ Pass → Field visible/writable | ❌ Fail → Field filtered out (silently)

Layer 4: Context Visibility (COD_ON_OFF)​

What: Validates field is visible in current operation context (center_dett).

When: Loading fields, validating write operations.

Enforced By: CoreQuery and CoreWrite when filtering fields

How it works:

See Field Visibility - COD_ON_OFF for full details.

Context Filtering​

-- For center_dett='visualizza' (list view)
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('L', COD_ON_OFF)  -- Must have 'L' flag
  AND (COD_UTENTE = '*' OR CAST(COD_UTENTE AS UNSIGNED) <= :peso)  -- Layer 3
ORDER BY COD_SEQUENZA;

-- For center_dett='nuovo' (create form)
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('N', COD_ON_OFF)  -- Must have 'N' flag
  AND (COD_UTENTE = '*' OR CAST(COD_UTENTE AS UNSIGNED) <= :peso)  -- Layer 3
ORDER BY COD_SEQUENZA;

Layering with peso:

Field must pass BOTH COD_ON_OFF and peso filters:

Field: XPRD_COST
- COD_ON_OFF = 'LDM'  (list, detail, modify)
- COD_UTENTE = '50'  (requires peso >= 50)

User A (peso=90, center_dett='visualizza'):
✅ Has peso >= 50 (Layer 3 pass)
✅ COD_ON_OFF has 'L' (Layer 4 pass)
→ Field VISIBLE

User B (peso=30, center_dett='visualizza'):
❌ Peso < 50 (Layer 3 FAIL)
→ Field HIDDEN (Layer 3 filters it before Layer 4 checks)

User C (peso=90, center_dett='nuovo'):
✅ Has peso >= 50 (Layer 3 pass)
❌ COD_ON_OFF missing 'N' (Layer 4 FAIL)
→ Field HIDDEN (not writable on create)

Result: ✅ Pass → Field included | ❌ Fail → Field filtered out

Layer 5: Data Isolation (ambiente)​

What: Validates user can access data from the requested environment.

When: Every data query, automatically injected into WHERE clause.

Enforced By: CoreQuery and CoreWrite when building SQL

How it works:

Ambiente in JWT​

{
  "ambiente": "1,2,3"  // User can access environments 1, 2, and 3
}

Ambiente in Data​

Every record stores ambiente in X{DIM}01 field:

SELECT PRD_ID, XPRD01, XPRD01_AMBIENTE
FROM TB_ANAG_PRD00;

-- Results:
-- PRD_ID | XPRD01      | XPRD01_AMBIENTE
-- 1      | Widget Pro  | 1          (prod only)
-- 2      | Widget Dev  | 2          (staging only)
-- 3      | Widget Test | 1,2,3      (all environments)

Automatic Filtering​

CoreQuery injects ambiente filter into ALL queries:

-- User request:
GET /api/v4/core/PRD

-- User's ambiente from JWT: '1,2'

-- CoreQuery generates:
SELECT PRD_ID, XPRD01, XPRD02
FROM TB_ANAG_PRD00
WHERE FIND_IN_SET('1', XPRD01_AMBIENTE) OR FIND_IN_SET('2', XPRD01_AMBIENTE)
-- Filters to records accessible in environments 1 or 2

User sees:

{
  "data": [
    {"PRD_ID": 1, "XPRD01": "Widget Pro"},      // ambiente='1' ✅
    {"PRD_ID": 3, "XPRD01": "Widget Test"}      // ambiente='1,2,3' ✅
    // Widget Dev (ambiente='2' only) ✅ also visible
  ]
}

User with ambiente='3' sees:

{
  "data": [
    {"PRD_ID": 3, "XPRD01": "Widget Test"}  // ambiente='1,2,3' ✅
    // Widget Pro (ambiente='1') ❌ HIDDEN
    // Widget Dev (ambiente='2') ❌ HIDDEN
  ]
}

Write Tagging​

CoreWrite automatically tags new records with user's ambiente:

// User creates product with ambiente='1,2'
POST /api/v4/core/PRD
{
  "source": "productManagement",
  "XPRD01": "New Widget"
}

// CoreWrite generates:
INSERT INTO TB_ANAG_PRD00 (
  XPRD01,
  XPRD01_AMBIENTE  -- Auto-populated
) VALUES (
  'New Widget',
  '1,2'  // From JWT ambiente
);

Result: ✅ Pass → Data visible | ❌ Fail → Data filtered out (silently)

How Layers Work Together​

Example: Product Query​

Request:

GET /api/v4/core/PRD/1?source=productDetail&center_dett=dettaglio

Authorization: Bearer eyJ...

JWT Claims:

{
  "userId": "user123",
  "peso": 50,
  "ambiente": "1,2",
  "profile": "manager",
  "grants": {
    "productDetail": 2
  }
}

Validation Flow:

Layer 1: JWT Validation
├─ Verify signature ✅
├─ Check expiration ✅
├─ Extract claims ✅
└─ Result: PASS

Layer 2: Grant Validation
├─ source = 'productDetail'
├─ Required grant: 2 (from TB_MENU)
├─ User grant: 2 (from JWT)
└─ Result: PASS (2 >= 2)

Layer 3: Field-Level Access (peso=50)
├─ Load TB_COST for DIM='PRD'
├─ Filter: COD_UTENTE='*' OR COD_UTENTE <= 50
├─ Fields visible: XPRD01(*), XPRD02(*), XPRD_COST(50)
├─ Fields hidden: XPRD_MARGIN(90)
└─ Result: PASS (some fields filtered)

Layer 4: Context Visibility (center_dett='dettaglio')
├─ Filter remaining fields by COD_ON_OFF
├─ Must have 'D' or 'L' flag
├─ XPRD01 (COD_ON_OFF='L,D,N,M,R') ✅
├─ XPRD02 (COD_ON_OFF='L,D,N,M,R') ✅
├─ XPRD_COST (COD_ON_OFF='LDM') ✅
└─ Result: PASS

Layer 5: Data Isolation (ambiente='1,2')
├─ Build WHERE clause
├─ WHERE FIND_IN_SET('1', XPRD01_AMBIENTE) OR FIND_IN_SET('2', XPRD01_AMBIENTE)
├─ Record 1: XPRD01_AMBIENTE='1' ✅
└─ Result: PASS

Final Query:
SELECT PRD_ID, XPRD01, XPRD02, XPRD_COST
FROM TB_ANAG_PRD00
WHERE PRD_ID = 1
  AND (FIND_IN_SET('1', XPRD01_AMBIENTE) OR FIND_IN_SET('2', XPRD01_AMBIENTE))

Response:
{
  "data": {
    "PRD_ID": 1,
    "XPRD01": "Widget Pro",
    "XPRD02": 49.99,
    "XPRD_COST": 30.00
    // XPRD_MARGIN filtered by peso (Layer 3)
  }
}

All 5 layers enforced independently - failure at any layer blocks access.

Permission Debugging​

Common Permission Failures​

1. 401 Unauthorized - Layer 1 Failure​

{
  "status": 401,
  "message": "Invalid or expired token"
}

Cause: JWT signature invalid, token expired, or missing required claims.

Fix: Refresh JWT token, ensure all claims present.

2. 403 Forbidden - Layer 2 Failure​

{
  "status": 403,
  "message": "Insufficient permissions. Required grant level: 4, User has: 2"
}

Cause: User's grant for source is insufficient.

Fix: Ensure JWT contains correct grant level for the source, or use different source.

3. Empty Response - Layer 3 or Layer 5 Failure​

{
  "data": []
}

Cause: All fields filtered by peso (Layer 3) or all data filtered by ambiente (Layer 5).

Fix:

  • Layer 3: Increase user's peso or lower field's COD_UTENTE
  • Layer 5: Add user's ambiente to record's XDIM01_AMBIENTE

4. Missing Fields - Layer 3 or Layer 4 Failure​

{
  "data": {
    "PRD_ID": 1,
    "XPRD01": "Widget"
    // XPRD_COST missing
  }
}

Cause: Field filtered by peso or COD_ON_OFF.

Fix:

  • Layer 3: Increase user's peso
  • Layer 4: Add appropriate flag to COD_ON_OFF or use correct center_dett

5. Validation Error - Layer 4 Failure on Write​

{
  "status": 400,
  "message": "Field XPRD_COST not writable on create (missing 'N' in COD_ON_OFF)"
}

Cause: Trying to write field not allowed in current context.

Fix: Remove field from request or add appropriate COD_ON_OFF flag.

Summary​

Q01's Multi-Level Permission Model:

LayerWhatEnforced ByFailure Result
1. JWT ValidationToken signature, expirationTokenValidatorSubscriber401 Unauthorized
2. Grant LevelSource permission (2 or 4)RequestValidatorSubscriber403 Forbidden
3. Field Access (peso)User weight vs COD_UTENTEField loading in ModulesSilent field filtering
4. Context VisibilityCOD_ON_OFF vs center_dettField loading in ModulesSilent field filtering
5. Data IsolationAmbiente matchingSQL WHERE clauseSilent data filtering

Defense in Depth:

  • No single layer failure exposes unauthorized data
  • Each layer independently validated
  • Attackers must bypass ALL layers to gain access
  • Failures at different layers produce different errors (aids debugging)

Best Practices:

  1. ✅ Include all required JWT claims
  2. ✅ Set appropriate grants in JWT for user's role
  3. ✅ Configure COD_UTENTE based on data sensitivity
  4. ✅ Use correct COD_ON_OFF flags for field visibility
  5. ✅ Tag data with appropriate ambiente values
