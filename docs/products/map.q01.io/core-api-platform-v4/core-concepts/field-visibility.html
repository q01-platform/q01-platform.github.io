Field Visibility: The COD_ON_OFF Bitmask

What is COD_ON_OFF?​

COD_ON_OFF is a compact string bitmask in TB_COST that controls when and where a field is visible.

Instead of creating separate field lists for each UI context, Q01 uses a single bitmask where each letter represents a context:

COD_ON_OFF = "L,D,N,M,R"

Means: Show field in List, Detail, New, Modify, Ricerca (search).

This is the #1 source of confusion for new developers - but once mastered, it's incredibly powerful.

The COD_ON_OFF Flags​

FlagMeaningContext (center_dett)Use Case
LLista (List)visualizza, modifica, eliminaGrid/table display
DDettaglio (Detail)dettaglio, dettaglioProdotto, edit, nuovoSingle record view
NNuovo (New)nuovo, donuovoCreate form
MModifica (Modify)edit, modifica, doeditUpdate form
RRicerca (Search)ricerca, doricercaSearch/filter form
XHidden-Included in data but not displayed
HHeader-Header field in master-detail
IHidden-Completely hidden from all contexts (ambiente)

How It Works​

Example Field Configurations​

-- Product Name - visible everywhere
UPDATE TB_COST SET COD_ON_OFF = 'L,D,N,M,R' WHERE COD_COST = 'XPRD01';

-- Price - visible in list, detail, and modify (NOT in search or new)
UPDATE TB_COST SET COD_ON_OFF = 'LDM' WHERE COD_COST = 'XPRD02';

-- Internal Notes - detail view only
UPDATE TB_COST SET COD_ON_OFF = 'D' WHERE COD_COST = 'XPRD14';

-- Auto-generated Code - list and detail only (not editable)
UPDATE TB_COST SET COD_ON_OFF = 'LD' WHERE COD_COST = 'XPRD03';

-- Timestamp - hidden but included in data
UPDATE TB_COST SET COD_ON_OFF = 'X' WHERE COD_COST = 'TIMESTAMP';

Filtering Logic​

CoreQuery/CoreWrite filter fields using FIND_IN_SET():

-- For center_dett = 'visualizza' (list view), show fields with 'L'
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('L', COD_ON_OFF)
ORDER BY COD_SEQUENZA;

-- For center_dett = 'dettaglio' (detail view), show fields with 'D' OR 'L'
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND (FIND_IN_SET('D', COD_ON_OFF) OR FIND_IN_SET('L', COD_ON_OFF))
ORDER BY COD_SEQUENZA;

-- For center_dett = 'nuovo' (create form), show fields with 'N'
SELECT *
FROM TB_COST
WHERE DIM_COD = 'PRD'
  AND FIND_IN_SET('N', COD_ON_OFF)
ORDER BY COD_SEQUENZA;

Mapping center_dett to Flags​

Read Operations (GET)​

center_dett ValueMatches COD_ON_OFFLogic
visualizzaLList view
modificaLList view (before edit)
eliminaLList view (before delete)
dettaglioD or LDetail view includes list fields
dettaglioProdottoD or LDetail view variant
ricercaRSearch form
doricercaRSearch execution

Write Operations (POST/PUT/PATCH)​

center_dett ValueMatches COD_ON_OFFLogic
nuovoNCreate form
donuovoNCreate execution
editMUpdate form
modificaMUpdate form variant
doeditMUpdate execution

Special Cases​

X (Hidden):

  • Field is NOT displayed in UI
  • Field IS included in API response data
  • Use for: timestamps, internal IDs, system fields

H (Header):

  • Master record field in master-detail relationships
  • Used in HD_DIM (header/detail) metadata

I (Hidden):

  • Field is COMPLETELY hidden from all contexts
  • Field is NOT included in API responses (unless explicitly requested)
  • NOT visible in any UI (list, detail, create, edit, search)
  • Use for: ambiente tracking fields (XDIM01, XDIM02), system-managed fields
  • Example: XART01 (amb), XART02 (amb_rif) in every dimension

Common Patterns​

Pattern 1: Read-Only Field​

Scenario: Field visible everywhere but never editable (e.g., auto-generated code).

UPDATE TB_COST
SET COD_ON_OFF = 'LD'  -- List + Detail only (no N or M)
WHERE COD_COST = 'XPRD21';

Result:

  • ✅ Visible in list view
  • ✅ Visible in detail view
  • ❌ NOT in create form (can't set on create)
  • ❌ NOT in edit form (can't modify)

Pattern 2: Create-Only Field​

Scenario: Field settable on create but not editable afterward (e.g., initial status).

UPDATE TB_COST
SET COD_ON_OFF = 'LDN'  -- List + Detail + New (no M)
WHERE COD_COST = 'XPRD27';

Result:

  • ✅ Set during create (N flag)
  • ✅ Visible in list and detail
  • ❌ NOT editable after creation (no M flag)

Pattern 3: Edit-Only Field​

Scenario: Field not set on create but editable afterward (e.g., approval status).

UPDATE TB_COST
SET COD_ON_OFF = 'LDM'  -- List + Detail + Modify (no N)
WHERE COD_COST = 'XPRD24';

Result:

  • ❌ NOT in create form (no N flag)
  • ✅ Editable in update form (M flag)
  • ✅ Visible in list and detail

Pattern 4: Fully Visible and Editable​

Scenario: Standard field visible and editable everywhere.

UPDATE TB_COST
SET COD_ON_OFF = 'L,D,N,M,R'  -- Everything
WHERE COD_COST = 'XPRD01';

Result:

  • ✅ List view
  • ✅ Detail view
  • ✅ Create form
  • ✅ Edit form
  • ✅ Search form

Pattern 5: Search-Only Field​

Scenario: Field used for filtering but not displayed in results.

UPDATE TB_COST
SET COD_ON_OFF = 'R'  -- Search only
WHERE COD_COST = 'XPRD19';

Result:

  • ✅ Available in search form
  • ❌ NOT in list view
  • ❌ NOT in detail view
  • ❌ NOT editable

Pattern 6: Internal/System Field​

Scenario: Field for system use, not displayed but needed in data.

UPDATE TB_COST
SET COD_ON_OFF = 'X'  -- Hidden
WHERE COD_COST = 'TIMESTAMP';

Result:

  • ❌ NOT displayed in any UI
  • ✅ Included in API response data
  • ✅ Used for internal logic (e.g., optimistic locking)

Pattern 7: Ambiente Tracking Fields (Hidden)​

Scenario: Platform-managed fields for multi-tenant environment tracking.

-- Every dimension has these two hidden fields
UPDATE TB_COST
SET COD_ON_OFF = 'I'  -- Hidden (completely hidden from UI and API)
WHERE COD_COST IN ('XART01', 'XART02');  -- amb, amb_rif

-- Or in TB_COST definition:
INSERT INTO TB_COST (COD_DIM, COD_COST, DESC_COST, TIPO_CAMPO, COD_ON_OFF, COD_SEQUENZA)
VALUES
('ART', 'XART01', 'amb', 'text|11', 'I', 1),
('ART', 'XART02', 'amb_rif', 'text|11', 'I', 2);

Result:

  • ❌ NOT displayed in any UI context
  • ❌ NOT included in API responses (filtered out automatically)
  • ❌ NOT writable by users
  • ✅ Managed automatically by Core APIs
  • ✅ Used for tenant isolation and context tracking

Why 'I' Instead of 'X':

  • 'X' fields are hidden but still included in responses
  • 'I' fields are completely removed from responses
  • 'I' ensures ambiente fields never leak to frontend

Real-World Example​

Product Dimension Field Configuration​

-- Ambiente tracking fields - hidden (every dimension has these)
UPDATE TB_COST SET COD_ON_OFF = 'I' WHERE COD_COST = 'XPRD01';  -- amb (ambiente)
UPDATE TB_COST SET COD_ON_OFF = 'I' WHERE COD_COST = 'XPRD02';  -- amb_rif (ambiente reference)

-- Auto-generated code - visible but not editable
UPDATE TB_COST SET COD_ON_OFF = 'LD' WHERE COD_COST = 'XPRD03';  -- Product Code

-- Core fields - always visible and editable
UPDATE TB_COST SET COD_ON_OFF = 'L,D,N,M,R' WHERE COD_COST = 'XPRD04';  -- Name
UPDATE TB_COST SET COD_ON_OFF = 'L,D,N,M,R' WHERE COD_COST = 'XPRD04';  -- Description
UPDATE TB_COST SET COD_ON_OFF = 'L,D,N,M,R' WHERE COD_COST = 'XPRD05';  -- Price

-- Initial category - set on create, not editable
UPDATE TB_COST SET COD_ON_OFF = 'LDN' WHERE COD_COST = 'XPRD28';

-- Approval fields - not set on create, editable later
UPDATE TB_COST SET COD_ON_OFF = 'LDM' WHERE COD_COST = 'XPRD26';
UPDATE TB_COST SET COD_ON_OFF = 'LDM' WHERE COD_COST = 'XPRD25';

-- Internal notes - detail view only (sensitive info)
UPDATE TB_COST SET COD_ON_OFF = 'D' WHERE COD_COST = 'XPRD18';

-- Search tags - search form only
UPDATE TB_COST SET COD_ON_OFF = 'R' WHERE COD_COST = 'XPRD19';

-- System fields - hidden but included in response
UPDATE TB_COST SET COD_ON_OFF = 'X' WHERE COD_COST = 'TIMESTAMP';
UPDATE TB_COST SET COD_ON_OFF = 'X' WHERE COD_COST = 'TREC';

Validation and COD_ON_OFF​

Create Validation (POST)​

CoreWrite validates that ONLY fields with 'N' flag are provided:

// Load writable fields for create
$writableFields = loadFields('PRD', 'N');  // COD_ON_OFF contains 'N'

// Validate request body
foreach ($requestBody as $field => $value) {
    if (!in_array($field, $writableFields)) {
        throw new Exception("Field $field not writable on create (missing 'N' in COD_ON_OFF)");
    }
}

// Validate required fields
foreach ($writableFields as $field) {
    if ($field['CHECK_CAMPO'] == 'not null' && !isset($requestBody[$field['COD_COST']])) {
        throw new Exception("Required field {$field['COD_COST']} missing");
    }
}

Example:

POST /api/v4/core/PRD
{
  "source": "productManagement",
  "XPRD01": "Widget Pro",
  "XPRD02": 49.99,
  "XPRD03": "PRD-CUSTOM-001"  ❌ ERROR!
}

Response:
{
  "status": 400,
  "message": "Field XPRD03 not writable on create (missing 'N' in COD_ON_OFF)"
}

XPRD03 has COD_ON_OFF='LD' (no 'N') → cannot be set on create.

Update Validation (PUT)​

CoreWrite validates that ONLY fields with 'M' flag are provided:

// Load writable fields for update
$writableFields = loadFields('PRD', 'M');  // COD_ON_OFF contains 'M'

// Validate request body
foreach ($requestBody as $field => $value) {
    if (!in_array($field, $writableFields)) {
        throw new Exception("Field $field not writable on update (missing 'M' in COD_ON_OFF)");
    }
}

Example:

PUT /api/v4/core/PRD/1
{
  "source": "productManagement",
  "XPRD01": "Updated Widget",
  "XPRD28": "NewCategory"  ❌ ERROR!
}

Response:
{
  "status": 400,
  "message": "Field XPRD28 not writable on update (missing 'M' in COD_ON_OFF)"
}

XPRD28 has COD_ON_OFF='LDN' (no 'M') → cannot be modified after creation.

Partial Update (PATCH)​

PATCH has relaxed validation - skips COD_ON_OFF and required field checks:

// In RequestValidatorSubscriber
if ($routeName == "core.patch") {
    $data_form[0]['skipCodOnOffCheck'] = true;
    $data_form[0]['skipRequiredFieldsCheck'] = true;
}

This allows updating ANY field, regardless of COD_ON_OFF.

Use PATCH sparingly - it bypasses metadata constraints.

Common Mistakes​

❌ Mistake 1: Not Setting COD_ON_OFF​

-- Field defined without COD_ON_OFF
INSERT INTO TB_COST (DIM_COD, COD_COST, DESC_COST, TIPO_CAMPO)
VALUES ('PRD', 'XPRD01', 'Name', 'varchar(255)');
-- COD_ON_OFF is NULL or empty

Result: Field never visible (no flags match any center_dett).

Fix:

UPDATE TB_COST
SET COD_ON_OFF = 'L,D,N,M,R'
WHERE COD_COST = 'XPRD01';

❌ Mistake 2: Wrong Flag for Context​

-- Field set with only 'D' flag
UPDATE TB_COST SET COD_ON_OFF = 'D' WHERE COD_COST = 'XPRD01';
-- Try to create with this field
POST /api/v4/core/PRD
{
  "source": "productManagement",
  "XPRD01": "Widget"  ❌ ERROR - no 'N' flag
}

Result: Validation error - field not writable on create.

Fix: Add 'N' flag:

UPDATE TB_COST SET COD_ON_OFF = 'LDN' WHERE COD_COST = 'XPRD01';

❌ Mistake 3: Confusing center_dett Values​

-- Using wrong center_dett for list view
GET /api/v4/core/PRD?center_dett=dettaglio

-- Expected list, got detail fields

Result: More fields than expected (detail includes list fields).

Fix: Use correct center_dett:

GET /api/v4/core/PRD?center_dett=visualizza

❌ Mistake 4: Not Including 'L' for List View​

-- Field visible in detail but not list
UPDATE TB_COST SET COD_ON_OFF = 'DNM' WHERE COD_COST = 'XPRD20';

Result: Field missing from list views (no 'L' flag).

Fix: Add 'L' flag:

UPDATE TB_COST SET COD_ON_OFF = 'LDNM' WHERE COD_COST = 'XPRD20';

COD_ON_OFF Decision Tree​

Is the field auto-generated or read-only?
├─ YES → COD_ON_OFF = 'LD'
└─ NO  → Continue

Should users set this on create?
├─ YES → Add 'N' flag
└─ NO  → Skip 'N' flag

Should users modify this after create?
├─ YES → Add 'M' flag
└─ NO  → Skip 'M' flag

Should this appear in list views?
├─ YES → Add 'L' flag (almost always YES)
└─ NO  → Skip 'L' flag

Should this appear in detail views?
├─ YES → Add 'D' flag (almost always YES)
└─ NO  → Skip 'D' flag

Is this a search field?
├─ YES → Add 'R' flag
└─ NO  → Skip 'R' flag

Is this internal/system only?
├─ YES → COD_ON_OFF = 'X'
└─ NO  → Use combined flags

Summary​

COD_ON_OFF is a compact, powerful visibility control:

FlagMeaningWhen Field Appears
LListGrid/table views (visualizza, modifica, elimina)
DDetailSingle record views (dettaglio, edit, nuovo)
NNewCreate forms (nuovo, donuovo)
MModifyUpdate forms (edit, modifica, doedit)
RSearchSearch forms (ricerca, doricerca)
XHiddenIncluded in data but not displayed
IHiddenCompletely hidden from all contexts (ambiente)

Key Principles:

  1. ✅ One bitmask controls visibility across all contexts
  2. ✅ center_dett parameter determines which flags to match
  3. ✅ Validation enforces COD_ON_OFF rules (except PATCH)
  4. ✅ L flag almost always needed for list views
  5. ✅ D flag almost always needed for detail views

Common Patterns:

  • Read-only: LD
  • Fully editable: LDNMR
  • Create-only: LDN
  • Edit-only: LDM
  • Hidden: X
  • Hidden (ambiente): I (ambiente tracking fields)
