Source Parameter

Overview​

The source parameter is a required identifier that appears in every Core API request. It links menu-based permissions (TB_MENU) to dimension access (TB_DIM), enabling the platform to determine which data the authenticated user can access.

Critical Importance:

  • ✅ Required in ALL API calls - no exceptions
  • ✅ Links permission hierarchy to data access
  • ✅ Enables multi-tenant isolation
  • ✅ Validates dimension access rights
  • ✅ Determines field visibility context

Example Values:

  • "settings-2" - Settings module, menu item 2
  • "sc-4" - Scorecard module, menu item 4
  • "inventory-1" - Inventory module, menu item 1
  • "sales-orders-3" - Sales orders module, menu item 3

What is the Source Parameter?​

The source parameter is a business logic identifier that combines:

  1. Microservice/Module Prefix - identifies the application area
  2. Numerical Suffix - identifies specific menu item

Format:

{microservice_prefix}-{menu_number}

Examples:
- settings-2
- sc-4
- inventory-1

Storage and Generation:

  • Stored in TB_MENU.COD_MENU field
  • Generated when creating menu items in the platform
  • Links to dimension permissions via business logic
  • Unique within the permission hierarchy

How Source Works​

Permission to Dimension Binding​

The source parameter creates a critical link in the permission chain:

User Authentication
    ↓
JWT Token (includes grants)
    ↓
API Request (includes source parameter)
    ↓
TB_MENU lookup (COD_MENU = source)
    ↓
Business Logic (resolve dimension from menu context)
    ↓
TB_DIM validation (user can access this dimension?)
    ↓
Grant Check (user has permission for this operation?)
    ↓
Data Access

Menu Structure Example​

-- TB_MENU example
COD_MENU        DESCRIZIONE              NLEFT  NRIGHT  COD_DIM_MENU
-----------------------------------------------------------------
settings-1      Settings Root            1      20      NULL
settings-2      Product Catalog          2      9       ART
settings-3      Price Lists              10     13      LST
settings-4      Departments              14     19      DEPFOOD

-- When user calls: GET /api/v4/core/ART?source=settings-2
-- Platform resolves: COD_MENU='settings-2' → COD_DIM_MENU='ART'
-- Validates: User has grant for settings-2 in nested set?
-- Result: Access granted or denied based on TB_MENU hierarchy

Required in All API Calls​

The source parameter must be included in every Core API request, either as:

Query Parameter (GET Requests)​

# List products
GET /api/v4/core/ART?source=settings-2

# Get single product
GET /api/v4/core/ART/123?source=settings-2

# Filter products
GET /api/v4/core/ART?source=settings-2&filter[XART06]=1

# Count products
GET /api/v4/core/ART/count?source=settings-2

Request Body (POST/PUT/PATCH Requests)​

// Create product
POST /api/v4/core/ART
{
  "source": "settings-2",
  "data": {
    "XART01": "2",
    "XART20": "Widget Pro"
  }
}

// Update product
PUT /api/v4/core/ART/123
{
  "source": "settings-2",
  "data": {
    "XART20": "Widget Pro Updated"
  }
}

// Partial update
PATCH /api/v4/core/ART/123
{
  "source": "settings-2",
  "data": {
    "XART06": 1
  }
}

DELETE Requests​

# Soft delete
DELETE /api/v4/core/ART/123?source=settings-2

# Force delete
DELETE /api/v4/core/ART/123?source=settings-2&force=true

Validation and Errors​

Missing Source Parameter​

Request:

GET /api/v4/core/ART
# Missing source parameter

Error Response:

{
  "status": 400,
  "message": "Missing required parameter: source",
  "code": "MISSING_SOURCE"
}

Invalid Source Value​

Request:

GET /api/v4/core/ART?source=invalid-menu-999

Error Response:

{
  "status": 400,
  "message": "Invalid DIM from provided source",
  "code": "INVALID_SOURCE",
  "details": "Source 'invalid-menu-999' does not exist in TB_MENU or does not map to dimension ART"
}

Source-Dimension Mismatch​

Request:

# source=settings-2 maps to ART dimension
GET /api/v4/core/ORD?source=settings-2
# Requesting ORD dimension with ART source

Error Response:

{
  "status": 400,
  "message": "Invalid DIM from provided source",
  "code": "SOURCE_DIM_MISMATCH",
  "details": "Source 'settings-2' is configured for dimension ART, not ORD"
}

How Source is Generated​

The source parameter is created when menu items are configured in the platform:

Step 1: Create Menu Item in TB_MENU​

INSERT INTO TB_MENU (
    COD_MENU,              -- Auto-generated: {prefix}-{number}
    DESCRIZIONE,           -- "Product Catalog"
    COD_DIM_MENU,          -- "ART" (dimension this menu accesses)
    NLEFT,                 -- Nested set left boundary
    NRIGHT,                -- Nested set right boundary
    PARENT_MENU            -- Parent menu code
) VALUES (
    'settings-2',          -- Generated source value
    'Product Catalog',
    'ART',
    2,
    9,
    'settings-1'
);

Step 2: Assign Grants to Users​

-- Grant user access to settings-2 menu
INSERT INTO TB_GRANTS (
    USER_ID,
    COD_MENU,
    GRANT_TYPE
) VALUES (
    'user123',
    'settings-2',
    'READ_WRITE'
);

Step 3: Use in API Calls​

# User can now access ART dimension via source=settings-2
GET /api/v4/core/ART?source=settings-2

Why Source Matters​

1. Security and Multi-Tenant Isolation​

Problem without source:

# Without source, all users could access any dimension
GET /api/v4/core/ART
# How does platform know which tenant's data?
# How does platform check permissions?

Solution with source:

GET /api/v4/core/ART?source=settings-2
# Platform resolves: settings-2 → ART dimension
# Platform checks: User has grant for settings-2?
# Platform filters: Only tenant data for this user
# Result: Secure, multi-tenant data access

2. Permission Context​

The source parameter provides the permission context for the request:

source = "settings-2"
    ↓
TB_MENU lookup
    ↓
Nested Set position (NLEFT=2, NRIGHT=9)
    ↓
User Grant Check
    ↓
Parent/Child permission inheritance
    ↓
Operation allowed/denied

3. Field Visibility Control​

Source parameter contributes to the context chain that determines field visibility:

// Context chain
{
  source: "settings-2",        // Permission context
  peso: 3,                      // User level
  ambiente: "P",                // Environment
  centro_dett: "visualizza",    // View mode
  modulo: "inventory",          // Module
  lingua: "IT"                  // Language
}

// Together determine which fields are visible via COD_ON_OFF

Common Patterns​

Pattern 1: Module-Specific Sources​

Organize sources by application module:

-- Inventory module
inventory-1    → Stock Management
inventory-2    → Warehouse Locations
inventory-3    → Stock Movements

-- Sales module
sales-1        → Orders
sales-2        → Invoices
sales-3        → Quotes

-- Customer module
customers-1    → Customer List
customers-2    → Customer Addresses
customers-3    → Customer Contacts

Pattern 2: Multi-Tenant Sources​

Different tenants use different source prefixes:

-- Tenant A (retail company)
retail-1       → Products (COD_DIM=ART, Tenant=A)
retail-2       → Orders (COD_DIM=ORD, Tenant=A)

-- Tenant B (wholesale company)
wholesale-1    → Products (COD_DIM=ART, Tenant=B)
wholesale-2    → Orders (COD_DIM=ORD, Tenant=B)

-- Same dimension (ART), different tenants, different sources

Pattern 3: Environment-Specific Sources​

Development vs production use different sources:

-- Development
dev-settings-2  → ART dimension (development data)

-- Production
prod-settings-2 → ART dimension (production data)

-- Controlled via ambiente in context chain

Best Practices​

✅ DO:​

Always include source:

# ✅ Good - source provided
GET /api/v4/core/ART?source=settings-2

Use correct source for dimension:

# ✅ Good - settings-2 maps to ART
GET /api/v4/core/ART?source=settings-2

Store source in client configuration:

// ✅ Good - centralized source management
const API_CONFIG = {
  products: 'settings-2',
  orders: 'sales-1',
  customers: 'customers-1'
};

const products = await fetch(
  `/api/v4/core/ART?source=${API_CONFIG.products}`
);

❌ DON'T:​

Don't omit source:

# ❌ Bad - missing source
GET /api/v4/core/ART

Don't hardcode source values:

// ❌ Bad - hardcoded source
const products = await fetch('/api/v4/core/ART?source=settings-2');

// ✅ Good - configurable source
const products = await fetch(`/api/v4/core/ART?source=${config.productSource}`);

Don't use wrong source for dimension:

# ❌ Bad - sales-1 maps to ORD, not ART
GET /api/v4/core/ART?source=sales-1

Integration with Other Concepts​

Source + Context Chain​

// Complete context for API request
{
  source: "settings-2",        // Permission context (TB_MENU)
  peso: 3,                      // User level (1-5)
  ambiente: "P",                // Environment (P/T/S)
  centro_dett: "visualizza",    // View mode
  modulo: "inventory",          // Module
  lingua: "IT"                  // Language
}

Source + Nested Set Permissions​

-- User has grant for parent menu
COD_MENU = 'settings-1' (NLEFT=1, NRIGHT=20)

-- Child menus inherit permission
settings-2 (NLEFT=2, NRIGHT=9) → GRANTED (2 between 1 and 20)
settings-3 (NLEFT=10, NRIGHT=13) → GRANTED (10 between 1 and 20)

-- Source parameter enables nested set check

Source + COD_ON_OFF​

-- Source determines permission context
-- centro_dett determines view mode
-- Together they select appropriate COD_ON_OFF flags

source=settings-2 + centro_dett=visualizza → fields with L flag
source=settings-2 + centro_dett=dettaglio  → fields with D flag

Troubleshooting​

Problem: "Invalid DIM from provided source"​

Cause: Source parameter doesn't map to the requested dimension.

Solution:

  1. Check TB_MENU for correct COD_MENU value
  2. Verify COD_DIM_MENU matches requested dimension
  3. Use correct source for the dimension you're accessing

Problem: "Missing required parameter: source"​

Cause: Source parameter not included in request.

Solution:

  1. Add source to query string (GET)
  2. Add source to request body (POST/PUT/PATCH/DELETE)

Problem: 403 Forbidden with valid source​

Cause: User doesn't have grant for the specified source.

Solution:

  1. Check user's grants in TB_GRANTS
  2. Verify nested set hierarchy in TB_MENU
  3. Ensure user has permission for source menu item

Summary​

  • ✅ Source parameter is required in ALL API calls
  • ✅ Links TB_MENU (permissions) → TB_DIM (data access)
  • ✅ Format: {microservice_prefix}-{menu_number}
  • ✅ Enables multi-tenant isolation and security
  • ✅ Validates dimension access rights
  • ✅ Contributes to context chain for field visibility
  • ✅ Generated when creating menu items in TB_MENU
  • ✅ Missing or invalid source → 400 Bad Request error

Key Takeaways:

  1. Never omit the source parameter - it's mandatory
  2. Use the correct source for the dimension you're accessing
  3. Store source values in configuration, not hardcoded
  4. Source is the bridge between permissions and data
  5. Validates via TB_MENU → COD_DIM_MENU mapping
