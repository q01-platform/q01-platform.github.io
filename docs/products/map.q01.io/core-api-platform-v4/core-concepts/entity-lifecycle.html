Entity Lifecycle: TREC State Management

What is TREC?​

TREC (Transaction Record State) is a single-character field present in every dimension table (TB_ANAG_{DIM}00) that tracks the lifecycle state of each record.

Combined with audit trail fields (OWNER, CDATA, LOWNER, LDATA, TIMESTAMP), TREC enables:

  • Soft delete - Non-destructive data archiving
  • State tracking - Understanding record history
  • Audit trails - Who created/modified what and when
  • Data recovery - Undeleting cancelled records

Key Insight:

TREC is not just a status field - it's the foundation for non-destructive data management throughout Q01 Core APIs.

The Four TREC States​

StateCodeMeaningSet ByUse Case
NewNNewly created recordCoreWrite on INSERTFresh records not yet modified
ModifiedMRecord has been updatedCoreWrite on UPDATERecords with changes since creation
CancelledCSoft deleted recordCoreWrite on DELETEArchived/deleted data (recoverable)
ProvvisorioPProvisional/temporary recordManual/system logicDraft records, temporary data, work in progress

Visual State Flow​

    CREATE              UPDATE              DELETE
      ↓                   ↓                   ↓
   ┌─────┐           ┌─────┐             ┌─────┐
   │  N  │  ------→  │  M  │  --------→  │  C  │
   └─────┘           └─────┘             └─────┘
     New             Modified            Cancelled
      │                 │                     ↑
      │                 │                     │
      └─────────────────┴─────────────────────┘
              (can also transition directly)

   ┌─────┐
   │  P  │  ← Set manually - provisional/draft state
   └─────┘
  Provvisorio

Audit Trail Fields​

Every dimension table includes these standard fields:

FieldTypePurposeAuto-Populated
OWNERvarchar(50)User ID who created record✅ On INSERT
CDATAvarchar(14)Creation date (YYYYMMDDHHmmss)✅ On INSERT
LOWNERvarchar(50)User ID who last modified record✅ On UPDATE
LDATAvarchar(14)Last modification date✅ On UPDATE
TIMESTAMPvarchar(20)Microsecond timestamp✅ On INSERT/UPDATE
TRECchar(1)Record state (N/M/C/P)✅ On INSERT/UPDATE/DELETE

Example Record:

{
  "PRD_ID": "123",
  "XPRD01": "Premium Widget",
  "XPRD02": 49.99,
  "OWNER": "user_5",
  "CDATA": "20250115103045",
  "LOWNER": "user_12",
  "LDATA": "20250118141522",
  "TIMESTAMP": "1737207322485000",
  "TREC": "M"
}

Interpretation:

  • Created by user_5 on January 15, 2025 at 10:30:45
  • Last modified by user_12 on January 18, 2025 at 14:15:22
  • Current state: Modified (M)

Automatic TREC Management​

CoreWrite automatically manages TREC transitions during CRUD operations:

1. CREATE Operation (POST)​

Default Behavior:

INSERT INTO TB_ANAG_PRD00 (
  XPRD01, XPRD02,
  OWNER, CDATA, LOWNER, LDATA, TIMESTAMP, TREC
) VALUES (
  'Widget', 49.99,
  'user_5', '20250115103045', 'user_5', '20250115103045', '1737207322485000', 'N'
);

TREC State: N (New)

Audit Fields:

  • OWNER = Current user from JWT
  • CDATA = Current timestamp (YYYYMMDDHHmmss)
  • LOWNER = Same as OWNER (first modification)
  • LDATA = Same as CDATA
  • TIMESTAMP = Microsecond timestamp

Code Reference: DataStore.php:876

$sql .= " TREC = '" . ($param['TREC'] ? $param['TREC'] : 'N') . "', ";

2. UPDATE Operation (PUT/PATCH)​

Default Behavior:

UPDATE TB_ANAG_PRD00 SET
  XPRD02 = 59.99,
  LOWNER = 'user_12',
  LDATA = '20250118141522',
  TIMESTAMP = '1737207322485000',
  TREC = 'M'
WHERE PRD_ID = '123';

TREC State: M (Modified)

Audit Fields:

  • OWNER and CDATA remain unchanged
  • LOWNER = Current user from JWT
  • LDATA = Current timestamp
  • TIMESTAMP = Updated microsecond timestamp

Code Reference: DataStore.php:822

$dupKey .= "TREC = 'M', ";

3. DELETE Operation (Soft Delete)​

Default Behavior:

UPDATE TB_ANAG_PRD00 SET
  TREC = 'C',
  LOWNER = 'user_12',
  LDATA = '20250119095030'
WHERE PRD_ID = '123';

TREC State: C (Cancelled)

What Happens:

  • Record remains in database
  • Marked as cancelled (TREC='C')
  • Standard queries exclude TREC='C' by default
  • LOWNER and LDATA track who deleted and when

Code Reference: DataStore.php:2420

$sqlDel = "update TB_ANAG_{$dim}00 set TREC = 'C', LDATA = :LDATA where {$dim}_ID = :value";

4. DELETE Operation (Physical Delete)​

Force Delete:

DELETE /api/v4/core/PRD/123?forceDelete=true

Behavior:

DELETE FROM TB_ANAG_PRD00 WHERE PRD_ID = '123';

TREC State: N/A (record physically removed)

When to Use:

  • Removing test data
  • Compliance requirements (GDPR right to be forgotten)
  • Data corruption cleanup

Warning: Physical deletes are irreversible and should be rare.

Code Reference: DataStore.php:2414

if ($flag && $this->_skipTREC == false) {
    // Soft delete (TREC='C')
} else {
    // Physical delete
}

Provvisorio State (TREC='P')​

What is TREC='P'?​

Records with TREC='P' represent provisional or draft data:

  • Temporary records not yet finalized
  • Draft documents or work in progress
  • Data pending approval or validation
  • Can be filtered out from production queries

Use Cases​

Example 1: Draft Order

-- Order being prepared, not yet submitted
INSERT INTO TB_ANAG_ORD00 (XORD01, XORD02, TREC, ...)
VALUES ('Draft Order', 'Customer Name', 'P', ...);

Example 2: Temporary Product Entry

-- Product being configured, not ready for sale
UPDATE TB_ANAG_PRD00 SET TREC = 'P'
WHERE PRD_ID = 123 AND XPRD13 = 'DRAFT';

Code Reference: CleanExpiredSessions.php:85

$sql = "select SESSIONS_ID, TREC, XSESSIONS05, XSESSIONS12
        from TB_ANAG_SESSIONS00
        where TREC != 'P'
        order by XSESSIONS05 desc";

Session cleanup excludes TREC='P' records (provisional sessions).

Querying by TREC State​

Default Behavior: Exclude Cancelled​

CoreQuery automatically excludes TREC='C' unless explicitly requested:

GET /api/v4/core/PRD?source=productList&center_dett=visualizza

Generated SQL:

SELECT * FROM TB_ANAG_PRD00
WHERE TREC != 'C'  -- Automatically added
ORDER BY PRD_ID;

Including Cancelled Records​

Explicit TREC Filter:

GET /api/v4/core/PRD?source=productList&center_dett=visualizza&cond_trec=N,M,C

Generated SQL:

SELECT * FROM TB_ANAG_PRD00
WHERE TREC IN ('N', 'M', 'C')  -- All states including cancelled
ORDER BY PRD_ID;

Only Cancelled Records​

Archive Query:

GET /api/v4/core/PRD?source=productList&center_dett=visualizza&cond_trec=C

Use Case: Recovering deleted data, audit reports.

Common Patterns​

Pattern 1: Soft Delete with Recovery​

Scenario: User deletes product by mistake, then wants to recover it.

Step 1: Soft Delete

DELETE /api/v4/core/PRD/123

Result: TREC='C', record still exists.

Step 2: Recover

PATCH /api/v4/core/PRD/123
{
  "TREC": "M"
}

Result: TREC='M', record visible again.

Pattern 2: Audit Trail Report​

Query:

GET /api/v4/core/PRD?source=productList&center_dett=visualizza&cond_trec=N,M,C&$select=PRD_ID,XPRD01,TREC,OWNER,CDATA,LOWNER,LDATA

Response:

[
  {
    "PRD_ID": "123",
    "XPRD01": "Widget A",
    "TREC": "M",
    "OWNER": "user_5",
    "CDATA": "20250115103045",
    "LOWNER": "user_12",
    "LDATA": "20250118141522"
  },
  {
    "PRD_ID": "124",
    "XPRD01": "Widget B",
    "TREC": "C",
    "OWNER": "user_8",
    "CDATA": "20250110080000",
    "LOWNER": "user_8",
    "LDATA": "20250119095030"
  }
]

Interpretation:

  • Product 123: Modified by user_12 on Jan 18
  • Product 124: Deleted by user_8 on Jan 19

Pattern 3: Working with Draft Records​

Mark as Provisional/Draft:

PATCH /api/v4/core/PRD/999
{
  "TREC": "P"
}

Result: Product 999 marked as provisional/draft, can be filtered from production queries.

TREC in Cascade Operations​

When cascading deletes, TREC propagates:

Example: Order with Items​

Dimension Setup:

TB_DIM: ORD (orders)
TB_DIM: ORDITEM (order items)

TB_COST:
- XORDITEM01 (foreign key to ORD)
- DELETE_CASCADE configured on XORDITEM01

Delete Order:

DELETE /api/v4/core/ORD/555

Cascade Behavior:

-- Parent record
UPDATE TB_ANAG_ORD00 SET TREC = 'C', LDATA = '20250119095030' WHERE ORD_ID = '555';

-- Cascaded child records
UPDATE TB_ANAG_ORDITEM00 SET TREC = 'C', LDATA = '20250119095030' WHERE XORDITEM01 = '555';

Result: Order and all items marked TREC='C', recoverable as unit.

Code Reference: DataStore.php:2372

$q = "update TB_ANAG_{$tbl}00 set TREC = 'C', LDATA = '" . date("YmdHis") . "' where {$joinField} = '{$cascadeVal}'";

Best Practices​

✅ DO: Trust Automatic TREC Management​

CoreWrite sets TREC correctly during operations:

  • INSERT → TREC='N'
  • UPDATE → TREC='M'
  • DELETE → TREC='C'

No manual TREC manipulation needed.

✅ DO: Use Soft Delete by Default​

Reserve physical delete (forceDelete=true) for exceptional cases:

  • Test data cleanup
  • Legal compliance (GDPR)
  • Data corruption

Default soft delete enables recovery.

✅ DO: Query Audit Trail for Compliance​

Use OWNER, CDATA, LOWNER, LDATA for:

  • Regulatory audits
  • Change tracking
  • User activity reports

Example Query:

GET /api/v4/core/PRD?$filter=LDATA gt 20250101000000&$select=PRD_ID,XPRD01,LOWNER,LDATA

All products modified after January 1, 2025.

✅ DO: Use TREC='P' for Draft/Provisional Data​

Mark temporary or work-in-progress records:

  • Draft orders before submission
  • Product configurations being set up
  • Documents pending approval

❌ DON'T: Manually Set TREC in Requests​

Wrong:

POST /api/v4/core/PRD
{
  "XPRD01": "Widget",
  "TREC": "N"
}

Why: CoreWrite sets TREC automatically. Manual values can cause inconsistencies.

Right:

POST /api/v4/core/PRD
{
  "XPRD01": "Widget"
}

CoreWrite adds TREC='N'.

❌ DON'T: Use Physical Delete for Routine Operations​

Wrong:

DELETE /api/v4/core/PRD/123?forceDelete=true

Why: Irreversible, no audit trail, breaks cascades.

Right:

DELETE /api/v4/core/PRD/123

Soft delete (TREC='C') preserves data.

❌ DON'T: Assume TREC='N' Means Unmodified​

Misconception: TREC='N' means record never changed.

Reality: TREC='N' means:

  • Created but not yet updated via PUT/PATCH
  • May have been modified via duplicate key logic (INSERT with ON DUPLICATE KEY UPDATE)

Check LDATA vs CDATA for true modification detection:

if (record.LDATA !== record.CDATA) {
  console.log("Record was modified");
}

Common Pitfalls​

Pitfall 1: Filtering TREC in Application Code​

Wrong:

// Client-side filtering
const activeProducts = allProducts.filter(p => p.TREC !== 'C');

Why: Wasteful - CoreQuery already excludes TREC='C' by default.

Right:

GET /api/v4/core/PRD?source=productList

CoreQuery returns only TREC !='C' automatically.

Pitfall 2: Not Handling Cascade TREC in UI​

Scenario: Deleting order marks items as TREC='C', but UI still shows items.

Solution: Refresh related data after delete:

await api.delete(`/core/ORD/${orderId}`);
// Refresh order items list - they're now TREC='C'
await fetchOrderItems(orderId, { cond_trec: 'N,M' });

Pitfall 3: Ignoring TIMESTAMP for Concurrency​

Scenario: Two users update same record simultaneously.

Problem: Last write wins, no conflict detection.

Solution: Use TIMESTAMP for optimistic locking:

PUT /api/v4/core/PRD/123
{
  "XPRD02": 59.99,
  "TIMESTAMP": "1737207322485000"  // From GET response
}

CoreWrite can validate TIMESTAMP hasn't changed (if configured).

Summary​

TREC State Management enables:

  1. ✅ Non-destructive deletes - Soft delete (TREC='C') preserves data
  2. ✅ State tracking - N/M/C/P lifecycle visibility
  3. ✅ Audit trails - OWNER, CDATA, LOWNER, LDATA track changes
  4. ✅ Data recovery - Undelete by changing TREC='C' to TREC='M'
  5. ✅ Protection - TREC='P' prevents accidental deletion

Key Insights:

TREC is not optional metadata - it's the foundation for safe data operations in Q01 Core APIs. Trust automatic TREC management, use soft delete by default, and leverage audit fields for compliance.
