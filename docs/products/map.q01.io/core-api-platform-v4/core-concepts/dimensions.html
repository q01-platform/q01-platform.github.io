Dimensions: The Foundation

What is a Dimension?​

A dimension is a business entity type registered in the Q01 metadata system. Think of it as a "table definition" or "entity class" - but defined in metadata, not code.

Examples of dimensions:

  • PRD - Products
  • CLI - Customers
  • ORD - Orders
  • INV - Invoices
  • USR - Users

Once a dimension is registered in TB_DIM, Core APIs automatically provide full CRUD operations without any code deployment.

Dimension vs. Traditional Table​

Traditional Approach (Hardcoded)​

// Need to write code for each entity ❌
class ProductService {
  async create(data) { /* custom code */ }
  async read(id) { /* custom code */ }
  async update(id, data) { /* custom code */ }
  async delete(id) { /* custom code */ }
}

class CustomerService {
  async create(data) { /* custom code */ }
  async read(id) { /* custom code */ }
  // ... more repetitive code
}

// Adding new entity = write more code, deploy, test

Q01 Approach (Metadata-Driven)​

-- Define dimension in metadata ✅
INSERT INTO TB_DIM (COD_DIM, TABLE_NAME_PUB, DESC_DIM, ...)
VALUES ('PRD', 'TB_ANAG_PRD00', 'Products', ...);

-- Define fields in TB_COST
INSERT INTO TB_COST (DIM_COD, COD_COST, DESC_COST, TIPO_CAMPO, ...)
VALUES ('PRD', 'XPRD01', 'Product Name', 'varchar(255)', ...);

-- Done! Core APIs now handle all CRUD:
-- GET /api/v4/core/PRD → lists products
-- POST /api/v4/core/PRD → creates product
-- PUT /api/v4/core/PRD/123 → updates product
-- DELETE /api/v4/core/PRD/123 → deletes product

No code deployment needed - just metadata registration.

TB_DIM: Dimension Registry​

All dimensions are registered in the TB_DIM table.

Key Fields​

FieldPurposeExample
COD_DIMDimension code (unique identifier)PRD, CLI, ORD
TABLE_NAME_PUBPhysical table nameTB_ANAG_PRD00
DESC_DIMHuman-readable descriptionProducts, Customers
TIPO_DIMDimension type0 (table), 1 (view), 2 (document)
TYPE_OBJObject type0 (standard), 2 (custom object)
COD_MENUAssociated menu/security areaproductManagement
MODULOModule contextDefault, POS, B2B
COD_COST_PUBUse public field codes1 (yes), 0 (no)

Example Registration​

INSERT INTO TB_DIM (
  COD_DIM,
  TABLE_NAME_PUB,
  DESC_DIM,
  TIPO_DIM,
  TYPE_OBJ,
  COD_MENU,
  MODULO
) VALUES (
  'PRD',                    -- Dimension code
  'TB_ANAG_PRD00',          -- Physical table
  'Products',               -- Description
  0,                        -- Standard table
  0,                        -- Standard object
  'productManagement',      -- Security area
  'Default'                 -- Module
);

Physical Storage: TB_ANAG_{DIM}00​

Data for each dimension is stored in a physical table following this naming convention:

TB_ANAG_{DIM}00

Examples:

  • TB_ANAG_PRD00 - Products data
  • TB_ANAG_CLI00 - Customers data
  • TB_ANAG_ORD00 - Orders data

Standard Fields (Auto-Added)​

Every dimension table automatically includes these audit/system fields:

FieldPurposeAuto-Populated
{DIM}_IDPrimary keyYes (auto-increment or sequence)
OWNERCreating user IDYes (from JWT)
LOWNERLast modifying user IDYes (from JWT)
CDATACreation timestamp (YmdHis)Yes (current datetime)
LDATALast modification timestampYes (current datetime)
TRECRecord state (N/M/C/P)Yes (N on create, M on update)
TIMESTAMPMicrosecond timestampYes (microtime)
X{DIM}01Ambiente (environment filter)Yes (from session)
X{DIM}02Ambiente_rif (reference environment)Yes (from session)

Example for PRD dimension:

CREATE TABLE TB_ANAG_PRD00 (
  PRD_ID INT PRIMARY KEY AUTO_INCREMENT,

  -- Business fields (defined in TB_COST)
  XPRD01 VARCHAR(255),  -- Product name
  XPRD02 DECIMAL(10,2), -- Price
  XPRD03 VARCHAR(50),   -- Product code

  -- Standard audit fields (auto-added)
  OWNER VARCHAR(50),
  LOWNER VARCHAR(50),
  CDATA VARCHAR(14),
  LDATA VARCHAR(14),
  TREC CHAR(1) DEFAULT 'N',
  TIMESTAMP BIGINT,
  XPRD01_AMBIENTE VARCHAR(255),  -- Note: 01 = ambiente
  XPRD02_AMBIENTE VARCHAR(255)   -- Note: 02 = ambiente_rif
);

Core APIs handle all standard fields automatically - you never need to populate them manually.

Dimension Types​

Type 0: Standard Table​

Most common type - represents a normalized database table.

INSERT INTO TB_DIM (COD_DIM, TIPO_DIM, TYPE_OBJ, ...)
VALUES ('PRD', 0, 0, ...);

Behavior:

  • CoreQuery generates SELECT statements
  • CoreWrite generates INSERT/UPDATE/DELETE statements
  • Fields defined in TB_COST
  • Standard CRUD operations

Type 1: View​

Represents a database view (read-only, computed data).

INSERT INTO TB_DIM (COD_DIM, TIPO_DIM, TYPE_OBJ, TABLE_NAME_PUB, ...)
VALUES ('PRD_SUMMARY', 1, 0, 'V_PRODUCTS_SUMMARY', ...);

Behavior:

  • CoreQuery can read from view
  • CoreWrite operations not allowed (views are read-only)
  • Useful for complex joins, aggregations

Type 2: Document/Hierarchy​

Represents hierarchical or nested data structures.

INSERT INTO TB_DIM (COD_DIM, TIPO_DIM, TYPE_OBJ, ...)
VALUES ('ORDER_DOCUMENT', 2, 0, ...);

Behavior:

  • Special /api/v4/document/{dim}/{id} endpoint
  • Fetches related data across multiple dimensions
  • Returns nested JSON with full hierarchy

Custom Objects (TYPE_OBJ=2)​

Special dimension type for custom business logic.

INSERT INTO TB_DIM (COD_DIM, TIPO_DIM, TYPE_OBJ, ...)
VALUES ('CUSTOM_REPORT', 0, 2, ...);

Behavior:

  • Routed to CustomModules service instead of standard Modules
  • Allows custom SQL or logic
  • Still uses metadata for field definitions

Dimension Lifecycle​

1. Register Dimension​

-- Step 1: Register in TB_DIM
INSERT INTO TB_DIM (COD_DIM, TABLE_NAME_PUB, DESC_DIM, ...)
VALUES ('PRD', 'TB_ANAG_PRD00', 'Products', ...);

2. Define Fields​

-- Step 2: Define fields in TB_COST
INSERT INTO TB_COST (DIM_COD, COD_COST, DESC_COST, TIPO_CAMPO, COD_ON_OFF, COD_SEQUENZA, ...)
VALUES
  ('PRD', 'XPRD01', 'Product Name', 'varchar(255)', 'LDNMR', 10, ...),
  ('PRD', 'XPRD02', 'Price', 'decimal(10,2)', 'LDNMR', 20, ...),
  ('PRD', 'XPRD03', 'Product Code', 'varchar(50)', 'LD', 30, ...);

3. Create Physical Table​

-- Step 3: Create actual database table
CREATE TABLE TB_ANAG_PRD00 (
  PRD_ID INT PRIMARY KEY AUTO_INCREMENT,
  XPRD01 VARCHAR(255),
  XPRD02 DECIMAL(10,2),
  XPRD03 VARCHAR(50),
  -- Standard fields
  OWNER VARCHAR(50),
  LOWNER VARCHAR(50),
  CDATA VARCHAR(14),
  LDATA VARCHAR(14),
  TREC CHAR(1) DEFAULT 'N',
  TIMESTAMP BIGINT,
  XPRD01_AMBIENTE VARCHAR(255),
  XPRD02_AMBIENTE VARCHAR(255)
);

4. Configure Permissions​

-- Step 4: Define security areas in TB_MENU
INSERT INTO TB_MENU (COD_MENU, DESC_MENU, DIM_COD, NLEVEL, NLEFT, NRIGHT, ...)
VALUES ('productManagement', 'Product Management', 'PRD', 2, 10, 20, ...);

5. Start Using​

# Step 5: Core APIs now work!
curl -X GET 'https://coreservice/api/v4/core/PRD?source=productManagement'

Total time: ~30 minutes to register a new dimension with full CRUD.

Querying Dimensions​

List All Dimensions​

SELECT COD_DIM, DESC_DIM, TABLE_NAME_PUB, TIPO_DIM
FROM TB_DIM
WHERE TIPO_DIM = 0  -- Standard tables only
ORDER BY DESC_DIM;

Find Dimension for Table​

SELECT COD_DIM, DESC_DIM
FROM TB_DIM
WHERE TABLE_NAME_PUB = 'TB_ANAG_PRD00';

Get Dimension Metadata​

# Via Core APIs
GET /api/v4/core/PRD/fields?source=productManagement&center_dett=nuovo

Returns TB_COST metadata for all fields visible in "nuovo" (create) context.

Dimension Naming Conventions​

COD_DIM (Dimension Code)​

Recommended format: 3-4 uppercase letters

Examples:

  • ✅ PRD - Products
  • ✅ CLI - Customers
  • ✅ ORD - Orders
  • ✅ ORDS - Order Details (4 letters)
  • ❌ products - Avoid lowercase
  • ❌ P - Too short, not descriptive

TABLE_NAME_PUB (Table Name)​

Mandatory format: TB_ANAG_{DIM}00

Examples:

  • ✅ TB_ANAG_PRD00 - Products table
  • ✅ TB_ANAG_CLI00 - Customers table
  • ❌ products - Wrong format
  • ❌ TB_PRD - Missing ANAG prefix and 00 suffix

Why this format?

  • TB_ - Standard table prefix
  • ANAG_ - Indicates "anagrafica" (master data)
  • {DIM} - Dimension code
  • 00 - Version suffix (allows for table versioning)

Field Naming (in TB_COST)​

Recommended format: X{DIM}{SEQ}

Examples:

  • ✅ XPRD01 - Product field #1 (name)
  • ✅ XPRD02 - Product field #2 (price)
  • ✅ XCLI01 - Customer field #1 (name)
  • ❌ product_name - Not Q01 convention
  • ❌ name - Missing dimension prefix

Why this format?

  • X - Indicates extended/custom field
  • {DIM} - Dimension code (PRD, CLI)
  • {SEQ} - Sequential number (01, 02, 03, ...)

Allows easily identifying which dimension a field belongs to.

Common Patterns​

Master-Detail Relationships​

Master dimension (header):

INSERT INTO TB_DIM (COD_DIM, ...) VALUES ('ORD', ...);  -- Orders

Detail dimension (line items):

INSERT INTO TB_DIM (COD_DIM, ...) VALUES ('ORDITEM', ...);  -- Order Items

-- Detail references master via foreign key
-- Define in TB_COST:
INSERT INTO TB_COST (DIM_COD, COD_COST, TIPO_CAMPO, ...)
VALUES ('ORDITEM', 'XORDITEM01', 'int', ...);

Cascades in metadata:

{
  "DELETE_CASCADE": {
    "srcKey": "ORD_ID",
    "applyTo": {
      "ORDITEM": {"destKey": "ORDER_ID"}
    }
  }
}

Deleting order → automatically deletes order items.

Lookup Tables​

Main dimension:

INSERT INTO TB_DIM (COD_DIM, ...) VALUES ('PRD', ...);

Lookup dimension:

INSERT INTO TB_DIM (COD_DIM, ...) VALUES ('CATEGORY', ...);

-- Reference in TB_COST with appropriate field type:
INSERT INTO TB_COST (DIM_COD, COD_COST, TIPO_CAMPO, ...)
VALUES ('PRD', 'XPRD05', 'combo', ...);

Result: Product's category field becomes dropdown that can be configured to reference CATEGORY dimension data.

Dimension Best Practices​

✅ DO:​

  • Use 3-4 letter uppercase codes for COD_DIM
  • Follow TB_ANAG_{DIM}00 naming for tables
  • Define all fields in TB_COST (never add table columns without metadata)
  • Use COD_SEQUENZA for field ordering (10, 20, 30, ... allows insertions)
  • Configure COD_ON_OFF for all fields
  • Set appropriate TIPO_DIM and TYPE_OBJ

❌ DON'T:​

  • Use lowercase or special characters in COD_DIM
  • Create tables outside TB_ANAG_ convention
  • Add table columns without corresponding TB_COST entries
  • Skip standard audit fields (OWNER, CDATA, TREC, etc.)
  • Use sequential numbering without gaps (use 10, 20, 30, not 1, 2, 3)

Summary​

Dimensions are the foundation of Q01 Core APIs:

  1. Registered in TB_DIM - defines dimension properties
  2. Stored in TB_ANAG_{DIM}00 - physical table with standard audit fields
  3. Fields defined in TB_COST - metadata-driven field definitions
  4. Automatically get CRUD operations - no code needed

Key Mental Model:

"A dimension is a registered entity type that Core APIs know how to handle automatically based on metadata."

Once you understand dimensions, you can move on to:

  • MAP Framework - the metadata tables that define dimensions
  • Field Visibility - how COD_ON_OFF controls what you see
  • Context Chain - how context affects dimension operations
