Counter Management

What is Counter Management?​

Counter Management is Q01's system for generating sequential, formatted, business-friendly identifiers (e.g., "INV-2025-00123", "ORD-ABC-000456") using metadata-driven configuration in TB_COUNTER.

Unlike simple auto-increment IDs, counters provide:

  • Custom formatting - Templates with year, padding, prefixes, separators
  • Partitioning - Separate sequences per year, store, category, etc.
  • Concurrency safety - FOR UPDATE locking prevents duplicates
  • Fallback mechanisms - TB_ANAG_COUNT00 stores last values for reliability

Key Insight:

Counters transform simple auto-increment into business-meaningful identifiers - "Order #000123" becomes "ORD-2025-NYC-00123" with metadata configuration, no code changes.

Why Counter Management Matters​

Simple Auto-Increment (Limited)​

Traditional approach:

CREATE TABLE TB_ANAG_ORD00 (
  ORD_ID INT PRIMARY KEY AUTO_INCREMENT,
  ...
);

INSERT INTO TB_ANAG_ORD00 (...) VALUES (...);
-- Result: ORD_ID = 123

Problems:

  • ❌ Not human-friendly ("Order 123" - what year? which store?)
  • ❌ No partitioning (sequence shared across all stores/years)
  • ❌ No formatting flexibility (can't add prefixes, separators)
  • ❌ Database-specific (MySQL AUTO_INCREMENT ≠ Oracle SEQUENCE)

Counter Management (Q01 Approach)​

Metadata-driven:

-- TB_COUNTER configuration
INSERT INTO TB_COUNTER (
  COD_DIM, FIELD_KEY, FIELD_TYPE,
  COUNTER_SQL, COUNTER_FORMAT, COUNTER_INITVAL
) VALUES (
  'ORD', 'XORD01', 'varchar',
  '{"select":"XORD01","from":"TB_ANAG_ORD00","where":"YEAR(CDATA)=YEAR(NOW())"}',
  'pad|%year%|2|0|LEFT__SEP__-__SEP__pad|#NEXT#|6|0|LEFT',
  1
);

Result: "ORD-25-000123" (auto-generated, partitioned by year, zero-padded)

Benefits:

  • ✅ Human-friendly format
  • ✅ Partitioned sequences (per year, store, category)
  • ✅ Configurable formatting
  • ✅ Concurrent-safe
  • ✅ Database-agnostic

TB_COUNTER Structure​

Table Schema​

CREATE TABLE TB_COUNTER (
  COUNTER_ID INT PRIMARY KEY AUTO_INCREMENT,
  COD_DIM VARCHAR(50),        -- Dimension code (e.g., 'ORD', 'INV')
  FIELD_KEY VARCHAR(100),     -- Field name (e.g., 'XORD01')
  FIELD_TYPE VARCHAR(50),     -- Data type ('int' or 'varchar')
  COUNTER_SQL TEXT,           -- JSON: SELECT query configuration
  COUNTER_FORMAT TEXT,        -- Template for formatting
  COUNTER_INITVAL INT         -- Starting value (default: 1)
);

Key Fields Explained​

FieldPurposeExample
COD_DIMDimension using counter"ORD", "INV", "PRD"
FIELD_KEYField to auto-populate"XORD01", "XINV_CODE"
FIELD_TYPEResult data type"int" (numeric) or "varchar" (formatted)
COUNTER_SQLJSON query config{"select":"...", "from":"...", "where":"..."}
COUNTER_FORMATFormat template`"pad
COUNTER_INITVALStarting number1, 1000, 10000

COUNTER_SQL Configuration​

JSON Structure​

{
  "select": "FIELD_NAME",
  "from": "TABLE_NAME",
  "where": "PARTITION_CONDITION",
  "extraselect": "ADDITIONAL_FIELDS"
}

Fields:

  • select: Field containing counter value
  • from: Table to query for MAX value
  • where: Optional partition condition (year, store, etc.)
  • extraselect: Additional fields to retrieve (optional)

Basic Counter (No Partitioning)​

Get next sequence from same table:

{
  "select": "XORD01",
  "from": "TB_ANAG_ORD00"
}

Generated SQL:

SELECT IFNULL(MAX(CAST(XORD01 AS UNSIGNED)), 0) + 1 AS NEXT
FROM TB_ANAG_ORD00
FOR UPDATE;

Result: Global sequence (1, 2, 3, ...) across all orders.

Year-Partitioned Counter​

Reset sequence each year:

{
  "select": "XORD01",
  "from": "TB_ANAG_ORD00",
  "where": "YEAR(CDATA) = YEAR(NOW())"
}

Generated SQL:

SELECT IFNULL(MAX(CAST(XORD01 AS UNSIGNED)), 0) + 1 AS NEXT
FROM TB_ANAG_ORD00
WHERE YEAR(CDATA) = YEAR(NOW())
FOR UPDATE;

Result: Per-year sequence:

  • 2024: 1, 2, 3, ...
  • 2025: 1, 2, 3, ... (resets)

Store-Partitioned Counter​

Separate sequence per store:

{
  "select": "XORD01",
  "from": "TB_ANAG_ORD00",
  "where": "XORD06 = '%codice_sede%'"
}

Generated SQL:

SELECT IFNULL(MAX(CAST(XORD01 AS UNSIGNED)), 0) + 1 AS NEXT
FROM TB_ANAG_ORD00
WHERE XORD06 = 'NYC'  -- From session variable
FOR UPDATE;

Result: Per-store sequence:

  • NYC: 1, 2, 3, ...
  • LAX: 1, 2, 3, ... (independent)

Code Reference: DataStore.php:158-165

$counterWhere = "";
if (isset($counterSql["where"]) && $counterSql["where"] != '') {
    $sess = $this->session->getArraySession();
    // Replace %varName% with session value
    $counterSql["where"] = $this->replaceVariablesInTemplate(
        $counterSql["where"],
        $sess["grant_session"],
        "%"
    );
    $counterWhere = " WHERE " . $counterSql["where"];
}

External Counter Table​

Use TB_ANAG_COUNT00 as counter source:

{
  "select": "XCOUNT03",
  "from": "TB_ANAG_COUNT00",
  "where": "XCOUNT01 = 'ORD' AND XCOUNT06 = YEAR(NOW())"
}

Use Case: Share counter across multiple dimensions or separate counter storage.

Code Reference: DataStore.php:180-185

// If counter table differs from dimension table, update counter table

COUNTER_FORMAT Template​

Template Syntax​

Format:

function|arg1|arg2|arg3|arg4__SEP__function|arg1|arg2|arg3|arg4

Separator: __SEP__ (joins multiple format segments)

Function: Currently only pad is supported

pad Function​

Syntax:

pad|input|length|padChar|direction

Parameters:

  • input: Value to pad (can use placeholders)
  • length: Target length after padding
  • padChar: Character to use for padding (e.g., 0, )
  • direction: LEFT, RIGHT, or BOTH

Placeholders​

PlaceholderTypeValueExample
%year%Date functionCurrent 2-digit year"25" (2025)
%day%Date functionCurrent day"19" (19th)
%now%Date functionYYYYMMDDHHmmss"20250119120000"
@varName@Session variableSession value@codice_sede@ → "NYC"
#NEXT#Query resultNext counter value"123"
#FIELD#Query resultExtra field from queryFrom extraselect
__fieldName__Request bodyField from POST body__XORD07__ → "RETAIL"

Code Reference: DataStore.php:232-271

Format Examples​

Example 1: Simple Year-Padded Counter​

Format:

pad|%year%|2|0|LEFT__SEP__pad|#NEXT#|6|0|LEFT

Breakdown:

  1. pad|%year%|2|0|LEFT → "25" (2025, padded to 2 digits)
  2. __SEP__ → Separator (nothing in output)
  3. pad|#NEXT#|6|0|LEFT → "000123" (sequence padded to 6 digits)

Result: "25000123"

Example 2: Formatted Invoice Number​

Format:

pad|INV|3|0|LEFT__SEP__-__SEP__pad|%year%|2|0|LEFT__SEP__-__SEP__pad|#NEXT#|5|0|LEFT

Breakdown:

  1. pad|INV|3|0|LEFT → "INV"
  2. - → Literal hyphen
  3. pad|%year%|2|0|LEFT → "25"
  4. - → Literal hyphen
  5. pad|#NEXT#|5|0|LEFT → "00123"

Result: "INV-25-00123"

Example 3: Store-Based Order Number​

Format:

pad|ORD|3|0|LEFT__SEP__-__SEP__pad|@codice_sede@|3| |LEFT__SEP__-__SEP__pad|#NEXT#|4|0|LEFT

Breakdown:

  1. pad|ORD|3|0|LEFT → "ORD"
  2. - → Hyphen
  3. pad|@codice_sede@|3| |LEFT → "NYC" (from session, space-padded)
  4. - → Hyphen
  5. pad|#NEXT#|4|0|LEFT → "0023"

Result: "ORD-NYC-0023"

Example 4: Body Field in Format​

Format:

pad|__XORD07__|6| |RIGHT__SEP__-__SEP__pad|#NEXT#|5|0|LEFT

Request Body:

{
  "XORD07": "RETAIL",
  "XORD01": "Customer Order"
}

Result: "RETAIL-00123"

COUNTER_INITVAL - Starting Value​

Purpose: Define minimum counter value.

Example:

COUNTER_INITVAL = 1     -- Start at 1 (default)
COUNTER_INITVAL = 1000  -- Start at 1000
COUNTER_INITVAL = 10000 -- Start at 10000

Behavior:

if ($initVal > $nextCounter) {
    $nextCounter = $initVal;
}

Use Cases:

  • Start invoice numbers at 10000 for professional appearance
  • Reserve ranges (1-999 for system, 1000+ for user)
  • Migrate from legacy system (continue from old max)

Code Reference: DataStore.php:145-151

if (is_numeric($dataCounter["COUNTER_INITVAL"]) && intval($dataCounter["COUNTER_INITVAL"]) > 0) {
    $initValIfNull = intval($dataCounter["COUNTER_INITVAL"]) - 1;
    $initVal = intval($dataCounter["COUNTER_INITVAL"]);
} else {
    $initValIfNull = 0;
    $initVal = 1;
}

TB_ANAG_COUNT00 - Counter Storage​

Purpose​

Persistent counter cache - stores last-used counter values for reliability.

Structure​

CREATE TABLE TB_ANAG_COUNT00 (
  COUNT_ID INT PRIMARY KEY AUTO_INCREMENT,
  XCOUNT01 VARCHAR(50),   -- Dimension code
  XCOUNT02 VARCHAR(100),  -- Field key
  XCOUNT03 INT,           -- Counter value
  XCOUNT04 VARCHAR(50),   -- Partition key (optional)
  XCOUNT05 VARCHAR(50),   -- Additional partition (optional)
  XCOUNT06 VARCHAR(4),    -- Year (for year-partitioned counters)
  -- Standard audit fields
  OWNER VARCHAR(50),
  CDATA VARCHAR(14),
  LOWNER VARCHAR(50),
  LDATA VARCHAR(14),
  TREC CHAR(1),
  TIMESTAMP BIGINT
);

Use Case​

Scenario: Dimension table doesn't exist yet, but you need counter.

Example: Shared invoice counter across multiple invoice types.

TB_COUNTER Configuration:

{
  "select": "XCOUNT03",
  "from": "TB_ANAG_COUNT00",
  "where": "XCOUNT01 = 'INV' AND XCOUNT06 = YEAR(NOW())"
}

Result: Counter stored in TB_ANAG_COUNT00 instead of dimension table.

Concurrency Safety​

FOR UPDATE Locking​

Every counter query uses FOR UPDATE:

SELECT IFNULL(MAX(CAST(XORD01 AS UNSIGNED)), 0) + 1 AS NEXT
FROM TB_ANAG_ORD00
WHERE YEAR(CDATA) = YEAR(NOW())
FOR UPDATE;  -- Row lock

Behavior:

  • First transaction locks row
  • Second transaction waits until first commits
  • No duplicate counters

Code Reference: DataStore.php:167

Transaction Flow​

Concurrent CREATE requests:

Transaction 1:

BEGIN;
SELECT MAX(XORD01) + 1 FROM TB_ANAG_ORD00 FOR UPDATE;  -- Returns 123, locks
INSERT INTO TB_ANAG_ORD00 (XORD01, ...) VALUES (123, ...);
COMMIT;  -- Releases lock

Transaction 2 (waiting):

BEGIN;
SELECT MAX(XORD01) + 1 FROM TB_ANAG_ORD00 FOR UPDATE;  -- Waits for T1 commit
-- After T1 commits, returns 124
INSERT INTO TB_ANAG_ORD00 (XORD01, ...) VALUES (124, ...);
COMMIT;

Result: No collision, sequential counters (123, 124).

Best Practices​

✅ DO: Use varchar for Formatted Counters​

Correct:

FIELD_TYPE = 'varchar',
COUNTER_FORMAT = 'pad|INV|3|0|LEFT__SEP__-__SEP__pad|#NEXT#|5|0|LEFT'

Result: "INV-00123" (string)

✅ DO: Partition by Logical Boundaries​

Examples:

  • Per year: "YEAR(CDATA) = YEAR(NOW())"
  • Per store: "XORD05 = '%codice_sede%'"
  • Per category: "XPRD05 = '__category__'"

Benefit: Logical grouping, predictable sequences.

✅ DO: Use COUNTER_INITVAL for Professional Appearance​

Start invoices at 10000:

COUNTER_INITVAL = 10000

Result: First invoice = "INV-25-010000" (looks established)

✅ DO: Test Counter with Concurrent Requests​

Simulate concurrent creation:

# Terminal 1
curl -X POST /api/v4/core/ORD -d '{"XORD01":"Order A"}'

# Terminal 2 (immediately after)
curl -X POST /api/v4/core/ORD -d '{"XORD01":"Order B"}'

Verify: No duplicate XORD01 values.

❌ DON'T: Use Counters Without Partitioning for Multi-Tenant​

Wrong:

{
  "select": "XORD01",
  "from": "TB_ANAG_ORD00"
}

Problem: All tenants share same sequence → security/privacy issue.

Right:

{
  "select": "XORD01",
  "from": "TB_ANAG_ORD00",
  "where": "XORD08 = '%ambiente%'"
}

Result: Separate sequence per tenant.

❌ DON'T: Change COUNTER_FORMAT After Production Use​

Problem: Inconsistent formatting across records.

Example:

  • Before: "INV-25-00123"
  • After change: "2025-INV-123"

Result: Mixed formats confuse users.

Solution: Migrate all existing values or use new counter field.

❌ DON'T: Use Small Padding for High-Volume Counters​

Wrong:

pad|#NEXT#|3|0|LEFT  -- Max 999

Problem: Exceeds format after 999 records.

Right:

pad|#NEXT#|6|0|LEFT  -- Max 999,999

Rule: Estimate max records, add buffer (10-year volume × 2).

Common Pitfalls​

Pitfall 1: Forgetting FOR UPDATE Lock​

Problem: Custom counter queries without FOR UPDATE → duplicates.

Wrong:

SELECT MAX(XORD01) + 1 FROM TB_ANAG_ORD00;  -- No lock!

Right:

SELECT MAX(XORD01) + 1 FROM TB_ANAG_ORD00 FOR UPDATE;

Pitfall 2: Using Session Variables in COUNTER_FORMAT​

Problem: Session variable not available at format time.

Wrong:

pad|%codice_sede%|3|0|LEFT  -- % prefix only works in COUNTER_SQL where clause

Right:

pad|@codice_sede@|3|0|LEFT  -- @ prefix for session variables in format

Pitfall 3: Not Testing Year Rollover​

Problem: Year-partitioned counter breaks at year boundary.

Test:

-- Manually set CDATA to December 31, 2024
UPDATE TB_ANAG_ORD00 SET CDATA = '20241231235959' WHERE ORD_ID = 1;

-- Create order (should reset counter for 2025)
POST /api/v4/core/ORD {...}

-- Verify counter = 1 for 2025

Summary​

Counter Management provides:

  1. ✅ Business-friendly IDs - "INV-2025-00123" instead of "123"
  2. ✅ Flexible formatting - Templates with year, padding, prefixes
  3. ✅ Logical partitioning - Per year, store, category, etc.
  4. ✅ Concurrency safety - FOR UPDATE locking prevents duplicates
  5. ✅ Database abstraction - Works with MySQL and Oracle
  6. ✅ Metadata-driven - Configure in TB_COUNTER, no code changes

Key Insight:

Counters are the ID beautification layer - transforming technical auto-increment into human-meaningful, formatted, partitioned business identifiers. Define format once in TB_COUNTER, benefit everywhere.
