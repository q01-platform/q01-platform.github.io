What is the Q01 Platform?

Platform vs. Application​

Q01 is a platform, not an application. This distinction is fundamental to understanding Core APIs and their role in the ecosystem.

Application Thinking​

Traditional applications:

  • Have direct database access
  • Implement business logic coupled to data access
  • Scale vertically (add resources to single application)
  • Are monolithic or service-oriented around business domains

Platform Thinking​

Q01 platform:

  • Forbids direct database access from application services
  • Separates data access (Core APIs) from business logic (application microservices)
  • Scales horizontally with shared metadata infrastructure
  • Is metadata-driven - behavior configured through metadata, not code

Why This Architecture?​

1. Enforce Data Consistency​

By channeling all data access through Core APIs:

  • Single validation point - all writes validated consistently
  • Single security enforcement - permissions checked uniformly
  • Single audit trail - all changes tracked in one place
  • Prevents rogue writes - no application can bypass validation

2. Multi-Tenancy at Scale​

Core APIs provide:

  • Tenant isolation - automatic filtering by ambiente (environment)
  • Shared metadata - one metadata schema serves all tenants
  • Independent databases - each tenant has separate operational database
  • Transparent routing - applications don't manage tenant context

3. Database Abstraction​

Applications don't know or care about:

  • Which database engine (MySQL vs Oracle)
  • Which tenant database to connect to
  • How to construct tenant-aware queries
  • How to handle sequences/auto-increment differences

Core APIs handle all database-specific logic.

4. Metadata-Driven Flexibility​

Add new entity types or fields without touching application code:

  • Register dimension in TB_DIM
  • Define fields in TB_COST
  • Configure permissions in TB_MENU
  • Core APIs automatically handle the new structure

5. CQRS Performance​

Reads and writes optimized independently:

  • Read operations use denormalized views, caching
  • Write operations ensure transactional consistency
  • Transparent to applications - same API surface

Core APIs: The Data Contract​

Core APIs are the contract between applications and data. They guarantee:

  1. Data will be validated according to metadata rules
  2. Permissions will be enforced at multiple levels
  3. Changes will be consistent across distributed systems (outbox pattern)
  4. Tenant data will be isolated automatically
  5. Database differences will be abstracted transparently

Applications trust Core APIs to handle these concerns correctly.

Application Responsibilities​

If Core APIs handle data access, what do applications do?

Applications Should:​

  • Implement business workflows (e.g., order fulfillment process)
  • Orchestrate across domains (e.g., create order → reserve inventory → notify customer)
  • Handle external integrations (payment gateways, shipping providers)
  • Implement domain-specific rules (e.g., discount calculation logic)
  • Render user interfaces (web, mobile, POS)
  • Real-time features (WebSocket notifications, collaborative editing)

Applications Should NOT:​

  • ❌ Access databases directly
  • ❌ Implement data validation (Core APIs do this via metadata)
  • ❌ Manage tenant context (Core APIs handle this via session)
  • ❌ Construct SQL queries (Core APIs generate these from metadata)
  • ❌ Enforce field-level permissions (Core APIs use COD_ON_OFF)

The MAP Framework​

MAP (Metadata Application Platform) is the heart of Q01's metadata-driven approach.

All behavior is defined in metadata tables:

  • TB_DIM - Dimensions (entity types like products, customers, orders)
  • TB_COST - Fields (column definitions, types, visibility, validation)
  • TB_TYPE_FIELDS - Field types (AC, check, combo, counter, DATE, int, text, etc.)
  • TB_MENU - Security areas (hierarchical permission zones)
  • TB_OBJECT - Objects (document hierarchies, DB views via TYPE_OBJ=1, select-of-selects via TYPE_OBJ=2)
  • TB_COUNTER - Counters (sequential identifier generation)

Core APIs interpret this metadata at runtime to:

  • Build SQL queries dynamically
  • Validate field values against rules
  • Filter visible fields by context
  • Enforce permissions at multiple levels
  • Generate unique identifiers
  • Execute cascading operations

Zero code deployment required to add entities, fields, or change behavior.

Multi-Tenant SaaS Model​

Q01 is designed for multi-tenant SaaS platforms where:

  • Each tenant has a dedicated database - complete data isolation at the database level
  • Shared metadata infrastructure - TB_DIM, TB_COST, TB_MENU, etc. are centralized
  • Tenants can have different environments (dev, staging, production)
  • Users within tenants have different permission levels (peso)

Tenant-Specific Customization​

When a microservice is deployed for a specific tenant:

  1. Development team creates the microservice with base dimension definitions (e.g., TB_ANAG_UT00 with 3 columns)
  2. Deployment process creates tenant-specific database and tables from the base schema
  3. Tenant administrators can customize via Metadata Manager:
    • Add custom fields to existing dimensions
    • Create new custom objects
    • Configure permissions and visibility rules
    • All via metadata configuration - no code changes required

Example: Team A develops msusers with TB_ANAG_UT00. When deployed to "Tenant Demo", the platform:

  • Creates a dedicated database for Tenant Demo
  • Provisions TB_ANAG_UT00 with the base 3 columns
  • Tenant Demo can then add custom fields via Metadata Manager
  • Changes are tenant-specific and don't affect other tenants

Core APIs manage this complexity through:

  • Dedicated databases - each tenant has isolated DB instance
  • Ambiente filtering - automatic data isolation by environment within tenant
  • Session management - tenant context travels with every request
  • Grant system - hierarchical permissions via TB_MENU
  • Weight-based access - field visibility controlled by user peso

Key Principles​

Understanding Q01 Core APIs requires embracing these principles:

  1. Code is configuration, metadata is code

    • Don't write code to add features
    • Configure metadata to enable functionality

  2. Context determines visibility

    • What you see depends on who you are (peso), where you are (source), and what you're doing (center_dett)

  3. Trust the platform

    • Don't re-implement validation in applications
    • Let Core APIs enforce consistency

  4. Metadata is the contract

    • Core APIs guarantee to respect metadata rules
    • Applications can rely on this guarantee

  5. Everything is a dimension

    • Any data structure can be registered as a dimension
    • Once registered, Core APIs provide full CRUD automatically

Real-World Example​

Let's see this in action:

Traditional Approach​

// Application code - direct database access ❌
async function createProduct(data) {
  // Manual validation
  if (!data.name) throw new Error("Name required");
  if (data.price < 0) throw new Error("Price invalid");

  // Manual tenant isolation
  const tenantDB = getTenantDatabase(request.tenantId);

  // Manual SQL construction
  const sql = `
    INSERT INTO products (name, price, tenant_id, created_by, created_at)
    VALUES (?, ?, ?, ?, NOW())
  `;

  // Execute
  await tenantDB.execute(sql, [data.name, data.price, request.tenantId, request.userId]);

  // Manual audit
  await auditLog.create({...});
}

Q01 Platform Approach​

// Application code - use Core APIs ✅
async function createProduct(data) {
  // Core APIs handle validation, tenant isolation, audit automatically
  const response = await fetch('https://coreservice/api/v4/core/PRD', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${token}`,  // Tenant context in JWT
    },
    body: JSON.stringify({
      source: 'productManagement',  // Security area
      XPRD01: data.name,            // Field names from metadata
      XPRD02: data.price,
    })
  });

  return response.json();
  // Validation, permissions, tenant isolation, sequences, audit trail,
  // outbox events all handled by Core APIs based on metadata
}

The application code is dramatically simpler because Core APIs handle all cross-cutting concerns.

Summary​

Q01 is a platform architecture where:

  • Core APIs are the ONLY data access layer
  • Applications implement business logic, not data access
  • Metadata defines behavior, not code
  • Multi-tenancy is built-in, not bolted-on
  • Consistency and security are guaranteed by the platform

Understanding this architecture is crucial to using Core APIs effectively.
