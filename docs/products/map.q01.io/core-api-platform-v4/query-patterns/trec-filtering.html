TREC Filtering

Overview​

TREC (Transaction Record) tracks the lifecycle state of every record. The cond_trec parameter filters records by their TREC state.

TREC States:

  • N - New (newly created)
  • M - Modified (updated)
  • C - Cancelled (soft deleted)
  • P - Processed (special state, rare)

Default TREC Filtering​

Automatic Exclusion of Cancelled Records​

By default, queries exclude TREC='C':

GET /api/v4/core/PRD?source=productList

Generated SQL:

WHERE TREC != 'C'

Result: Returns only N, M, P records (active records).

Why? Cancelled records are "soft deleted" and should not appear in normal queries.

cond_trec Parameter​

Include Specific States​

Show only new records:

GET /api/v4/core/PRD?source=productList&\
  cond_trec=N

Show new and modified:

GET /api/v4/core/PRD?source=productList&\
  cond_trec=N,M

Show cancelled records:

GET /api/v4/core/PRD?source=productList&\
  cond_trec=C

Show all records (including cancelled):

GET /api/v4/core/PRD?source=productList&\
  cond_trec=N,M,C,P

Common Use Cases​

Use Case 1: Active Records Only (Default)​

Most common - normal queries:

GET /api/v4/core/PRD?source=productList
# Automatically excludes TREC='C'

Use Case 2: Audit Trail (All Records)​

Show everything including deleted:

GET /api/v4/core/PRD?source=productAudit&\
  cond_trec=N,M,C&\
  $order=LDATA DESC

Use: Audit logs, compliance reports, data recovery.

Use Case 3: Recently Modified Only​

Show only updated records:

GET /api/v4/core/PRD?source=productChanges&\
  cond_trec=M&\
  $filter=LDATA ge '20251201000000'

Use: Change tracking, sync operations.

Use Case 4: Soft Deleted Records​

Show cancelled records for recovery:

GET /api/v4/core/PRD?source=productTrash&\
  cond_trec=C&\
  $order=LDATA DESC

Use: "Trash" view, undelete functionality.

TREC Lifecycle​

State Transitions​

NEW (N) ──────┐
              ├──→ MODIFIED (M) ──→ CANCELLED (C)
              └────────────────────────→ CANCELLED (C)

Flow:

  1. POST creates record with TREC='N'
  2. PUT/PATCH updates to TREC='M'
  3. DELETE (soft) sets TREC='C'
  4. DELETE (force) physically removes record

Audit Fields Updated with TREC​

On CREATE (TREC='N'):

  • OWNER = current user
  • CDATA = current timestamp
  • TREC = 'N'

On UPDATE (TREC='M'):

  • LOWNER = current user
  • LDATA = current timestamp
  • TREC = 'M'

On DELETE (TREC='C'):

  • LOWNER = current user
  • LDATA = current timestamp
  • TREC = 'C'

Combining TREC with Filters​

TREC + Business Logic​

Active products in Electronics:

GET /api/v4/core/PRD?source=productList&\
  cond_trec=N,M&\
  $filter=XPRD05 eq 'Electronics' AND XPRD06 eq true

Recently deleted Electronics:

GET /api/v4/core/PRD?source=productTrash&\
  cond_trec=C&\
  $filter=XPRD05 eq 'Electronics' AND LDATA ge '20251201000000'

JavaScript Examples​

TREC Filter Helper​

class TrecFilter {
  static ACTIVE = 'N,M';        // Normal queries
  static ALL = 'N,M,C,P';       // Audit queries
  static NEW_ONLY = 'N';        // New records only
  static MODIFIED_ONLY = 'M';   // Modified records only
  static DELETED = 'C';         // Soft deleted records

  static buildUrl(baseUrl, trecFilter) {
    const url = new URL(baseUrl);
    if (trecFilter) {
      url.searchParams.append('cond_trec', trecFilter);
    }
    return url;
  }
}

// Usage
const activeUrl = TrecFilter.buildUrl(
  `${apiBase}/api/v4/core/PRD?source=productList`,
  TrecFilter.ACTIVE
);

const trashUrl = TrecFilter.buildUrl(
  `${apiBase}/api/v4/core/PRD?source=productTrash`,
  TrecFilter.DELETED
);

Audit Trail Component​

async function fetchAuditTrail(dimension, recordId) {
  const url = new URL(`${apiBase}/api/v4/core/${dimension}`);
  url.searchParams.append('source', 'audit');
  url.searchParams.append('cond_trec', 'N,M,C'); // All states
  url.searchParams.append('$filter', `${dimension}_ID eq ${recordId}`);
  url.searchParams.append('$order', 'LDATA DESC');

  const response = await fetch(url, {
    headers: { 'Authorization': `Bearer ${token}` }
  });

  const result = await response.json();

  return result.data.map(record => ({
    state: record.TREC,
    timestamp: record.LDATA || record.CDATA,
    user: record.LOWNER || record.OWNER,
    action: getTrecActionName(record.TREC)
  }));
}

function getTrecActionName(trec) {
  switch (trec) {
    case 'N': return 'Created';
    case 'M': return 'Modified';
    case 'C': return 'Deleted';
    default: return 'Unknown';
  }
}

Best Practices​

✅ DO:​

Rely on default for normal queries:

# ✅ Good - automatic TREC filtering
GET /api/v4/core/PRD?source=productList

Explicitly include cancelled for audit:

# ✅ Good - audit trail
cond_trec=N,M,C

Filter deleted by date:

# ✅ Good - recently deleted
cond_trec=C&$filter=LDATA ge '20251201000000'

❌ DON'T:​

Don't include cancelled in normal queries:

# ❌ Bad - shows deleted records
cond_trec=N,M,C

Don't forget cond_trec in audit queries:

# ❌ Bad - missing deleted records
GET /api/v4/core/PRD?source=audit
# Should use: cond_trec=N,M,C

Common Mistakes​

Mistake 1: Forgetting Cancelled Records in Audit​

# ❌ Wrong - missing deleted records
GET /api/v4/core/PRD?source=audit

# ✅ Correct
GET /api/v4/core/PRD?source=audit&cond_trec=N,M,C

Mistake 2: Including Cancelled in Normal List​

# ❌ Wrong - shows deleted products
GET /api/v4/core/PRD?source=productList&cond_trec=N,M,C

# ✅ Correct - rely on default
GET /api/v4/core/PRD?source=productList

Summary​

  • ✅ Default excludes TREC='C' (cancelled)
  • ✅ Use cond_trec to control which states to include
  • ✅ Format: comma-separated states (N,M,C,P)
  • ✅ Include C for audit trails and trash views
  • ✅ TREC changes automatically on CREATE/UPDATE/DELETE

Key Takeaways:

  1. Normal queries: rely on default (no cond_trec)
  2. Audit queries: use cond_trec=N,M,C
  3. Trash view: use cond_trec=C
  4. TREC tracks record lifecycle automatically
