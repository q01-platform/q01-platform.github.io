Query Performance

Overview​

Query performance determines user experience. Core APIs provide efficient data access, but query design matters. This guide covers optimization strategies, common pitfalls, and performance monitoring.

Key Factors:

  • Index usage
  • Field projection
  • Filter placement
  • Pagination strategy
  • N+1 query problems
  • Caching

Index Optimization​

Understanding Indexes​

Indexes speed up queries by creating ordered data structures:

-- Without index: Full table scan (slow)
SELECT * FROM TB_ANAG_PRD00 WHERE XPRD05 = 'Electronics'; -- Scans all rows

-- With index: Index seek (fast)
CREATE INDEX idx_category ON TB_ANAG_PRD00(XPRD05);
SELECT * FROM TB_ANAG_PRD00 WHERE XPRD05 = 'Electronics'; -- Uses index

Performance Impact:

  • Without index: O(n) - scans every row
  • With index: O(log n) - binary search

Example:

  • 100,000 records without index: ~500ms
  • 100,000 records with index: ~5ms (100x faster)

Indexed Fields in Core APIs​

Primary Key (Always Indexed):

# ✅ Fast - primary key lookup
GET /api/v4/core/PRD/123

Counter Fields (XPRD03 - Indexed):

# ✅ Fast - counter field is indexed
$filter=XPRD03 eq 'PRD-2025-001'

Category/Foreign Keys (Usually Indexed):

# ✅ Fast - foreign key index
$filter=XPRD05 eq 'cat_electronics'

Text Fields (Not Indexed by Default):

# ⚠️ Slower - no index on text field
$filter=contains(XPRD04,'description')

Multi-Column Indexes​

Indexes support left-to-right matching:

-- Index on (XPRD05, XPRD06)
CREATE INDEX idx_category_active ON TB_ANAG_PRD00(XPRD05, XPRD06);

Uses Index:

# ✅ Good - uses index
$filter=XPRD05 eq 'Electronics' AND XPRD06 eq true

Partial Index Usage:

# ✅ Good - uses index (first column)
$filter=XPRD05 eq 'Electronics'

No Index Usage:

# ❌ Bad - cannot use index (missing first column)
$filter=XPRD06 eq true

Field Projection​

Response Size Reduction​

Field projection dramatically reduces response size:

# ❌ Bad - 10KB response, 250ms query
GET /api/v4/core/PRD?source=productList

# ✅ Good - 1KB response, 25ms query
GET /api/v4/core/PRD?source=productList&\
  $select=PRD_ID,XPRD01,XPRD02

Performance Impact:

  • 90% smaller response size
  • 10x faster network transfer
  • 5x faster JSON parsing
  • Lower memory usage

Select Only Needed Fields​

List views (minimal fields):

# ✅ Good - list view
$select=PRD_ID,XPRD01,XPRD02,XPRD12

Detail views (more fields):

# ✅ Good - detail view
$select=PRD_ID,XPRD01,XPRD02,XPRD04,XPRD05,XPRD06,XPRD09,XPRD12

Autocomplete (absolute minimum):

# ✅ Good - autocomplete
$select=PRD_ID,XPRD01

Filter Placement​

Filter Order Matters​

Apply most selective filters first:

# ✅ Good - specific filter first (reduces result set early)
$filter=XPRD03 eq 'PRD-2025-001' AND XPRD06 eq true

# ⚠️ Slower - broad filter first
$filter=XPRD06 eq true AND XPRD03 eq 'PRD-2025-001'

Why? Database query optimizer evaluates filters left-to-right (in most cases). Specific filters reduce the working set faster.

Indexed Filter First​

# ✅ Good - indexed field first
$filter=XPRD05 eq 'Electronics' AND contains(XPRD04,'description')

# ⚠️ Slower - non-indexed field first
$filter=contains(XPRD04,'description') AND XPRD05 eq 'Electronics'

Pagination Strategy​

Limit Result Sets​

Always paginate large queries:

# ❌ Bad - fetches all records (slow, memory intensive)
GET /api/v4/core/PRD?source=productList

# ✅ Good - limited result set
GET /api/v4/core/PRD?source=productList&$num_rows=25&$offset=0

Performance Impact:

  • 100,000 records without pagination: 5-10 seconds, 50MB response
  • 25 records with pagination: 50ms, 50KB response (100x faster)

Deep Pagination Problem​

Performance degrades with high $offset:

# ✅ Fast - page 1 (~25ms)
$offset=0&$num_rows=25

# ⚠️ Slower - page 100 (~100ms)
$offset=2475&$num_rows=25

# ❌ Very slow - page 1000 (~1000ms)
$offset=24975&$num_rows=25

Why? Database must scan and skip first N rows before returning results.

Solution 1: Limit maximum page number

const MAX_PAGE = 100;
if (pageNumber > MAX_PAGE) {
  throw new Error('Page number too high');
}

Solution 2: Cursor-based pagination

# Page 1
GET /api/v4/core/PRD?$num_rows=25&$order=PRD_ID ASC

# Page 2 - use last ID as cursor
GET /api/v4/core/PRD?$filter=PRD_ID gt 25&$num_rows=25&$order=PRD_ID ASC

Consistent Sorting​

Always specify $order with pagination:

# ❌ Bad - unpredictable order (duplicates/missing records)
$num_rows=25&$offset=0
$num_rows=25&$offset=25

# ✅ Good - consistent order
$num_rows=25&$offset=0&$order=PRD_ID ASC
$num_rows=25&$offset=25&$order=PRD_ID ASC

N+1 Query Problem​

The Problem​

Fetching related data in loops causes N+1 queries:

// ❌ Bad - N+1 queries (1 + 100 = 101 API calls)
async function loadProductsWithCategory() {
  // 1 query to get products
  const products = await fetch(`${apiBase}/api/v4/core/PRD?$num_rows=100`);

  // N queries to get each category (100 additional calls)
  for (const product of products.data) {
    const category = await fetch(`${apiBase}/api/v4/core/CAT/${product.XPRD05}`);
    product.categoryName = category.data.XCAT01;
  }
}

Solution 1: Use TB_OBJECT Views​

Join data in single query:

-- Define view in TB_OBJECT
INSERT INTO TB_OBJECT (NAME_OBJ, COD_DIM, TYPE_OBJ, VIEW_SCHEMA) VALUES (
  'V_PRODUCTS_CATEGORY', 'PRD', 1,
  'SELECT p.PRD_ID, p.XPRD01, p.XPRD02, c.XCAT01 as CATEGORY_NAME
   FROM TB_ANAG_PRD00 p
   LEFT JOIN TB_ANAG_CAT00 c ON p.XPRD05 = c.CAT_ID
   WHERE p.TREC != ''C'''
);
// ✅ Good - single query with join
async function loadProductsWithCategory() {
  const products = await fetch(
    `${apiBase}/api/v4/core/PRD?source=productsWithCategory&$num_rows=100`
  );
  // All data included in single response
}

Solution 2: Batch Fetching​

Fetch related records in single request:

// ✅ Good - 2 queries instead of 101
async function loadProductsWithCategory() {
  // 1 query to get products
  const products = await fetch(`${apiBase}/api/v4/core/PRD?$num_rows=100`);

  // Extract unique category IDs
  const categoryIds = [...new Set(products.data.map(p => p.XPRD05))];

  // 1 query to get all categories
  const categories = await fetch(
    `${apiBase}/api/v4/core/CAT?$filter=CAT_ID IN (${categoryIds.join(',')})`
  );

  // Map categories to products
  const categoryMap = {};
  categories.data.forEach(cat => categoryMap[cat.CAT_ID] = cat.XCAT01);
  products.data.forEach(p => p.categoryName = categoryMap[p.XPRD05]);
}

Caching Strategies​

Client-Side Caching​

Cache API responses in browser:

class CachedAPIClient {
  constructor(apiBase, token) {
    this.apiBase = apiBase;
    this.token = token;
    this.cache = new Map();
    this.cacheTTL = 5 * 60 * 1000; // 5 minutes
  }

  getCacheKey(url) {
    return url.toString();
  }

  isCacheValid(entry) {
    return Date.now() - entry.timestamp < this.cacheTTL;
  }

  async fetch(url) {
    const key = this.getCacheKey(url);
    const cached = this.cache.get(key);

    // Return cached if valid
    if (cached && this.isCacheValid(cached)) {
      return cached.data;
    }

    // Fetch from API
    const response = await fetch(url, {
      headers: { 'Authorization': `Bearer ${this.token}` }
    });
    const data = await response.json();

    // Store in cache
    this.cache.set(key, {
      data,
      timestamp: Date.now()
    });

    return data;
  }

  invalidate(pattern) {
    // Clear cache entries matching pattern
    for (const key of this.cache.keys()) {
      if (key.includes(pattern)) {
        this.cache.delete(key);
      }
    }
  }
}

// Usage
const api = new CachedAPIClient(apiBase, token);

// First call - fetches from API
const products1 = await api.fetch(
  new URL(`${apiBase}/api/v4/core/PRD?source=productList&$num_rows=25`)
);

// Second call within 5 minutes - returns cached
const products2 = await api.fetch(
  new URL(`${apiBase}/api/v4/core/PRD?source=productList&$num_rows=25`)
);

// Invalidate cache after update
await updateProduct(123);
api.invalidate('/api/v4/core/PRD');

Cache Invalidation​

Invalidate cache after writes:

class ProductService {
  constructor(apiBase, token) {
    this.api = new CachedAPIClient(apiBase, token);
  }

  async getProducts(options) {
    const url = new URL(`${this.api.apiBase}/api/v4/core/PRD`);
    url.searchParams.append('source', 'productList');
    // ... add options
    return this.api.fetch(url);
  }

  async updateProduct(id, data) {
    // Update product
    const response = await fetch(`${this.api.apiBase}/api/v4/core/PRD/${id}`, {
      method: 'PUT',
      headers: {
        'Authorization': `Bearer ${this.api.token}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify(data)
    });

    // Invalidate product list cache
    this.api.invalidate('/api/v4/core/PRD');

    return response.json();
  }
}

HTTP Cache Headers​

Core APIs support standard HTTP caching:

// Let browser handle caching
const response = await fetch(url, {
  headers: { 'Authorization': `Bearer ${token}` },
  cache: 'default' // Use browser cache
});

Cache-Control headers:

  • Cache-Control: public, max-age=300 - Cache for 5 minutes
  • Cache-Control: no-cache - Validate before use
  • Cache-Control: no-store - Never cache

Performance Monitoring​

Measure Query Performance​

class PerformanceMonitor {
  static async measureQuery(queryFn, label) {
    const startTime = performance.now();
    const result = await queryFn();
    const endTime = performance.now();
    const duration = endTime - startTime;

    console.log(`[${label}] Query took ${duration.toFixed(2)}ms`);

    // Alert if slow
    if (duration > 1000) {
      console.warn(`[${label}] Slow query detected: ${duration.toFixed(2)}ms`);
    }

    return result;
  }
}

// Usage
const products = await PerformanceMonitor.measureQuery(
  () => fetch(`${apiBase}/api/v4/core/PRD?source=productList&$num_rows=25`),
  'Product List'
);

Common Performance Pitfalls​

Pitfall 1: Fetching All Records​

// ❌ Bad - fetches 100k records
const products = await fetch(`${apiBase}/api/v4/core/PRD?source=productList`);

// ✅ Good - fetches 25 records
const products = await fetch(`${apiBase}/api/v4/core/PRD?source=productList&$num_rows=25`);

Pitfall 2: No Field Projection​

// ❌ Bad - returns all 50 fields (500KB response)
const products = await fetch(`${apiBase}/api/v4/core/PRD?source=productList&$num_rows=25`);

// ✅ Good - returns 3 fields (50KB response, 10x faster)
const products = await fetch(
  `${apiBase}/api/v4/core/PRD?source=productList&$num_rows=25&$select=PRD_ID,XPRD01,XPRD02`
);

Pitfall 3: N+1 Queries​

// ❌ Bad - 101 API calls
for (const product of products) {
  const category = await fetch(`${apiBase}/api/v4/core/CAT/${product.XPRD05}`);
}

// ✅ Good - 1 API call with TB_OBJECT view
const products = await fetch(
  `${apiBase}/api/v4/core/PRD?source=productsWithCategory&$num_rows=100`
);

Pitfall 4: No Sorting with Pagination​

// ❌ Bad - inconsistent results across pages
const page1 = await fetch(`${apiBase}/api/v4/core/PRD?$num_rows=25&$offset=0`);
const page2 = await fetch(`${apiBase}/api/v4/core/PRD?$num_rows=25&$offset=25`);

// ✅ Good - consistent results
const page1 = await fetch(`${apiBase}/api/v4/core/PRD?$num_rows=25&$offset=0&$order=PRD_ID ASC`);
const page2 = await fetch(`${apiBase}/api/v4/core/PRD?$num_rows=25&$offset=25&$order=PRD_ID ASC`);

Pitfall 5: Leading Wildcard Search​

// ❌ Bad - full table scan on 100k records
const products = await fetch(
  `${apiBase}/api/v4/core/PRD?$filter=contains(XPRD01,'widget')`
);

// ✅ Good - prefix search uses index
const products = await fetch(
  `${apiBase}/api/v4/core/PRD?$filter=startwith(XPRD01,'widget')`
);

Best Practices Summary​

✅ DO:​

Use indexes:

# ✅ Filter by indexed fields
$filter=XPRD05 eq 'Electronics'

Project fields:

# ✅ Select only needed fields
$select=PRD_ID,XPRD01,XPRD02

Always paginate:

# ✅ Limit result sets
$num_rows=25&$offset=0

Sort consistently:

# ✅ Specify $order with pagination
$order=PRD_ID ASC

Use TB_OBJECT views:

-- ✅ Join data in database
CREATE VIEW V_PRODUCTS_CATEGORY AS SELECT ...

❌ DON'T:​

Don't fetch all records:

# ❌ No pagination
GET /api/v4/core/PRD?source=productList

Don't skip field projection:

# ❌ Returns all fields
GET /api/v4/core/PRD?source=productList

Don't use N+1 queries:

// ❌ Loop with API calls
for (const product of products) {
  await fetch(`${apiBase}/api/v4/core/CAT/${product.XPRD05}`);
}

Don't paginate without sorting:

# ❌ Missing $order
$num_rows=25&$offset=25

Summary​

  • ✅ Index usage is critical for performance
  • ✅ Field projection reduces response size 10-17x
  • ✅ Pagination is mandatory for large datasets
  • ✅ Consistent sorting prevents duplicate/missing records
  • ✅ TB_OBJECT views solve N+1 query problems
  • ✅ Caching reduces API calls and improves UX
  • ✅ Monitor performance to identify slow queries

Key Takeaways:

  1. Always use indexed fields in filters
  2. Project only needed fields ($select)
  3. Paginate all list queries ($num_rows/$offset)
  4. Sort consistently with pagination ($order)
  5. Avoid N+1 queries (use TB_OBJECT views)
  6. Cache read-heavy queries (5-minute TTL)
  7. Monitor and optimize slow queries (>1s)

Performance Checklist:

  • Filters use indexed fields
  • $select reduces response size
  • $num_rows limits result set
  • $order ensures consistency
  • No N+1 query patterns
  • Caching for read-heavy queries
  • Performance monitoring enabled
