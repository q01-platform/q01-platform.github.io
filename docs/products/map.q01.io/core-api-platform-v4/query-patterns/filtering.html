Filtering

Overview​

The Core APIs support powerful filtering capabilities for narrowing query results. Filters can be applied to any field marked with the R flag (search/filter) in COD_ON_OFF.

Key Features:

  • ✅ Multiple filter operators (equals, not equals, greater than, less than, like, in, between)
  • ✅ Logical combinations (AND, OR)
  • ✅ Nested conditions
  • ✅ Special TREC filtering via cond_trec parameter
  • ✅ Case-sensitive and case-insensitive matching
  • ✅ NULL value filtering

Filter Syntax:

?filter[FIELD_NAME][$operator]=value

Basic Filter Operators​

Equals​

Filter records where field equals specific value:

GET /api/v4/core/ART?source=settings-2&filter[XART06]=1
# WHERE XART06 = 1 (active products only)

Multiple equals (implicit OR):

GET /api/v4/core/ART?source=settings-2&filter[XART01]=1&filter[XART01]=2
# WHERE XART01 = 1 OR XART01 = 2

Not Equals ($ne)​

Exclude records with specific value:

GET /api/v4/core/ART?source=settings-2&filter[TREC][$ne]=C
# WHERE TREC != 'C' (exclude soft-deleted records)

Greater Than ($gt) / Greater Than or Equal ($gte)​

Filter by numeric or date comparison:

# Greater than
GET /api/v4/core/ART?source=settings-2&filter[XART25][$gt]=100
# WHERE XART25 > 100 (price greater than 100)

# Greater than or equal
GET /api/v4/core/ART?source=settings-2&filter[XART25][$gte]=100
# WHERE XART25 >= 100

Less Than ($lt) / Less Than or Equal ($lte)​

Filter by upper bound:

# Less than
GET /api/v4/core/ART?source=settings-2&filter[XART25][$lt]=50
# WHERE XART25 < 50 (price less than 50)

# Less than or equal
GET /api/v4/core/ART?source=settings-2&filter[XART25][$lte]=50
# WHERE XART25 <= 50

IN ($in)​

Match any value in a list:

GET /api/v4/core/ART?source=settings-2&filter[XART01][$in]=1,2,3
# WHERE XART01 IN (1, 2, 3) (categories 1, 2, or 3)

NOT IN ($nin)​

Exclude multiple values:

GET /api/v4/core/ART?source=settings-2&filter[XART01][$nin]=4,5,6
# WHERE XART01 NOT IN (4, 5, 6)

LIKE ($like)​

Pattern matching (SQL LIKE):

# Starts with
GET /api/v4/core/ART?source=settings-2&filter[XART20][$like]=Widget%
# WHERE XART20 LIKE 'Widget%'

# Ends with
GET /api/v4/core/ART?source=settings-2&filter[XART20][$like]=%Pro
# WHERE XART20 LIKE '%Pro'

# Contains
GET /api/v4/core/ART?source=settings-2&filter[XART20][$like]=%Premium%
# WHERE XART20 LIKE '%Premium%'

BETWEEN ($between)​

Range filtering:

GET /api/v4/core/ART?source=settings-2&filter[XART25][$between]=50,100
# WHERE XART25 BETWEEN 50 AND 100 (price between 50 and 100)

IS NULL ($isnull) / IS NOT NULL ($notnull)​

Filter by NULL values:

# IS NULL
GET /api/v4/core/ART?source=settings-2&filter[XART53][$isnull]=true
# WHERE XART53 IS NULL

# IS NOT NULL
GET /api/v4/core/ART?source=settings-2&filter[XART53][$notnull]=true
# WHERE XART53 IS NOT NULL

Logical Combinations​

AND (Multiple Filters)​

All conditions must be true:

GET /api/v4/core/ART?source=settings-2&filter[XART06]=1&filter[XART01]=2
# WHERE XART06 = 1 AND XART01 = 2 (active products in category 2)

Multiple operators on same field:

GET /api/v4/core/ART?source=settings-2&filter[XART25][$gte]=50&filter[XART25][$lte]=100
# WHERE XART25 >= 50 AND XART25 <= 100 (price between 50-100)

OR (Same Field Multiple Values)​

Any condition must be true:

# Using $in operator (recommended)
GET /api/v4/core/ART?source=settings-2&filter[XART01][$in]=1,2,3
# WHERE XART01 IN (1,2,3)

# Using multiple filters on same field (implicit OR)
GET /api/v4/core/ART?source=settings-2&filter[XART01]=1&filter[XART01]=2&filter[XART01]=3
# WHERE XART01 = 1 OR XART01 = 2 OR XART01 = 3

Special TREC Filtering​

The cond_trec Parameter​

Problem: The platform automatically adds WHERE TREC NOT IN ('C') to all queries to exclude soft-deleted records. You cannot override this default behavior using standard filter[TREC] because it would be ANDed with the default condition.

Solution: Use the special cond_trec parameter to control TREC filtering behavior.

Default Behavior (No cond_trec)​

GET /api/v4/core/ART?source=settings-2
# Automatic SQL: WHERE TREC NOT IN ('C')
# Returns only active records (TREC = 'N' or 'M')

Explicit Default (cond_trec=C)​

GET /api/v4/core/ART?source=settings-2&cond_trec=C
# SQL: WHERE TREC NOT IN ('C')
# Same as default - excludes cancelled records

Custom TREC Exclusions​

Exclude multiple TREC states:

GET /api/v4/core/ART?source=settings-2&cond_trec=S,A
# SQL: WHERE TREC NOT IN ('S', 'A')
# Excludes records with TREC = 'S' or 'A'

Example Use Case:

# Exclude both cancelled (C) and processed (P) records
GET /api/v4/core/ORD?source=sales-1&cond_trec=C,P
# SQL: WHERE TREC NOT IN ('C', 'P')
# Returns only new (N) and modified (M) orders

Bypass TREC Filter (cond_trec=null)​

⚠️ Warning: Returns ALL records including soft-deleted:

GET /api/v4/core/ART?source=settings-2&cond_trec=null
# SQL: No TREC filter applied
# Returns all records: TREC = 'N', 'M', 'C', 'P'

When to use cond_trec=null:

  • Data migration/export (include deleted records)
  • Administrative auditing (view full history)
  • Restore deleted records workflow
  • Debugging data issues

⚠️ Caution:

  • Production UIs should NEVER use cond_trec=null by default
  • Soft-deleted records (TREC='C') are logically deleted
  • Only use when you explicitly need deleted records

Filter Examples​

Example 1: Active Products in Specific Category​

GET /api/v4/core/ART?source=settings-2&filter[XART06]=1&filter[XART01]=2
# WHERE XART06 = 1 AND XART01 = 2 AND TREC NOT IN ('C')

Returns:

  • Active products only (XART06 = 1)
  • In category 2 (XART01 = 2)
  • Not soft-deleted (TREC != 'C')

Example 2: Price Range with Category Filter​

GET /api/v4/core/ART?source=settings-2&filter[XART25][$gte]=50&filter[XART25][$lte]=100&filter[XART01]=3
# WHERE XART25 >= 50 AND XART25 <= 100 AND XART01 = 3 AND TREC NOT IN ('C')

Returns:

  • Price between 50-100 (XART25)
  • Category 3 (XART01)
  • Active records only

Example 3: Search Products by Name Pattern​

GET /api/v4/core/ART?source=settings-2&filter[XART20][$like]=%Widget%&filter[XART06]=1
# WHERE XART20 LIKE '%Widget%' AND XART06 = 1 AND TREC NOT IN ('C')

Returns:

  • Product name contains "Widget"
  • Active products only
  • Not soft-deleted

Example 4: Recently Created Records​

GET /api/v4/core/ART?source=settings-2&filter[CDATA][$gte]=20251201000000
# WHERE CDATA >= '20251201000000' AND TREC NOT IN ('C')

Returns:

  • Created on or after December 1, 2025
  • Active records only

Example 5: Multiple Categories (OR)​

GET /api/v4/core/ART?source=settings-2&filter[XART01][$in]=1,2,3
# WHERE XART01 IN (1, 2, 3) AND TREC NOT IN ('C')

Returns:

  • Products in category 1, 2, or 3
  • Active records only

Example 6: Include Deleted Records​

GET /api/v4/core/ART?source=settings-2&cond_trec=null&filter[XART01]=2
# WHERE XART01 = 2
# NO TREC filter - includes soft-deleted records

Returns:

  • All products in category 2
  • Including soft-deleted (TREC='C')

Case Sensitivity​

Case-Sensitive Filtering (Default)​

Exact match required:

GET /api/v4/core/ART?source=settings-2&filter[XART20]=Widget
# WHERE XART20 = 'Widget' (case-sensitive)
# Matches: "Widget"
# Does NOT match: "widget", "WIDGET", "WiDgEt"

Case-Insensitive Filtering​

Use LIKE with uppercase/lowercase:

GET /api/v4/core/ART?source=settings-2&filter[UPPER(XART20)][$like]=%WIDGET%
# WHERE UPPER(XART20) LIKE '%WIDGET%'
# Matches: "Widget", "widget", "WIDGET", "WiDgEt Pro"

Filtering by Related Dimensions​

Filter by Foreign Key Field​

Filter by AC/VARS field:

GET /api/v4/core/ART?source=settings-2&filter[XART53]=rep1
# WHERE XART53 = 'rep1' (department code)

Filter by foreign key value (joined table):

This requires using the Document API or $expand parameter (covered in separate sections).

Filter Validation​

Field Must Have R Flag​

Only fields with COD_ON_OFF containing 'R' can be filtered:

-- TB_COST example
COD_DIM: ART
COD_VAR: XART20
COD_ON_OFF: LDNMR  -- R flag present → can filter

COD_DIM: ART
COD_VAR: XART99
COD_ON_OFF: LD     -- No R flag → cannot filter

Invalid Filter:

GET /api/v4/core/ART?source=settings-2&filter[XART99]=value
# Error: Field XART99 does not have R flag - cannot be used in filters

Error Response​

{
  "status": 400,
  "message": "Field XART99 cannot be used in filters (missing R flag in COD_ON_OFF)",
  "code": "FIELD_NOT_FILTERABLE",
  "field": "XART99"
}

Performance Considerations​

Indexed Fields​

Filter on indexed fields for best performance:

-- Recommended indexes
CREATE INDEX idx_art_active ON TB_ANAG_ART00(XART06);
CREATE INDEX idx_art_category ON TB_ANAG_ART00(XART01);
CREATE INDEX idx_art_created ON TB_ANAG_ART00(CDATA);
CREATE INDEX idx_art_trec ON TB_ANAG_ART00(TREC);

Fast queries:

# ✅ Good - indexed field
GET /api/v4/core/ART?source=settings-2&filter[XART06]=1

# ⚠️ Slow - LIKE with leading wildcard (full table scan)
GET /api/v4/core/ART?source=settings-2&filter[XART20][$like]=%Widget

Limit Results​

Always paginate large result sets:

GET /api/v4/core/ART?source=settings-2&filter[XART01]=2&$num_rows=100&$offset=0

Best Practices​

✅ DO:​

Use specific operators:

# ✅ Good - explicit operator
GET /api/v4/core/ART?source=settings-2&filter[XART25][$gte]=50

Combine multiple filters:

# ✅ Good - narrow results
GET /api/v4/core/ART?source=settings-2&filter[XART06]=1&filter[XART01]=2

Use $in for multiple values:

# ✅ Good - single query parameter
GET /api/v4/core/ART?source=settings-2&filter[XART01][$in]=1,2,3

Filter on indexed fields:

# ✅ Good - fast query
GET /api/v4/core/ART?source=settings-2&filter[XART06]=1

❌ DON'T:​

Don't use LIKE with leading wildcard:

# ❌ Bad - slow full table scan
GET /api/v4/core/ART?source=settings-2&filter[XART20][$like]=%Widget

Don't filter on non-indexed text fields:

# ❌ Bad - slow query
GET /api/v4/core/ART?source=settings-2&filter[XART_DESCRIPTION][$like]=%long text%

Don't use cond_trec=null in production UIs:

# ❌ Bad - includes deleted records
GET /api/v4/core/ART?source=settings-2&cond_trec=null

Don't fetch all records without filters:

# ❌ Bad - returns everything
GET /api/v4/core/ART?source=settings-2

# ✅ Good - filtered and paginated
GET /api/v4/core/ART?source=settings-2&filter[XART06]=1&$num_rows=100

Summary​

  • ✅ Multiple filter operators: equals, $ne, $gt, $gte, $lt, $lte, $in, $nin, $like, $between, $isnull, $notnull
  • ✅ Logical combinations: AND (multiple filters), OR ($in or same field)
  • ✅ Special TREC filtering via cond_trec parameter
  • ✅ Default: TREC NOT IN ('C') excludes soft-deleted records
  • ✅ cond_trec=null bypasses TREC filter (returns deleted records)
  • ✅ Fields must have R flag in COD_ON_OFF to be filterable
  • ✅ Case-sensitive by default, use UPPER() for case-insensitive
  • ✅ Always paginate and filter on indexed fields for performance

Key Takeaways:

  1. Use filter operators to narrow query results
  2. Combine multiple filters with AND logic
  3. Use $in operator for OR logic on same field
  4. Default behavior excludes soft-deleted records (TREC='C')
  5. Use cond_trec parameter for custom TREC filtering
  6. Only fields with R flag can be filtered
  7. Filter on indexed fields for best performance
  8. Always paginate large result sets
