Error Codes

Overview​

Q01 Core APIs return structured error responses with HTTP status codes, error types, and actionable messages. This reference catalogs all error codes and provides troubleshooting guidance.

Error Response Format:

{
  "error": "ErrorType",
  "message": "Human-readable error description",
  "code": "ERROR_CODE",
  "status": 400,
  "field": "field_name",
  "details": {}
}

Nested Error Responses​

When a request passes through multiple microservices (e.g., Client → Backend → CoreService → CoreQuery), errors can be nested with each layer wrapping the previous error message.

Nested Error Structure:

{
  "code": 400,
  "message": {
    "code": 400,
    "message": {
      "status": 400,
      "message": "Invalid field: XPRD99 does not exist in dimension PRD",
      "code": "INVALID_FIELD",
      "errors": {}
    }
  }
}

Understanding Nested Errors:

Client
  ↓ HTTP 400
Backend (your application service)
  ↓ wraps error in {"code": 400, "message": {...}}
CoreService (API Gateway)
  ↓ wraps error in {"code": 400, "message": {...}}
CoreQuery (Data Service)
  ↓ generates original error: {"status": 400, "message": "Invalid field...", "code": "INVALID_FIELD"}

How to Extract Root Cause:

function getRootError(errorResponse) {
  let current = errorResponse;

  // Traverse nested messages until we find the root
  while (current.message && typeof current.message === 'object') {
    current = current.message;
  }

  return current;
}

// Usage
try {
  const response = await fetch('/api/v4/core/PRD');
  if (!response.ok) {
    const error = await response.json();
    const rootError = getRootError(error);
    console.error('Root cause:', rootError.message);
    console.error('Error code:', rootError.code);
  }
} catch (error) {
  console.error('Request failed:', error);
}

Example with Real Nested Error:

Response:

{
  "code": 400,
  "message": {
    "code": 400,
    "message": {
      "status": 400,
      "message": "Missing required field: XPRD05",
      "code": "REQUIRED_FIELD_MISSING",
      "field": "XPRD05"
    }
  }
}

Extracted Root Error:

{
  "status": 400,
  "message": "Missing required field: XPRD05",
  "code": "REQUIRED_FIELD_MISSING",
  "field": "XPRD05"
}

Best Practice: Always extract the innermost error message for the actual root cause. Outer wrappers are from intermediate routing layers and don't contain the detailed error information.

HTTP Status Codes​

2xx Success​

200 OK​

Meaning: Request succeeded

Used For:

  • GET requests (list, single record)
  • PATCH requests (partial update)
  • PUT requests (full update)

Example:

{
  "data": {
    "PRD_ID": 123,
    "XPRD01": "Product Name"
  }
}

201 Created​

Meaning: Resource created successfully

Used For:

  • POST requests (create record)

Example:

{
  "data": {
    "PRD_ID": 124,
    "TREC": "N",
    "XPRD01": "New Product"
  },
  "message": "Record created successfully"
}

205 Reset Content​

Meaning: Resource deleted successfully - client should reset view

Used For:

  • DELETE requests (soft delete or force delete)

Why 205 instead of 204?

  • 204 No Content = response has no body
  • 205 Reset Content = response has body AND client should refresh/reset view
  • Q01 DELETE operations return metadata about deleted record
  • 205 signals client to refresh list/table views

Example Response:

{
  "status": 205,
  "message": "Record deleted successfully",
  "data": {
    "PRD_ID": 123,
    "TREC": "C",
    "deleted_at": "2025-12-19T16:00:00Z"
  }
}

Client Behavior:

// Handle DELETE response
const response = await fetch(`/api/v4/core/PRD/123`, {
  method: 'DELETE'
});

if (response.status === 205) {
  // Reset/refresh client view
  refreshProductList();  // Reload table data
  clearSelection();      // Clear selected rows
  showNotification('Record deleted successfully');
}

Soft Delete (TREC='C'):

DELETE /api/v4/core/PRD/123?source=productManagement
# Sets TREC='C', returns 205

Force Delete (Physical Removal):

DELETE /api/v4/core/PRD/123?source=productManagement&force=true
# Removes record from database, returns 205

4xx Client Errors​

400 Bad Request​

Meaning: Invalid request syntax or malformed parameters

Common Causes:

  • Invalid JSON syntax
  • Malformed query parameters
  • Invalid filter operators
  • Missing required request headers

Example Error:

{
  "error": "BadRequest",
  "message": "Invalid JSON syntax in request body",
  "code": "INVALID_JSON",
  "status": 400
}

Resolution:

  • Validate JSON syntax
  • Check query parameter format
  • Review API documentation for correct syntax

401 Unauthorized​

Meaning: Missing or invalid authentication token

Common Causes:

  • No Authorization header
  • Expired JWT token
  • Invalid token signature
  • Token from wrong issuer

Error Codes:

TOKEN_MISSING​
{
  "error": "Unauthorized",
  "message": "Authorization header is required",
  "code": "TOKEN_MISSING",
  "status": 401
}

Resolution:

// Include Authorization header
headers: {
  'Authorization': `Bearer ${token}`
}
TOKEN_EXPIRED​
{
  "error": "Unauthorized",
  "message": "JWT token has expired",
  "code": "TOKEN_EXPIRED",
  "status": 401,
  "expired_at": "2025-12-19T15:30:45Z"
}

Resolution:

// Refresh token
const response = await fetch('/auth/refresh', {
  method: 'POST',
  body: JSON.stringify({ refresh_token })
});
const { access_token } = await response.json();
TOKEN_INVALID​
{
  "error": "Unauthorized",
  "message": "Invalid JWT token signature",
  "code": "TOKEN_INVALID",
  "status": 401
}

Resolution:

  • Obtain new token via /auth/login
  • Verify token is not corrupted
  • Check token issuer matches API

403 Forbidden​

Meaning: Valid authentication but insufficient permissions

Common Causes:

  • User lacks required grant (TB_MENU)
  • Source/centro_dett mismatch
  • Peso level insufficient
  • Field not visible due to COD_ON_OFF

Error Codes:

INSUFFICIENT_PERMISSIONS​
{
  "error": "Forbidden",
  "message": "User lacks required grant: products.write",
  "code": "INSUFFICIENT_PERMISSIONS",
  "status": 403,
  "required_grant": "products.write",
  "user_grants": ["products.read"]
}

Resolution:

// Check user grants before operation
if (!hasGrant('products.write')) {
  // Hide create/edit UI
  hideCreateButton();
}
CONTEXT_MISMATCH​
{
  "error": "Forbidden",
  "message": "Record belongs to different source",
  "code": "CONTEXT_MISMATCH",
  "status": 403,
  "record_source": "tenant_A",
  "user_source": "tenant_B"
}

Resolution:

  • Verify session context (source, centro_dett)
  • Ensure user is operating within their tenant
PESO_INSUFFICIENT​
{
  "error": "Forbidden",
  "message": "User peso level insufficient to access record",
  "code": "PESO_INSUFFICIENT",
  "status": 403,
  "record_peso": 1,
  "user_peso": 3
}

Resolution:

  • User with peso=3 cannot access records with peso=1 (admin level)
  • Filter records by user's peso level

404 Not Found​

Meaning: Requested resource does not exist

Common Causes:

  • Invalid dimension code
  • Record ID doesn't exist
  • Record soft-deleted (TREC='C')

Error Codes:

DIMENSION_NOT_FOUND​
{
  "error": "NotFound",
  "message": "Dimension 'XYZ' not found in TB_DIM",
  "code": "DIMENSION_NOT_FOUND",
  "status": 404,
  "dimension": "XYZ"
}

Resolution:

  • Verify dimension code exists in TB_DIM
  • Check for typos in dimension code
RECORD_NOT_FOUND​
{
  "error": "NotFound",
  "message": "Record with ID 999 not found",
  "code": "RECORD_NOT_FOUND",
  "status": 404,
  "dimension": "PRD",
  "record_id": 999
}

Resolution:

  • Verify record ID exists
  • Check if record was soft-deleted (TREC='C')
  • Query with ?filter[TREC][$in]=N,M,C to include deleted
FIELD_NOT_FOUND​
{
  "error": "NotFound",
  "message": "Field 'XPRD99' not found in dimension PRD",
  "code": "FIELD_NOT_FOUND",
  "status": 404,
  "dimension": "PRD",
  "field": "XPRD99"
}

Resolution:

  • Query /api/v4/core/PRD/fields for available fields
  • Check field name spelling

422 Unprocessable Entity​

Meaning: Request syntax valid but validation failed

Common Causes:

  • Missing required fields
  • Field not allowed in operation (COD_ON_OFF)
  • Invalid field value (data type mismatch)
  • Constraint violation (unique, foreign key)

Error Codes:

REQUIRED_FIELD_MISSING​
{
  "error": "ValidationError",
  "message": "Required field missing: XPRD05",
  "code": "REQUIRED_FIELD_MISSING",
  "status": 422,
  "field": "XPRD05"
}

Resolution:

// Include all required fields
const requiredFields = fields.filter(f => f.REQUIRED === true);
FIELD_NOT_ALLOWED_ON_CREATE​
{
  "error": "ValidationError",
  "message": "Field XPRD03 cannot be set on creation (COD_ON_OFF missing 'N' flag)",
  "code": "FIELD_NOT_ALLOWED_ON_CREATE",
  "status": 422,
  "field": "XPRD03",
  "cod_on_off": "LDR"
}

Resolution:

// Only include fields with 'N' flag on POST
const createFields = fields.filter(f => f.COD_ON_OFF.includes('N'));
FIELD_NOT_MODIFIABLE​
{
  "error": "ValidationError",
  "message": "Field XPRD03 cannot be modified (COD_ON_OFF missing 'M' flag)",
  "code": "FIELD_NOT_MODIFIABLE",
  "status": 422,
  "field": "XPRD03",
  "cod_on_off": "LDRN"
}

Resolution:

// Only include fields with 'M' flag on PUT/PATCH
const updateFields = fields.filter(f => f.COD_ON_OFF.includes('M'));
FIELD_NOT_SEARCHABLE​
{
  "error": "ValidationError",
  "message": "Field XPRD99 cannot be used in filters (COD_ON_OFF missing 'R' flag)",
  "code": "FIELD_NOT_SEARCHABLE",
  "status": 422,
  "field": "XPRD99",
  "cod_on_off": "DN"
}

Resolution:

// Only use fields with 'R' flag in filters
const searchableFields = fields.filter(f => f.COD_ON_OFF.includes('R'));
DATA_TYPE_MISMATCH​
{
  "error": "ValidationError",
  "message": "Field XPRD02 expects DECIMAL, received STRING",
  "code": "DATA_TYPE_MISMATCH",
  "status": 422,
  "field": "XPRD02",
  "expected_type": "DECIMAL",
  "received_type": "STRING",
  "received_value": "abc"
}

Resolution:

// Validate data types before sending
const price = parseFloat(priceInput);  // Convert to number
UNIQUE_CONSTRAINT_VIOLATION​
{
  "error": "ValidationError",
  "message": "Field XPRD03 must be unique, value 'SKU-001' already exists",
  "code": "UNIQUE_CONSTRAINT_VIOLATION",
  "status": 422,
  "field": "XPRD03",
  "value": "SKU-001",
  "existing_record_id": 456
}

Resolution:

  • Generate unique value
  • Check for existing value before creating
FOREIGN_KEY_VIOLATION​
{
  "error": "ValidationError",
  "message": "Foreign key violation: Category 'cat_999' does not exist",
  "code": "FOREIGN_KEY_VIOLATION",
  "status": 422,
  "field": "XPRD05",
  "value": "cat_999",
  "referenced_dimension": "CAT"
}

Resolution:

  • Verify referenced record exists
  • Create referenced record first

429 Too Many Requests​

Meaning: Rate limit exceeded

Error Response:

{
  "error": "RateLimitExceeded",
  "message": "Rate limit exceeded. Retry after 300 seconds.",
  "code": "RATE_LIMIT_EXCEEDED",
  "status": 429,
  "retry_after": 300,
  "limit": 1000,
  "remaining": 0,
  "reset": 1735574400
}

Resolution:

// Implement exponential backoff
async function retryWithBackoff(fn, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn();
    } catch (error) {
      if (error.status === 429) {
        const retryAfter = error.retry_after || Math.pow(2, i) * 1000;
        await sleep(retryAfter);
      } else {
        throw error;
      }
    }
  }
}

5xx Server Errors​

500 Internal Server Error​

Meaning: Unexpected server error

Common Causes:

  • Database connection failure
  • Unhandled exception
  • Configuration error

Error Response:

{
  "error": "InternalServerError",
  "message": "An unexpected error occurred",
  "code": "INTERNAL_ERROR",
  "status": 500,
  "request_id": "uuid-..."
}

Resolution:

  • Retry request (may be transient)
  • Check server logs with request_id
  • Contact technical support if persistent

502 Bad Gateway​

Meaning: CoreService cannot reach backend (CoreQuery/CoreWrite)

Common Causes:

  • Backend service down
  • Network issue
  • Backend timeout

Error Response:

{
  "error": "BadGateway",
  "message": "Backend service unavailable",
  "code": "BACKEND_UNAVAILABLE",
  "status": 502,
  "backend": "corewrite.internal"
}

Resolution:

  • Retry request after delay
  • Check service health status
  • Monitor for service recovery

503 Service Unavailable​

Meaning: Service temporarily unavailable

Common Causes:

  • Maintenance mode
  • Database maintenance
  • Server overloaded

Error Response:

{
  "error": "ServiceUnavailable",
  "message": "Service temporarily unavailable for maintenance",
  "code": "SERVICE_UNAVAILABLE",
  "status": 503,
  "retry_after": 600
}

Resolution:

  • Wait for maintenance window to complete
  • Retry after specified time
  • Check status page for updates

504 Gateway Timeout​

Meaning: Backend did not respond within timeout

Common Causes:

  • Slow database query
  • Backend processing timeout
  • Network congestion

Error Response:

{
  "error": "GatewayTimeout",
  "message": "Backend request timed out",
  "code": "GATEWAY_TIMEOUT",
  "status": 504,
  "timeout": 30000
}

Resolution:

  • Optimize query (reduce filters, limit results)
  • Use pagination for large datasets
  • Implement client-side timeout handling

Troubleshooting Guide​

Checklist for 401 Errors​

  1. Check Authorization header:
console.log(headers.Authorization); // "Bearer eyJ..."
  1. Verify token not expired:
const payload = JSON.parse(atob(token.split('.')[1]));
console.log(payload.exp > Date.now() / 1000); // Should be true
  1. Refresh token if expired:
if (error.code === 'TOKEN_EXPIRED') {
  const newToken = await refreshAccessToken();
}

Checklist for 403 Errors​

  1. Check user grants:
const hasGrant = user.grants.includes('products.write');
  1. Verify context:
console.log(session.source);       // Should match tenant
console.log(session.centro_dett);  // Should match org unit
console.log(session.peso);         // Should be sufficient
  1. Check COD_ON_OFF:
const canCreate = field.COD_ON_OFF.includes('N');
const canModify = field.COD_ON_OFF.includes('M');

Checklist for 422 Errors​

  1. Validate required fields:
const required = fields.filter(f => f.REQUIRED);
const missing = required.filter(f => !data[f.COD_VAR]);
  1. Check COD_ON_OFF flags:
// For POST
const allowedOnCreate = fields.filter(f => f.COD_ON_OFF.includes('N'));

// For PUT/PATCH
const allowedOnUpdate = fields.filter(f => f.COD_ON_OFF.includes('M'));
  1. Validate data types:
if (field.DATA_TYPE === 'DECIMAL') {
  value = parseFloat(value);
}

Checklist for 429 Errors​

  1. Check rate limit headers:
console.log(response.headers.get('X-RateLimit-Remaining'));
console.log(response.headers.get('X-RateLimit-Reset'));
  1. Implement retry logic:
if (response.status === 429) {
  const retryAfter = response.headers.get('Retry-After');
  await sleep(retryAfter * 1000);
  return retry();
}

Summary​

  • ✅ Structured error responses with error type, code, and message
  • ✅ HTTP status codes follow REST conventions
  • ✅ Application-specific codes for detailed error categorization
  • ✅ Actionable messages for quick troubleshooting
  • ✅ Field-level errors for validation failures

Most Common Errors:

  • 401 TOKEN_EXPIRED - Refresh token
  • 403 INSUFFICIENT_PERMISSIONS - Check grants
  • 404 RECORD_NOT_FOUND - Verify ID
  • 422 REQUIRED_FIELD_MISSING - Include all required fields
  • 422 FIELD_NOT_MODIFIABLE - Check COD_ON_OFF
  • 429 RATE_LIMIT_EXCEEDED - Implement retry logic

Error Handling Best Practices:

  1. Always include Authorization header
  2. Refresh tokens before expiration
  3. Check user grants before operations
  4. Validate COD_ON_OFF before field access
  5. Implement exponential backoff for rate limits
  6. Log request_id for server errors
