Security Checklist

Overview​

This checklist summarizes security best practices for Q01 Core APIs. Use this as a reference when implementing applications, reviewing code, or conducting security audits.

Authentication​

✅ Token Management​

  • Use HTTPS in production - Never send JWT tokens over unencrypted HTTP
  • Store tokens securely - Use httpOnly cookies (not localStorage)
  • Implement token refresh - Refresh access tokens before expiration
  • Handle token expiration - Gracefully handle 401 errors with refresh
  • Rotate JWT secrets - Change JWT_SECRET regularly (monthly)
  • Set appropriate TTL - Access tokens: 1 hour, Refresh tokens: 30 days
  • Include complete context - Ensure tokens have source/centro_dett/peso/ambiente
  • Clear tokens on logout - Remove all tokens from storage

❌ Common Mistakes​

// ❌ Bad - localStorage (XSS vulnerable)
localStorage.setItem('token', jwt);

// ✅ Good - httpOnly cookie
document.cookie = `token=${jwt}; HttpOnly; Secure; SameSite=Strict`;

// ❌ Bad - token in URL
fetch(`/api/products?token=${token}`);

// ✅ Good - Authorization header
fetch('/api/products', {
  headers: { 'Authorization': `Bearer ${token}` }
});

// ❌ Bad - logging tokens
console.log('Token:', token);

// ✅ Good - log without sensitive data
console.log('Token expires at:', payload.exp);

Authorization​

✅ Grant Management​

  • Check grants server-side - Never trust client-side grant checks alone
  • Use hierarchical grants - Leverage TB_MENU nested set for broad permissions
  • Hide unavailable actions - Check grants client-side for better UX
  • Request minimal grants - Apply principle of least privilege
  • Use roles - Group related grants into roles for easier management
  • Validate on every request - Don't cache grant decisions
  • Handle 403 gracefully - Show clear "access denied" messages

❌ Common Mistakes​

// ❌ Bad - client-side only
if (hasGrant('products.write')) {
  await createProduct(data);  // Server must still check!
}

// ✅ Good - client-side for UX, server validates
{hasGrant('products.write') && <CreateButton />}
await createProduct(data);  // Server validates grant

// ❌ Bad - hardcoded permissions
if (user.email === 'admin@example.com') {
  // Allow
}

// ✅ Good - grant-based
if (checkGrant(user.grants, 'products.write')) {
  // Allow
}

// ❌ Bad - exposing all grants in error
{
  "error": "Grant denied",
  "user_grants": ["secret.admin", "internal.grant"]
}

// ✅ Good - minimal info
{
  "error": "Insufficient permissions",
  "code": "GRANT_DENIED"
}

Tenant Isolation​

✅ Multi-Tenant Security​

  • Always set source during login - Explicit tenant identification
  • Never override source - Trust automatic filtering
  • Validate session context - Ensure source/centro_dett present
  • Test cross-tenant access - Verify isolation in tests
  • Use source for customization - Tenant-specific features
  • Monitor cross-tenant attempts - Alert on suspicious access patterns
  • Document tenant structure - Clear source naming conventions

❌ Common Mistakes​

// ❌ Bad - missing tenant
await login(username, password);

// ✅ Good - explicit tenant
await login(username, password, 'tenantA', 'admin');

// ❌ Bad - trying to override
await createProduct({
  XPRD01: 'Widget',
  PRD_SOURCE: 'otherTenant'  // Error!
});

// ✅ Good - let Core APIs set source
await createProduct({
  XPRD01: 'Widget'
  // PRD_SOURCE set automatically
});

// ❌ Bad - assuming cross-tenant access
const product = await getProduct(otherTenantId);  // 404

// ✅ Good - respect tenant boundaries
const product = await getProduct(myTenantId);

Context Security​

✅ Context Management​

  • Set complete context at login - source/centro_dett/peso/ambiente
  • Use peso appropriately - Admin=1, Manager=2, User=3
  • Separate environments - Never mix production and test
  • Respect hierarchical peso - Lower numbers see more
  • Match centro_dett exactly - No hierarchical inference
  • Trust automatic filtering - Core APIs handle all context filtering
  • Test context isolation - Verify peso/ambiente filtering works

❌ Common Mistakes​

// ❌ Bad - incomplete context
await login(username, password, source);

// ✅ Good - complete context
await login(username, password, {
  source: 'retailChain',
  centro_dett: 'store_001',
  peso: '2',
  ambiente: 'production'
});

// ❌ Bad - trying to override peso
await createProduct({
  XPRD01: 'Admin Product',
  PRD_PESO: '1'  // Error!
});

// ✅ Good - peso from session
await createProduct({
  XPRD01: 'Product'
  // PRD_PESO set from session
});

// ❌ Bad - mixing environments
if (ambiente === 'test') {
  // But still accessing production data?
}

// ✅ Good - separate sessions entirely
// Test session: ambiente='test'
// Production session: ambiente='production'

Rate Limiting​

✅ API Usage​

  • Monitor rate limit headers - Check X-RateLimit-Remaining
  • Implement exponential backoff - Retry with increasing delays
  • Throttle batch operations - Limit concurrent requests (10-20 max)
  • Cache responses - Reduce redundant API calls
  • Handle 429 gracefully - Wait for Retry-After
  • Use appropriate limits - Different limits for reads vs writes
  • Monitor API usage - Track rate limit consumption

❌ Common Mistakes​

// ❌ Bad - ignoring rate limits
await fetch(url);  // No rate limit awareness

// ✅ Good - monitor and respect
const remaining = response.headers.get('X-RateLimit-Remaining');
if (remaining < 50) {
  console.warn('Low rate limit');
}

// ❌ Bad - immediate retry on 429
if (response.status === 429) {
  return fetch(url);  // No wait!
}

// ✅ Good - exponential backoff
if (response.status === 429) {
  const retryAfter = response.headers.get('Retry-After');
  await sleep(retryAfter * 1000);
  return fetch(url);
}

// ❌ Bad - unbounded parallel requests
await Promise.all(
  largeArray.map(item => createItem(item))
);

// ✅ Good - throttled
const throttler = new RequestThrottler(10);
await Promise.all(
  largeArray.map(item =>
    throttler.throttle(() => createItem(item))
  )
);

Audit Trail​

✅ Compliance​

  • Trust automatic audit fields - Don't try to override
  • Use soft delete - Preserve audit trail
  • Query OUTBOX for history - Complete operation log
  • Export audit data - GDPR data access requests
  • Implement data retention - Cleanup old audit records
  • Monitor suspicious activity - Alert on unusual patterns
  • Document audit procedures - Compliance requirements

❌ Common Mistakes​

// ❌ Bad - overriding audit fields
await createProduct({
  XPRD01: 'Widget',
  CREATED_BY: 'other@example.com'  // Error!
});

// ✅ Good - automatic audit
await createProduct({
  XPRD01: 'Widget'
  // CREATED_BY set automatically
});

// ❌ Bad - force delete (no audit)
DELETE /api/v4/core/PRD/123?force=true

// ✅ Good - soft delete (preserves audit)
DELETE /api/v4/core/PRD/123

// ❌ Bad - modifying OUTBOX
UPDATE OUTBOX SET USER = 'anonymous';

// ✅ Good - OUTBOX is append-only
// Query for audit history, don't modify

Input Validation​

✅ Data Safety​

  • Validate all inputs - Never trust client data
  • Sanitize user input - Prevent XSS attacks
  • Use parameterized queries - Prevent SQL injection (Core APIs handle this)
  • Enforce required fields - Check before API call
  • Validate data types - Numbers, dates, emails
  • Limit string lengths - Prevent buffer overflow
  • Check field permissions - Respect COD_ON_OFF flags

❌ Common Mistakes​

// ❌ Bad - no validation
const data = req.body;
await createProduct(data);

// ✅ Good - validate first
const validated = validateProduct(req.body);
if (validated.errors) {
  return res.status(400).json(validated.errors);
}
await createProduct(validated.data);

// ❌ Bad - storing sensitive data
await createUser({
  username: 'user@example.com',
  password: 'plaintext'  // Never!
});

// ✅ Good - hash passwords
const hashedPassword = await bcrypt.hash(password, 10);
await createUser({
  username: 'user@example.com',
  password_hash: hashedPassword
});

// ❌ Bad - trusting client-side validation only
// Client: <input required />
await createProduct(data);  // Server must also validate!

// ✅ Good - server-side validation
// Client: <input required />
// Server: if (!data.XPRD01) throw ValidationError

Error Handling​

✅ Secure Errors​

  • Hide implementation details - No stack traces in production
  • Use error codes - Machine-readable error handling
  • Log errors securely - Don't log sensitive data
  • Return appropriate status codes - 401, 403, 404, 500
  • Handle errors gracefully - User-friendly messages
  • Don't leak information - 404 for cross-tenant access (not 403)
  • Monitor error rates - Alert on unusual patterns

❌ Common Mistakes​

// ❌ Bad - exposing stack trace
catch (error) {
  res.status(500).json({ error: error.stack });
}

// ✅ Good - generic error message
catch (error) {
  console.error('Error:', error);  // Log server-side
  res.status(500).json({
    error: 'Internal server error',
    code: 'INTERNAL_ERROR'
  });
}

// ❌ Bad - revealing existence
if (!canAccess(record)) {
  return 403;  // Reveals record exists
}

// ✅ Good - uniform response
if (!canAccess(record)) {
  return 404;  // Same as "not found"
}

// ❌ Bad - logging sensitive data
console.log('User credentials:', { username, password });

// ✅ Good - log without secrets
console.log('Login attempt:', { username, success: true });

HTTPS and Transport Security​

✅ Encryption​

  • Always use HTTPS in production - No exceptions
  • Enforce TLS 1.2+ - Disable older protocols
  • Use strong cipher suites - Modern encryption algorithms
  • Implement HSTS - Force HTTPS
  • Validate certificates - Proper certificate chain
  • Use secure cookies - Secure, HttpOnly, SameSite flags

❌ Common Mistakes​

// ❌ Bad - HTTP in production
const apiBase = 'http://api.example.com';

// ✅ Good - HTTPS only
const apiBase = 'https://api.example.com';

// ❌ Bad - insecure cookie
document.cookie = `token=${jwt}`;

// ✅ Good - secure cookie
document.cookie = `token=${jwt}; Secure; HttpOnly; SameSite=Strict`;

// ❌ Bad - ignoring certificate errors
process.env.NODE_TLS_REJECT_UNAUTHORIZED = '0';

// ✅ Good - validate certificates
// Use proper certificate chain

Security Testing​

✅ Testing Checklist​

  • Test authentication flow - Token generation, refresh, expiration
  • Test authorization - Grant checking, 403 responses
  • Test tenant isolation - Cross-tenant access attempts
  • Test context filtering - Peso, ambiente, centro_dett
  • Test rate limiting - 429 responses, backoff
  • Test audit trail - Verify CREATED_BY, UPDATED_BY
  • Test input validation - Invalid data rejection
  • Penetration testing - Professional security audit
  • Test error handling - No sensitive data leaked
  • Test HTTPS - Certificate validation

Security Test Examples​

// Test tenant isolation
test('cannot access other tenant data', async () => {
  // Login as tenant A
  await login('userA@example.com', 'password', 'tenantA', 'admin');

  // Try to access tenant B's product
  const response = await getProduct(tenantBProductId);

  expect(response.status).toBe(404);  // Not 403!
});

// Test grant enforcement
test('cannot create without grant', async () => {
  // Login with read-only grant
  await login('user@example.com', 'password', 'tenantA', 'admin');

  // Try to create product
  const response = await createProduct({ XPRD01: 'Widget' });

  expect(response.status).toBe(403);
  expect(response.body.code).toBe('GRANT_DENIED');
});

// Test rate limiting
test('rate limit enforced', async () => {
  const requests = Array(1001).fill().map(() =>
    getProducts()
  );

  const responses = await Promise.all(requests);
  const rateLimited = responses.filter(r => r.status === 429);

  expect(rateLimited.length).toBeGreaterThan(0);
});

Deployment Checklist​

✅ Production Readiness​

  • Change default secrets - JWT_SECRET, database passwords
  • Enable HTTPS - Valid SSL certificates
  • Configure CORS - Restrict allowed origins
  • Set secure headers - HSTS, CSP, X-Frame-Options
  • Enable rate limiting - Appropriate limits configured
  • Configure audit logging - Centralized log collection
  • Set up monitoring - Security alerts and dashboards
  • Document security procedures - Incident response plan
  • Regular security updates - Patch dependencies
  • Backup audit logs - Immutable audit trail

Summary​

Critical Security Requirements:

  1. ✅ HTTPS in production
  2. ✅ Secure token storage (httpOnly cookies)
  3. ✅ Server-side grant validation
  4. ✅ Complete tenant isolation (source filtering)
  5. ✅ Context-based security (peso/ambiente/centro_dett)
  6. ✅ Rate limiting with backoff
  7. ✅ Automatic audit trail
  8. ✅ Input validation
  9. ✅ Secure error handling
  10. ✅ Regular security testing

Top Security Mistakes to Avoid:

  1. ❌ Using HTTP in production
  2. ❌ Storing tokens in localStorage
  3. ❌ Client-side only validation/authorization
  4. ❌ Missing tenant context (source)
  5. ❌ Ignoring rate limits
  6. ❌ Force deleting (no audit trail)
  7. ❌ Exposing sensitive data in errors
  8. ❌ No input validation
  9. ❌ Mixing environments (test/production)
  10. ❌ Hardcoded credentials or permissions

Security Review Questions:

  • Are all API calls over HTTPS?
  • Are tokens stored securely?
  • Is server-side validation implemented?
  • Is tenant isolation verified?
  • Are rate limits respected?
  • Is audit trail complete?
  • Is input sanitized?
  • Are errors handled securely?
  • Is security testing automated?
  • Are security procedures documented?
