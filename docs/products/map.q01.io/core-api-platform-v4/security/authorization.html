Authorization

Overview​

Q01 Core APIs use TB_MENU nested set model for hierarchical permissions. After authentication validates who you are, authorization determines what you can do. Every operation checks if the user has the required grant.

Authorization Flow:

API Request (authenticated)
    ↓
1. Extract grants from JWT token
2. Determine required grant for operation
3. Query TB_MENU nested set
4. Check if user has grant (exact or parent)
5. Grant found → proceed
6. Grant missing → 403 Forbidden

Benefits:

  • ✅ Hierarchical permissions (parent includes children)
  • ✅ Efficient grant checking (nested set queries)
  • ✅ Fine-grained or broad permissions
  • ✅ Centralized permission management
  • ✅ Role-based access control

TB_MENU Table​

Schema​

Nested set model uses NLEFT/NRIGHT for hierarchy:

CREATE TABLE TB_MENU (
    MENU_ID VARCHAR(100) PRIMARY KEY,    -- Grant identifier
    NLEFT INT NOT NULL,                   -- Left boundary
    NRIGHT INT NOT NULL,                  -- Right boundary
    DESCRIZIONE VARCHAR(255),             -- Description
    MENU_TYPE VARCHAR(10),                -- Type (module, operation)
    PARENT_ID VARCHAR(100),               -- Parent grant
    CREATED_AT TIMESTAMP,
    UPDATED_AT TIMESTAMP
);

CREATE INDEX idx_menu_nleft_nright ON TB_MENU(NLEFT, NRIGHT);

Grant Checking​

Direct Grant Check​

Check if user has specific grant:

-- User has grant 'products.read'?
SELECT COUNT(*) FROM TB_MENU
WHERE MENU_ID = 'products.read'
  AND MENU_ID IN (user_grants);

Hierarchical Grant Check​

Check if user has grant or parent grant:

-- User has 'products.create' or any parent grant?
SELECT COUNT(*) > 0 as HAS_GRANT
FROM TB_MENU parent
INNER JOIN TB_MENU child
    ON child.NLEFT BETWEEN parent.NLEFT AND parent.NRIGHT
WHERE parent.MENU_ID IN (user_grants)
  AND child.MENU_ID = 'products.create';

Example:

  • User has grant: products
  • Required grant: products.create
  • Check: Is products.create (NLEFT=5, NRIGHT=6) within products (NLEFT=1, NRIGHT=10)?
  • Result: ✅ Yes (5 between 1 and 10)

CoreQuery Grant Validation​

RequestValidatorSubscriber.php:

public function checkGrant(array $userGrants, string $requiredGrant): bool {
    // 1. Get required grant from TB_MENU
    $childNode = $this->pdo->query(
        "SELECT NLEFT, NRIGHT FROM TB_MENU WHERE MENU_ID = :grant",
        ['grant' => $requiredGrant]
    )->fetch();

    if (!$childNode) {
        // Grant doesn't exist in system
        return false;
    }

    // 2. Check if user has grant or parent grant
    $hasGrant = $this->pdo->query(
        "SELECT COUNT(*) as count FROM TB_MENU parent
         INNER JOIN TB_MENU child
             ON child.NLEFT BETWEEN parent.NLEFT AND parent.NRIGHT
         WHERE parent.MENU_ID IN (:user_grants)
           AND child.NLEFT = :child_left
           AND child.NRIGHT = :child_right",
        [
            'user_grants' => $userGrants,
            'child_left' => $childNode['NLEFT'],
            'child_right' => $childNode['NRIGHT']
        ]
    )->fetch();

    return $hasGrant['count'] > 0;
}

Grant Hierarchy​

Parent Grants Include Children​

Grant Inheritance:

User has: products
    ↓
User can:
├── products.read ✅
├── products.write ✅
│   └── products.create ✅
└── products.delete ✅

SQL Query:

-- Find all grants user has (including children)
SELECT child.MENU_ID
FROM TB_MENU parent
INNER JOIN TB_MENU child
    ON child.NLEFT BETWEEN parent.NLEFT AND parent.NRIGHT
WHERE parent.MENU_ID IN ('products');

-- Result:
-- products
-- products.read
-- products.write
-- products.create
-- products.delete

Fine-Grained Grants​

User has specific grant only:

User has: products.read
    ↓
User can:
└── products.read ✅

User cannot:
├── products.write ❌
├── products.create ❌
└── products.delete ❌

Grant Determination Logic​

CoreQuery determines required grant from dimension:

public function getRequiredGrant(string $dimension, string $operation): string {
    $dimensionLower = strtolower($dimension);

    switch ($operation) {
        case 'GET':
            return "{$dimensionLower}.read";

        case 'POST':
            return "{$dimensionLower}.create";

        case 'PUT':
        case 'PATCH':
            return "{$dimensionLower}.write";

        case 'DELETE':
            return "{$dimensionLower}.delete";

        default:
            throw new \Exception("Unknown operation: $operation");
    }
}

JWT Token Grants​

Token Structure​

Grants stored in JWT payload:

{
  "user_id": "user@example.com",
  "grants": [
    "products.read",
    "products.write",
    "orders.read",
    "customers.read",
    "customers.write"
  ],
  "exp": 1735574400
}

Grant Assignment​

Grants assigned during login:

// Auth service generates JWT
public function generateToken(User $user): string {
    // Get user's grants from database
    $grants = $this->getUserGrants($user->id);

    $payload = [
        'user_id' => $user->email,
        'tenant_id' => $user->tenant_id,
        'source' => $user->source,
        'centro_dett' => $user->centro_dett,
        'peso' => $user->peso,
        'ambiente' => $user->ambiente,
        'grants' => $grants,  // Array of grant IDs
        'exp' => time() + 3600,
        'iat' => time()
    ];

    return $this->jwt->encode($payload);
}

private function getUserGrants(string $userId): array {
    return $this->pdo->query(
        "SELECT MENU_ID FROM USER_GRANTS WHERE USER_ID = :user_id",
        ['user_id' => $userId]
    )->fetchAll(PDO::FETCH_COLUMN);
}

Authorization Errors​

403 Forbidden​

Response when grant check fails:

{
  "error": "AuthorizationError",
  "message": "Insufficient permissions for this operation",
  "code": "GRANT_DENIED",
  "status": 403,
  "required_grant": "products.write",
  "user_grants": ["products.read"],
  "user": "user@example.com"
}

Client-Side Grant Checking​

Hiding UI Elements​

Check grants before showing actions:

class GrantChecker {
  constructor(grants) {
    this.grants = grants;  // From JWT token
  }

  hasGrant(requiredGrant) {
    // Client-side approximation (not authoritative)
    // Server ALWAYS checks grants

    return this.grants.some(grant => {
      // Exact match
      if (grant === requiredGrant) {
        return true;
      }

      // Parent match (simple check)
      // e.g., 'products' includes 'products.read'
      if (requiredGrant.startsWith(grant + '.')) {
        return true;
      }

      return false;
    });
  }

  canRead(dimension) {
    return this.hasGrant(`${dimension}.read`);
  }

  canWrite(dimension) {
    return this.hasGrant(`${dimension}.write`) ||
           this.hasGrant(`${dimension}.create`);
  }

  canDelete(dimension) {
    return this.hasGrant(`${dimension}.delete`);
  }
}

// Usage in React
function ProductList({ grants }) {
  const grantChecker = new GrantChecker(grants);

  return (
    <div>
      <h1>Products</h1>

      {/* Show create button only if user can write */}
      {grantChecker.canWrite('products') && (
        <button onClick={handleCreate}>Create Product</button>
      )}

      <table>
        {products.map(product => (
          <tr key={product.PRD_ID}>
            <td>{product.XPRD01}</td>

            {/* Show delete button only if user can delete */}
            {grantChecker.canDelete('products') && (
              <td>
                <button onClick={() => handleDelete(product.PRD_ID)}>
                  Delete
                </button>
              </td>
            )}
          </tr>
        ))}
      </table>
    </div>
  );
}

Decoding JWT Client-Side​

class AuthService {
  decodeToken(token) {
    const parts = token.split('.');
    const payload = JSON.parse(atob(parts[1]));
    return payload;
  }

  getUserGrants() {
    const token = this.getAccessToken();
    if (!token) return [];

    const payload = this.decodeToken(token);
    return payload.grants || [];
  }
}

// Usage
const authService = new AuthService();
const grants = authService.getUserGrants();
const grantChecker = new GrantChecker(grants);

if (grantChecker.canWrite('products')) {
  showCreateButton();
}

Role-Based Access Control​

Roles as Grant Bundles​

Define roles as collections of grants:

-- Admin role (full access)
INSERT INTO ROLE_GRANTS (ROLE_ID, MENU_ID) VALUES
('admin', 'products'),
('admin', 'orders'),
('admin', 'customers');

-- Manager role (read/write, no delete)
INSERT INTO ROLE_GRANTS (ROLE_ID, MENU_ID) VALUES
('manager', 'products.read'),
('manager', 'products.write'),
('manager', 'orders.read'),
('manager', 'orders.write');

-- User role (read only)
INSERT INTO ROLE_GRANTS (ROLE_ID, MENU_ID) VALUES
('user', 'products.read'),
('user', 'orders.read');

Assign Roles to Users​

-- Assign role to user
INSERT INTO USER_ROLES (USER_ID, ROLE_ID)
VALUES ('user@example.com', 'manager');

-- Get user's grants via roles
SELECT DISTINCT rg.MENU_ID
FROM USER_ROLES ur
INNER JOIN ROLE_GRANTS rg ON ur.ROLE_ID = rg.ROLE_ID
WHERE ur.USER_ID = 'user@example.com';

Best Practices​

✅ DO:​

Use hierarchical grants:

-- ✅ Good - parent includes children
User has: products
User can do: products.read, products.write, products.delete

Check grants server-side:

// ✅ Good - server always validates
public function beforeQuery(string $dimension): void {
    $requiredGrant = "{$dimension}.read";
    if (!$this->checkGrant($this->userGrants, $requiredGrant)) {
        throw new AuthorizationException("Grant denied: $requiredGrant");
    }
}

Hide UI elements based on grants:

// ✅ Good - better UX
{hasGrant('products.write') && <CreateButton />}

Use roles for grant bundles:

-- ✅ Good - easier management
INSERT INTO USER_ROLES (USER_ID, ROLE_ID)
VALUES ('user@example.com', 'manager');

❌ DON'T:​

Don't skip server-side validation:

// ❌ Bad - client check only
if (hasGrant('products.write')) {
  await createProduct(data);  // Server MUST still check!
}

Don't hardcode grants in code:

// ❌ Bad - inflexible
if ($user->email === 'admin@example.com') {
    // Allow
}

// ✅ Good - grant-based
if ($this->checkGrant($user->grants, 'products.write')) {
    // Allow
}

Don't expose all grants in error:

// ❌ Bad - security leak
{
  "error": "Grant denied",
  "user_grants": ["secret.grant", "internal.admin"]
}

// ✅ Good - minimal info
{
  "error": "Insufficient permissions",
  "code": "GRANT_DENIED",
  "status": 403
}

Summary​

  • ✅ TB_MENU nested set model for hierarchical permissions
  • ✅ NLEFT/NRIGHT for efficient grant checking
  • ✅ Parent grants include all children
  • ✅ Grants stored in JWT token
  • ✅ Server validates every request
  • ✅ 403 Forbidden when grant missing
  • ✅ Client-side checking for better UX
  • ✅ Roles bundle multiple grants

Key Takeaways:

  1. Authorization checks what you can do after authentication
  2. TB_MENU nested set enables hierarchical permissions
  3. Parent grant includes all children (e.g., products → products.read)
  4. JWT token carries user's grants
  5. Server validates grants for every operation
  6. Client-side grant checking hides unavailable actions
  7. Roles simplify grant management

products (broad)
├── products.read (specific)
├── products.write (specific)
│   └── products.create (very specific)
└── products.delete (specific)

User with "products" grant can do EVERYTHING.
User with "products.read" grant can ONLY read.
