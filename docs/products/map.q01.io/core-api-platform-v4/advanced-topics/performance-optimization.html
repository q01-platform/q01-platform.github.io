Performance Optimization

Overview​

Q01 Core APIs performance depends on query optimization, proper indexing, connection management, and caching strategies. This guide covers techniques to maximize throughput and minimize latency.

Key Areas:

  1. Query Optimization - Efficient SQL patterns
  2. Database Indexing - Strategic index design
  3. Connection Pooling - Resource management
  4. N+1 Query Prevention - Avoiding query explosions
  5. Pagination - Efficient large dataset handling
  6. Monitoring - Performance metrics and profiling

Benefits:

  • ✅ Faster response times (reduced latency)
  • ✅ Higher throughput (more requests/second)
  • ✅ Lower database load (reduced CPU/IO)
  • ✅ Better user experience
  • ✅ Cost savings (fewer resources needed)

Query Optimization​

Use EXPLAIN to Analyze Queries​

Identify slow queries:

-- MySQL
EXPLAIN SELECT * FROM TB_ANAG_PRD00
WHERE XPRD05 = 'electronics'
AND XPRD06 = 'Y'
LIMIT 10;

-- Look for:
-- - type: ALL (bad - full table scan)
-- - type: ref/index (good - using index)
-- - Extra: Using filesort (bad - sorting without index)
-- - Extra: Using temporary (bad - temporary table)

Example output:

+----+-------------+----------------+------+---------------+------+---------+------+-------+-------------+
| id | select_type | table          | type | possible_keys | key  | key_len | ref  | rows  | Extra       |
+----+-------------+----------------+------+---------------+------+---------+------+-------+-------------+
|  1 | SIMPLE      | TB_ANAG_PRD00  | ALL  | NULL          | NULL | NULL    | NULL | 50000 | Using where |
+----+-------------+----------------+------+---------------+------+---------+------+-------+-------------+

❌ Problem: Full table scan (type=ALL) scanning 50,000 rows

✅ Solution: Add composite index on (XPRD05, XPRD06)

Select Only Required Fields​

# ❌ Bad - fetches all 50 fields
curl "https://api.q01.io/api/v4/core/products?limit=100"

# ✅ Good - fetches only 5 fields
curl "https://api.q01.io/api/v4/core/products?limit=100&fields=PRD_ID,XPRD01,XPRD02,XPRD03,XPRD05"

Performance impact:

  • Reduced network transfer (5x smaller payload)
  • Faster serialization (less data to encode)
  • Lower memory usage (smaller result set)

Use Indexed Fields in WHERE Clauses​

-- ❌ Bad - function on indexed column
SELECT * FROM TB_ANAG_PRD00
WHERE UPPER(XPRD03) = 'ABC123';

-- ✅ Good - direct comparison
SELECT * FROM TB_ANAG_PRD00
WHERE XPRD03 = 'ABC123';

-- ❌ Bad - leading wildcard
SELECT * FROM TB_ANAG_PRD00
WHERE XPRD01 LIKE '%widget%';

-- ✅ Good - prefix search
SELECT * FROM TB_ANAG_PRD00
WHERE XPRD01 LIKE 'widget%';

Avoid SELECT DISTINCT When Possible​

-- ❌ Bad - DISTINCT forces sorting
SELECT DISTINCT XPRD05 FROM TB_ANAG_PRD00;

-- ✅ Good - GROUP BY with index
SELECT XPRD05 FROM TB_ANAG_PRD00
GROUP BY XPRD05;

-- ✅ Better - use dimension relationships
SELECT CAT_ID, XCAT01 FROM TB_ANAG_CAT00;

Database Indexing​

Strategic Index Design​

Single-column indexes:

-- High cardinality columns (many unique values)
CREATE INDEX idx_prd_code ON TB_ANAG_PRD00(XPRD03);
CREATE INDEX idx_prd_created ON TB_ANAG_PRD00(CREATED_AT);

Composite indexes:

-- Most selective column first
CREATE INDEX idx_prd_category_active
ON TB_ANAG_PRD00(XPRD05, XPRD06);

-- Order matters: (category, active) ≠ (active, category)
-- Good for: WHERE XPRD05 = 'electronics'
-- Good for: WHERE XPRD05 = 'electronics' AND XPRD06 = 'Y'
-- Bad for:  WHERE XPRD06 = 'Y' (doesn't use index)

Covering indexes:

-- Include frequently accessed columns
CREATE INDEX idx_prd_category_active_name
ON TB_ANAG_PRD00(XPRD05, XPRD06, XPRD01);

-- Query can be satisfied entirely from index (no table lookup)
SELECT XPRD01 FROM TB_ANAG_PRD00
WHERE XPRD05 = 'electronics' AND XPRD06 = 'Y';

Index Best Practices​

✅ DO index:

  • Primary keys (automatic in most databases)
  • Foreign keys for JOIN operations
  • Fields used in WHERE clauses
  • Fields used in ORDER BY
  • Fields used in GROUP BY
  • High cardinality columns

❌ DON'T index:

  • Low cardinality columns (e.g., TREC with only 4 values)
  • Frequently updated columns (index maintenance overhead)
  • Small tables (<1000 rows)
  • Columns with many NULL values

Monitor Index Usage​

-- MySQL: Show unused indexes
SELECT
  t.TABLE_NAME,
  s.INDEX_NAME,
  s.CARDINALITY
FROM information_schema.TABLES t
LEFT JOIN information_schema.STATISTICS s
  ON t.TABLE_NAME = s.TABLE_NAME
WHERE t.TABLE_SCHEMA = 'q01_data'
  AND s.INDEX_NAME NOT IN ('PRIMARY')
ORDER BY s.CARDINALITY DESC;

-- Check index usage statistics
SHOW INDEX FROM TB_ANAG_PRD00;

Pagination Best Practices​

Keyset Pagination (Recommended)​

# ✅ Good - cursor-based pagination
# First page
curl "https://api.q01.io/api/v4/core/products?limit=50&order_by=PRD_ID"

# Response includes last ID: {"last_id": 1050, "data": [...]}

# Next page
curl "https://api.q01.io/api/v4/core/products?limit=50&order_by=PRD_ID&after=1050"

SQL implementation:

-- First page
SELECT * FROM TB_ANAG_PRD00
WHERE TREC = 'N'
ORDER BY PRD_ID ASC
LIMIT 50;

-- Next page
SELECT * FROM TB_ANAG_PRD00
WHERE TREC = 'N'
AND PRD_ID > 1050
ORDER BY PRD_ID ASC
LIMIT 50;

Benefits:

  • ✅ Consistent performance regardless of offset
  • ✅ No skipped/duplicate records
  • ✅ Works with real-time data changes

Avoid Large Offsets​

-- ❌ Bad - scans and discards 1M rows
SELECT * FROM TB_ANAG_PRD00
ORDER BY PRD_ID
LIMIT 50 OFFSET 1000000;

-- ✅ Good - uses keyset pagination
SELECT * FROM TB_ANAG_PRD00
WHERE PRD_ID > 1000050
ORDER BY PRD_ID
LIMIT 50;

Performance comparison:

  • OFFSET 0: ~10ms
  • OFFSET 10000: ~100ms
  • OFFSET 100000: ~1000ms
  • OFFSET 1000000: ~10000ms (10 seconds!)
  • Keyset: ~10ms (constant time)

N+1 Query Prevention​

Problem: Query Explosion​

// ❌ Bad - N+1 queries
async function getProductsWithCategories() {
  // 1 query
  const products = await api.get('/api/v4/core/products?limit=100');

  // 100 queries!
  for (const product of products.data) {
    product.category = await api.get(`/api/v4/core/categories/${product.XPRD05}`);
  }

  return products;
}
// Total: 101 queries (1 + 100)

Solution: Batch Requests​

// ✅ Good - 2 queries total
async function getProductsWithCategories() {
  // 1 query
  const products = await api.get('/api/v4/core/products?limit=100');

  // 1 query with IN clause
  const categoryIds = [...new Set(products.data.map(p => p.XPRD05))];
  const categories = await api.get(
    `/api/v4/core/categories?filters=CAT_ID:in:${categoryIds.join(',')}`
  );

  // Map in memory
  const categoryMap = Object.fromEntries(
    categories.data.map(c => [c.CAT_ID, c])
  );

  products.data.forEach(p => {
    p.category = categoryMap[p.XPRD05];
  });

  return products;
}
// Total: 2 queries (1 + 1)

Use Field Expansion​

# ✅ Best - single query with expansion
curl "https://api.q01.io/api/v4/core/products?limit=100&expand=XPRD05"

# Response includes expanded category data
{
  "data": [
    {
      "PRD_ID": 123,
      "XPRD01": "Widget",
      "XPRD05": 5,
      "_expanded": {
        "XPRD05": {
          "CAT_ID": 5,
          "XCAT01": "Electronics",
          "XCAT02": "Electronic goods"
        }
      }
    }
  ]
}

Connection Pooling​

Configure Connection Pools​

PHP (PDO):

class DatabasePool {
    private static $pool = [];
    private const MAX_CONNECTIONS = 20;
    private const MIN_CONNECTIONS = 5;

    public static function getConnection(): PDO {
        // Reuse idle connection
        if (!empty(self::$pool)) {
            return array_pop(self::$pool);
        }

        // Create new connection
        return new PDO(
            "mysql:host=db;dbname=q01_data",
            "user",
            "pass",
            [
                PDO::ATTR_PERSISTENT => true,
                PDO::ATTR_TIMEOUT => 5,
                PDO::MYSQL_ATTR_INIT_COMMAND => "SET NAMES utf8mb4"
            ]
        );
    }

    public static function releaseConnection(PDO $conn): void {
        if (count(self::$pool) < self::MAX_CONNECTIONS) {
            self::$pool[] = $conn;
        }
    }
}

Node.js (mysql2):

const mysql = require('mysql2/promise');

const pool = mysql.createPool({
  host: 'db',
  user: 'user',
  password: 'pass',
  database: 'q01_data',
  waitForConnections: true,
  connectionLimit: 20,      // Max connections
  maxIdle: 10,              // Max idle connections
  idleTimeout: 60000,       // Close idle after 60s
  queueLimit: 0,            // Unlimited queue
  enableKeepAlive: true,
  keepAliveInitialDelay: 0
});

// Usage
async function query(sql, params) {
  const connection = await pool.getConnection();
  try {
    const [rows] = await connection.query(sql, params);
    return rows;
  } finally {
    connection.release();  // Return to pool
  }
}

Connection Pool Sizing​

Formula: connections = (core_count * 2) + effective_spindle_count

Example:

  • 4 CPU cores
  • 1 SSD (effectively unlimited spindles)
  • Recommended: (4 * 2) + 1 = 9 connections

Tuning:

  • Too few: Connection queueing, slow response
  • Too many: Memory overhead, context switching
  • Monitor: Connection usage, wait times, query queue

Query Result Caching​

Application-Level Caching​

const Redis = require('ioredis');
const redis = new Redis();

class CachedQuery {
  async query(sql, params, ttl = 300) {
    const cacheKey = `query:${this.hash(sql, params)}`;

    // Check cache
    const cached = await redis.get(cacheKey);
    if (cached) {
      console.log('Cache hit');
      return JSON.parse(cached);
    }

    // Execute query
    console.log('Cache miss');
    const result = await db.query(sql, params);

    // Store in cache
    await redis.set(cacheKey, JSON.stringify(result), 'EX', ttl);

    return result;
  }

  hash(sql, params) {
    const crypto = require('crypto');
    return crypto
      .createHash('md5')
      .update(sql + JSON.stringify(params))
      .digest('hex');
  }
}

Database Query Cache​

MySQL Query Cache (Legacy):

-- Enable query cache (MySQL 5.7 and earlier)
SET GLOBAL query_cache_type = 1;
SET GLOBAL query_cache_size = 268435456; -- 256MB

-- Check cache statistics
SHOW STATUS LIKE 'Qcache%';

Modern approach: Result set caching

-- Use indexed materialized views
CREATE TABLE product_stats_cache AS
SELECT
  XPRD05 as category_id,
  COUNT(*) as product_count,
  AVG(XPRD02) as avg_price,
  MAX(UPDATED_AT) as last_updated
FROM TB_ANAG_PRD00
WHERE TREC = 'N'
GROUP BY XPRD05;

-- Add index
CREATE INDEX idx_category ON product_stats_cache(category_id);

-- Refresh periodically (via cron or trigger)
TRUNCATE product_stats_cache;
INSERT INTO product_stats_cache
SELECT XPRD05, COUNT(*), AVG(XPRD02), MAX(UPDATED_AT)
FROM TB_ANAG_PRD00 WHERE TREC = 'N' GROUP BY XPRD05;

Monitoring and Profiling​

Enable Slow Query Log​

MySQL:

-- Enable slow query log
SET GLOBAL slow_query_log = 1;
SET GLOBAL long_query_time = 1;  -- Log queries > 1 second
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- Analyze slow queries
SELECT
  DIGEST_TEXT,
  COUNT_STAR,
  AVG_TIMER_WAIT / 1000000000 as avg_ms,
  SUM_TIMER_WAIT / 1000000000 as total_ms
FROM performance_schema.events_statements_summary_by_digest
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 10;

API Response Time Monitoring​

const promClient = require('prom-client');

// Histogram for response times
const httpDuration = new promClient.Histogram({
  name: 'http_request_duration_seconds',
  help: 'HTTP request duration',
  labelNames: ['method', 'route', 'status']
});

// Middleware
app.use((req, res, next) => {
  const timer = httpDuration.startTimer();

  res.on('finish', () => {
    timer({
      method: req.method,
      route: req.route?.path || req.path,
      status: res.statusCode
    });
  });

  next();
});

// Expose metrics
app.get('/metrics', async (req, res) => {
  res.set('Content-Type', promClient.register.contentType);
  res.end(await promClient.register.metrics());
});

Performance Checklist​

✅ DO:​

Optimize queries:

-- ✅ Good - selective WHERE clause
SELECT PRD_ID, XPRD01 FROM TB_ANAG_PRD00
WHERE XPRD05 = 123 AND TREC = 'N'
LIMIT 10;

Use proper indexes:

-- ✅ Good - composite index
CREATE INDEX idx_prd_category_trec ON TB_ANAG_PRD00(XPRD05, TREC);

Implement caching:

// ✅ Good - cache with TTL
await cache.set(key, data, 300);

Monitor performance:

// ✅ Good - track metrics
const timer = histogram.startTimer();
await processRequest();
timer();

❌ DON'T:​

*Avoid SELECT :

-- ❌ Bad - fetches all columns
SELECT * FROM TB_ANAG_PRD00;

-- ✅ Good - specific columns
SELECT PRD_ID, XPRD01, XPRD02 FROM TB_ANAG_PRD00;

Don't use functions on indexed columns:

-- ❌ Bad - function prevents index usage
WHERE UPPER(XPRD03) = 'ABC123'

-- ✅ Good - direct comparison
WHERE XPRD03 = 'ABC123'

Avoid large offsets:

-- ❌ Bad - scans millions of rows
LIMIT 50 OFFSET 1000000

-- ✅ Good - keyset pagination
WHERE PRD_ID > 1000050 LIMIT 50

Summary​

  • ✅ Use EXPLAIN to analyze and optimize queries
  • ✅ Create strategic indexes on frequently queried fields
  • ✅ Implement keyset pagination for large datasets
  • ✅ Prevent N+1 queries with batch requests
  • ✅ Configure connection pooling appropriately
  • ✅ Cache query results at application level
  • ✅ Monitor slow queries and response times
  • ✅ Select only required fields

Key Takeaways:

  1. Query optimization starts with proper indexing
  2. Use EXPLAIN to identify performance bottlenecks
  3. Keyset pagination outperforms offset-based pagination
  4. Prevent N+1 queries with batching or expansion
  5. Connection pooling prevents resource exhaustion
  6. Cache query results with appropriate TTL
  7. Monitor performance metrics continuously
  8. Profile slow queries to identify optimization opportunities
